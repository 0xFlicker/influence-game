Game MCP OAuth Productionization

A focused next slice for turning the existing local Game MCP OAuth bridge into a deployed, client-installable MCP endpoint for production/staging game validation.

Date
2026-06-19
Topic
game-mcp-oauth-productionization

repo-grounded
Boundary
scope=mcp + valid token = global MCP access

Grounding Context

Core boundary: keep authorization intentionally simple. A user must have the mcp role, request scope=mcp, and receive a valid OAuth token. That token grants global access to the wired MCP surface. No per-user, per-agent, private-agent, or per-game authorization boundaries are introduced in this slice.

Current Auth

The API has /api/oauth/mcp/authorize, /token, and /introspect. Tokens are opaque, one-hour, role-checked, and carry aud=game-mcp, purpose=mcp_access, and scope=mcp.

Current Client Flow

The web app hosts /oauth/mcp/authorize for browser approval. The engine login helper opens that page, receives a loopback callback, exchanges the code, and stores a local token file.

Current MCP

The Game MCP is stdio and filesystem-backed. It reads packages/engine/docs/simulations, exposes read-only JSON-RPC tools, and advertises protocol 2025-06-18.

Deployed Data

The API already has durable read models for persisted game events, replayed projections, sanitized durable-run summaries, evidence counts, and private trace manifests. Those are the natural backing seams for deployed game data.

Direct Answers

1. Smallest Production-Grade V0

The smallest real V0 is an API-hosted https://<api>/mcp Streamable HTTP resource server, protected by the existing MCP OAuth tokens, serving read-only tools against deployed API game data. It should return normal JSON responses for POSTed JSON-RPC requests, return 405 for unsupported GET/SSE in V0, validate bearer auth on every request, and advertise exactly scope=mcp.

V0 tools should be the deployed analog of the existing Game MCP plus the developer evidence/private reasoning surfaces intentionally selected for this MCP role: list deployed games, read/replay projection, filter canonical events, read a player timeline, inspect durable-run state, and expose explicit private trace manifest/content/search tools behind the same global scope=mcp gate.

2. Missing Protocol/Auth Requirements

  • A deployed Streamable HTTP MCP endpoint with JSON-RPC POST handling and proper Accept / Content-Type behavior.
  • WWW-Authenticate challenges that point at protected resource metadata and include scope="mcp".
  • /.well-known/oauth-protected-resource on the MCP resource origin with resource, authorization_servers, and scopes_supported: ["mcp"].
  • Authorization server metadata, either OAuth or OIDC-shaped, including authorization/token endpoints, PKCE S256 support, scopes, grant types, and token auth methods.
  • Acceptance and validation of the OAuth resource parameter on authorization and token requests, binding issued tokens to the canonical MCP resource URI.
  • A client identity path compatible with fresh clients: Client ID Metadata Document, Dynamic Client Registration, or predefined public clients. This needs an empirical compatibility spike because Codex docs confirm OAuth login behavior but do not expose all registration internals.
  • For ChatGPT Apps runway, per-tool OAuth securitySchemes and auth error metadata that can trigger linking.

3. What Moves From Local to Deployed

  • Transport moves from stdio newline JSON-RPC to HTTPS Streamable HTTP.
  • Auth moves from local token file plus bridge-side introspection secret to API-side bearer validation on the MCP request itself.
  • Read model moves from filesystem simulation sessions to deployed Postgres game rows, canonical events, projection replay, durable-run inspection, and selected private trace read-model paths.
  • Resource identity moves from influence-game://sessions/<session>/games/<n> to API game identities such as influence-game://games/<slug-or-id>/events.
  • Docs move from local validation commands to install/auth recipes for Codex, Claude Code, MCP Inspector, API Playground, and ChatGPT developer mode.

4. Client Install/Auth Paths To Target First

  • Codex first: configure a Streamable HTTP server URL in ~/.codex/config.toml, run Codex MCP OAuth login, confirm it uses advertised scopes_supported, stores the token, and can list/call tools.
  • Claude Code second: run claude mcp add --transport http influence <url>, authenticate via /mcp, and verify both automatic discovery and a pinned oauth.scopes = "mcp" config.
  • ChatGPT developer mode third: connect the HTTPS endpoint in developer mode or API Playground, prove OAuth linking, tool discovery, auth challenge behavior, and basic golden prompts. Do not build iframe UI yet.

5. Explicitly Out Of Scope

  • Per-user, per-game, per-agent, private-agent, or private-trace authorization boundaries.
  • Adding per-resource authorization or private-reasoning redaction/removal after the global MCP gate.
  • Mutating game state through MCP.
  • Public ChatGPT app submission, app listing copy, or embedded UI components.
  • Crash-safe game resume or checkpoint hydration behavior.
  • OAuth refresh tokens unless a client compatibility smoke proves they are required for V0.

6. Workstreams And Rough Order

  1. Run a thin compatibility spike for Codex/Claude/ChatGPT metadata and client registration behavior.
  2. Add OAuth discovery metadata, resource validation, and challenge responses while preserving scope=mcp.
  3. Add the API-hosted Streamable HTTP JSON-RPC adapter.
  4. Add the API-backed Game MCP read model and deployed resource URI scheme.
  5. Add security envelope: origin validation, size/output limits, audit events, token revocation checks, and no token leakage in logs. Defer rate limiting to a future gateway-backed durable deploy hardening slice.
  6. Add client install docs and staging smoke runbooks.
  7. Run client proof: MCP Inspector, Codex, Claude Code, ChatGPT developer mode/API Playground.

7. Security Risks Even For Test/Validation

  • Global access is powerful: a stolen token can read every wired MCP game surface until expiry or revocation.
  • OAuth discovery is attack surface: client registration and metadata fetches can create open redirect, confused deputy, or SSRF risks if implemented casually.
  • Prompt injection travels through data: game transcripts and logs are model-consumed content, so tool descriptions and docs must frame results as untrusted game artifacts.
  • Producer data can leak: V0 intentionally exposes selected developer evidence/private reasoning tools behind scope=mcp, so logs, docs, and tool descriptions must make that trusted-developer boundary visible.
  • Operational drift matters: staging/prod base URLs, HTTPS, metadata cache headers, callback URLs, and role assignment can fail independently of code correctness.

8. Readiness Proof

  • Unit tests for protected resource metadata, authorization server metadata, resource validation, invalid scope, invalid audience, missing token, expired token, revoked role, and exact redirect matching.
  • Transport tests for JSON-RPC initialize, tools/list, tools/call, resources/list, resources/read, notifications returning 202, unsupported GET returning 405, and auth challenge behavior.
  • Read-model tests proving deployed game listing, event replay, projection summary, player timeline, durable-run summary, and explicit private trace tools expose the intended data classes without accidental leakage through the wrong tool.
  • Staging smoke with MCP Inspector and raw curl before clients.
  • Fresh Codex session install/auth/tool-call proof.
  • Fresh Claude Code install/auth/tool-call proof.
  • ChatGPT developer mode or API Playground linking and golden prompt proof.
  • Ops check: audit log entries, no bearer tokens in logs, token revocation path, request/output bound behavior, and role removal invalidates existing tokens.

1. Hosted /mcp Resource Server Wedge

Axis
Protocol surface
Confidence
88%
Complexity
Medium

Description: Add a deployed API route, likely POST /mcp, that adapts the current JSON-RPC server shape to MCP Streamable HTTP. V0 can be intentionally simple: JSON responses for requests, 202 for notifications/responses, no SSE stream, no server-managed sessions, and 405 for GET. Every request is authenticated with Authorization: Bearer.

Basis: direct: MCP Streamable HTTP requires a single endpoint supporting POST/GET, and the repo already has a transport-neutral-ish GameMcpJsonRpcServer.handle() that can be wrapped.

Rationale: This is the smallest slice that makes the client experience real. Fresh Codex/Claude/ChatGPT clients can only install a deployed MCP if there is an HTTP MCP endpoint to install.

Downsides: Requires careful HTTP conformance and auth errors. It also forces a decision about whether to advertise latest protocol 2025-11-25 or remain on 2025-06-18 pending smoke results.

2. API-Backed Game Read Model

Axis
Deployed read model
Confidence
84%
Complexity
Medium

Description: Build a deployed read model with the same read-only discipline as local Game MCP, but backed by API data. It should list games, read persisted canonical events, replay projections, filter timelines, expose durable-run summaries, and wire selected private trace manifest/content/search paths as explicit developer tools. Keep the local simulation filesystem read model intact; do not pretend API games have simulation sessionId + gameNumber identities.

Basis: direct: game-event-read-model.ts, game-projection-read-model.ts, game-durable-run.ts, and private-trace-read-model.ts already validate event integrity, replay projections, summarize durable inspection output, and read private trace evidence.

Rationale: This turns deployed games into the MCP corpus without coupling the engine package to API storage, while keeping private reasoning available through intentional developer tools behind the same MCP gate.

Downsides: Tool schemas may diverge from local Game MCP because API games do not naturally have simulation sessions. The docs need to name that difference plainly.

3. OAuth Discovery and Resource Binding Compliance

Axis
OAuth and client identity
Confidence
90%
Complexity
Medium

Description: Publish the metadata and challenge surface that real MCP clients expect: protected resource metadata, authorization server metadata, WWW-Authenticate challenges, scopes_supported: ["mcp"], PKCE S256 support, token endpoint auth method, and canonical resource validation on authorization and token exchange.

Basis: external: MCP authorization and OpenAI Apps SDK docs require protected resource metadata, AS metadata, bearer token usage, and audience/resource validation for OAuth-protected MCP servers.

Rationale: The current repo has a working local OAuth-like flow, but clients cannot discover it as a standards-shaped protected MCP resource. This idea converts the same boundary into the protocol surface clients understand.

Downsides: Requires production URL canonicalization. If staging and production use different web/API origins, metadata must stay exact and cache-safe.

4. Client Registration Compatibility Shim

Axis
OAuth and client identity
Confidence
72%
Complexity
Medium/High

Description: Add the least risky client identity path that works across target clients. The likely route is a short empirical spike followed by Client ID Metadata Document support plus a minimal Dynamic Client Registration fallback, or DCR first if Codex requires it. Public clients should use exact redirect URI matching and no client secret unless a specific client requires one.

Basis: external: latest MCP authorization lists preregistration, Client ID Metadata Documents, and DCR; Claude Code documents DCR/CIMD/preconfigured support; OpenAI Apps SDK says ChatGPT can use CIMD, DCR, or predefined clients.

Rationale: A fresh client cannot complete OAuth if the authorization server only accepts one hardcoded local client ID. This is the most important unknown to settle early.

Downsides: DCR/CIMD can introduce SSRF and redirect risks. Do not ship a broad generic client registry without tight validation and logging.

5. Client Compatibility Lab

Axis
Client proof and docs
Confidence
86%
Complexity
Low/Medium

Description: Treat client compatibility as an artifact, not a hope. Create a staging checklist and fixtures for raw curl, MCP Inspector, Codex, Claude Code, and ChatGPT developer mode/API Playground. Capture exact install commands, callback URLs, scopes requested, metadata fetched, token storage behavior, and first successful tool calls.

Basis: reasoned: the docs expose overlapping but not identical behavior, especially around OAuth client identity. A cheap compatibility lab prevents building a beautiful standards surface that one target client cannot actually use.

Rationale: This is the guardrail that keeps V0 honest. It also produces the eventual install docs almost for free.

Downsides: Requires access to all three clients during validation. If ChatGPT developer mode is gated by org settings, API Playground can be the interim proof.

6. Security Envelope for Global mcp Tokens

Axis
Security and operations
Confidence
82%
Complexity
Medium

Description: Add explicit production safeguards around the simple global access boundary: Origin validation on HTTP transport, request body/output size caps, audit events for connect/list/call/read, role-removal invalidation, token revocation visibility, and log redaction that never prints bearer tokens. Keep rate limiting out of this slice until it can live behind a real gateway.

Basis: external: MCP transport docs call out Origin validation and authentication; MCP auth docs emphasize token theft, audience validation, and exact redirects. Direct: current OAuth audit events exist for authorize/token/introspect, but deployed MCP calls do not exist yet.

Rationale: Simple auth is fine only when the operational envelope is explicit. The consent page should say global Game MCP access because that is the product decision.

Downsides: Adds non-feature work to V0. Still worth doing because a deployed MCP endpoint is a new public attack surface.

Axis
Client proof and docs

Complexity
Low/Medium

Description: Without building a full ChatGPT app, shape tool metadata for that future: titles, descriptions, output schemas where useful, per-tool OAuth securitySchemes, and auth error metadata that can trigger ChatGPT linking. Keep UI resources, iframe components, and submission material out of V0.

Basis: external: Apps SDK docs describe MCP as the backbone for ChatGPT apps, recommend Streamable HTTP, and require per-tool auth policy metadata for OAuth linking.

Rationale: This keeps the V0 endpoint compatible with ChatGPT developer mode now and reduces rework if it becomes an installable app later.

Downsides: Easy to overdo. Stop at metadata and developer-mode testing.

Rejection Summary

# Idea Reason Rejected
1 Local bridge-only productionization Already close to the current state and does not prove fresh clients can install/connect to a deployed MCP server.
2 Full private Trace MCP over production OAuth Scope overrun and high privacy risk. Private trace content remains local producer/debug tooling in this V0.
3 Per-user, per-agent, or per-game authorization Explicitly violates the requested boundary: scope=mcp grants global MCP access.
4 Public ChatGPT app submission Useful later, but public submission and UI packaging are not required to productionize deployed MCP install/auth.
5 Mutation-capable Game MCP Subject replacement. The existing use case is read-only test/validation against game data.
