Ideation: Automatic Avatar PFP Generation

Date: 2026-07-02 Topic: automatic-avatar-pfp-generation Mode: repo-grounded Focus: Katana image generation for The House agent PFPs

Recommendation: ship PFP generation as an idempotent, async, budget-accounted server-side enhancement to agent creation. Start with one generated image per new agent when the user leaves the form without an avatar and confirms, plus automatic generation for MCP-created agents that omit avatarUrl.

The non-negotiable storage rule: Katana URLs are acquisition URLs only. Download the completed original_data_url, validate/resize if needed, write it through Influence profile-picture storage, and persist only the resulting durable Influence URL.

The non-negotiable cost rule: do not model this as a cute UX toggle. Model it as paid provider spend with a ledger from day one. At roughly $0.015 per image, the unit cost is tiny until automation finds the button. Then it becomes confetti with an invoice.

Grounding Context

The existing agent service already normalizes supplied avatar URLs before insert/update, so the generated path should feed the same durable field instead of inventing a second cosmetic store. The MCP create/update tools already accept avatarUrl for owned agents, so automatic generation can hang off the same management service rather than being only a web-only feature.

Profile-picture storage already has the right separation: local development can write local files; hosted deployments use S3-compatible object storage; upload URLs are temporary, while public URLs are the durable field. The README documents the same split-origin local behavior and Doppler-held Katana credentials.

External abuse context matters because this introduces a server-funded paid action. OWASP classifies automated account creation and cost-inflation fraud as distinct automated threats. OWASP's authentication guidance treats CAPTCHA as defense-in-depth, not prevention. Cloudflare Turnstile is useful later because it can run less intrusive challenges without requiring Cloudflare CDN, but it should not be the core accounting primitive.

1.Budgeted Avatar Generation Pipeline

Conf 92% · Cx M

Description: Add a backend service that takes an agent profile, builds a Katana gen request, polls it, downloads the completed image, writes the bytes into the existing profile-picture storage backend, and patches agent_profiles.avatar_url. The operation should be async and idempotent per agentId + generationKind, so retries do not buy ten portraits for one agent because a request timed out or a browser tab got dramatic.

Axis: Storage and durability

Basis: direct: createOwnedAgent already stores avatarUrl after normalization, and update does the same for mutable profiles; storage already exposes public durable URLs separately from temporary upload URLs. See packages/api/src/services/agent-profile-management.ts:260 and packages/api/src/lib/storage.ts:115.

Rationale: This keeps generated PFPs inside the same rendering contract as uploaded PFPs, which avoids a parallel "AI image URL" path and prevents Katana asset links from becoming product state.

Downsides: Requires a small job-state model or DB table if we want true idempotency and user-visible progress instead of a synchronous route.

2.Exit-Intent PFP Prompt, Not Form-Blocking Wizardry

Conf 86% · Cx S

Description: When a user creates or edits an agent and leaves the avatar empty, show one focused prompt after save or on close: "Generate a portrait for this agent?" If yes, start the backend job and return them to the agent list with a generating state. Do not block profile creation on image generation, because Katana's image endpoint is polled and can legitimately take a while.

Axis: Generation trigger

Basis: direct: the current form has a distinct avatarUrl state and submits successfully with it undefined; personality generation is already optional. See packages/web/src/app/dashboard/agents/agent-form.tsx:27 and packages/web/src/app/dashboard/agents/agent-form.tsx:79.

Rationale: This preserves the current fast "make an agent" flow while nudging visual completeness. The prompt appears only when useful, not as another setup chore.

Downsides: Needs a small pending/generating UI state. Without that, users will wonder whether the button did anything, which is how support tickets are born.

3.MCP Auto-PFP With Explicit Result Metadata

Conf 84% · Cx M

Description: For MCP create_agent, if the caller omits avatarUrl, automatically enqueue one PFP generation and include avatarGeneration metadata in the tool result: status, reason, and optionally budgetRemaining. Do not require the MCP client to call a second tool just to make the new agent look alive.

Axis: MCP behavior

Basis: direct: the MCP create tool already accepts avatarUrl but does not require it, and calls into the same createOwnedAgent service. See packages/api/src/game-mcp/server.ts:729 and packages/api/src/game-mcp/server.ts:274.

Rationale: MCP-created agents are likely to be authored by remote LLMs or assistant flows; expecting them to manage image generation and storage is needless plumbing. The product should make the good default easy while returning honest metadata when generation is skipped or rate-limited.

Downsides: Automatic spend on MCP writes raises abuse pressure because it moves the cost trigger behind an API-like tool. This idea depends on the spend ledger in idea 4.

4.Spend Ledger Before CAPTCHA

Conf 90% · Cx M

Description: Introduce a generic account spend ledger with entries for provider, operation, estimated cost, actual cost, status, and idempotency key. Give new accounts a tiny free budget for first-agent niceties, then enforce per-account and per-IP/provider windows before calling Katana. CAPTCHA or Turnstile can come later as adaptive friction when the ledger says spend risk is weird.

Axis: Cost and abuse control

Basis: external: OWASP identifies automated account creation and cost-inflation fraud as separate web automation threats; OWASP also frames CAPTCHA as defense-in-depth rather than prevention. Cloudflare Turnstile can reduce friction by adapting challenges to visitor risk, but it is still a gate, not accounting.

Rationale: The first durable primitive should answer "who spent how much, for what, and was it allowed?" That primitive will also support future paid tiers, comped generations, OpenAI personality generation controls, and any future media generation.

Downsides: It is more infrastructure than a one-off counter. Worth it, because one-off counters become the haunted basement of every freemium app.

5.Persona-Aware Prompt Compiler

Conf 78% · Cx S

Description: Generate a portrait prompt from the agent's name, archetype, public biography, personality, and strategy style, with strict house style constraints: square portrait, readable at small size, no text/logo/watermark, consistent game-art direction. Store the prompt and Katana request ID only in admin/debug metadata, not in public profile payloads.

Axis: Generation trigger

Basis: direct: agent profile data already captures name, biography/backstory, personality, strategy style, and persona key. See packages/api/src/db/schema.ts:97 and packages/api/src/services/agent-profile-management.ts:285.

Rationale: One-shot generation will only feel good if the first prompt is strong. Prompt compilation is cheap compared with user-visible bad portraits and avoids asking the user for art-direction work they did not sign up for.

Downsides: Needs careful prompt hygiene so user-authored personality text cannot smuggle instructions like "include this text" into the image request. Mildly cursed, but tractable.

6.One-Shot Free Now, Regenerate Later

Conf 74% · Cx M

Description: For v1, give each agent at most one free automatic generated PFP. If the result is bad, the user can upload their own image; regeneration stays disabled or producer-only until paid tiers exist. Later, make regenerate a budgeted action with clear cost copy and maybe a "use this one" review loop.

Axis: Future paid tiers

Basis: reasoned: the current stated goal is to eat the roughly $0.015 cost for now, but repeated image retries are the obvious way a tiny unit cost stops being tiny. One-shot gives visual polish without opening a slot machine.

Rationale: This gets the product benefit now and leaves a natural paid-tier affordance later. It also keeps the first implementation out of gallery/retry/review complexity.

Downsides: Some users will dislike the first image. That is acceptable for a default PFP, not acceptable for a paid/custom portrait product.

7.Avatar Change Ledger Toward a Moderation Queue

Conf 68% · Cx M

Description: Record every avatar change, not just generated-avatar jobs: user upload, Katana generation, MCP-created default, manual replacement, failed generation, and removal. Each event should capture user, agent, previous avatar URL/storage key, new avatar URL/storage key, source, actor, status, cost metadata when present, and enough provider/debug context to investigate without exposing raw prompts publicly. The future shape is a moderation review queue where producers can approve, reject, rollback to a prior avatar, ban a specific avatar asset/hash, or escalate to account-level action; that review queue is explicitly not the first implementation slice.

Axis: Cost and abuse control

Basis: direct: the app already has admin surfaces and producer/debug patterns for trace metadata, while public payloads avoid raw provider details. Agent avatars can already change through web profile updates and MCP update_agent, and generated PFPs would add another mutation path; moderation needs the shared mutation history, not a Katana-only inbox.

Rationale: Avatar changes are user-visible identity changes. Once the app funds or accepts images, the operator needs a clean answer to "who changed this, what was there before, can we undo it, and should this asset/user be blocked?" Designing the ledger now keeps the eventual moderation queue from becoming a panic-built side table full of vibes and regret.

Downsides: Easy to overbuild. For the PFP generation slice, capture the ledger fields and perhaps a read-only admin table; defer workflow actions like review assignment, rollback buttons, avatar-hash bans, and account sanctions to a dedicated moderation slice.

Rejection Summary

#IdeaReason Rejected
1Generate synchronously during agent saveBad UX and brittle runtime behavior; image generation is polled and can take long enough to make a save route feel broken.
2Persist Katana asset URLs directlyViolates the durable-avatar contract and recreates the signed/external URL footgun in a new hat.
3Require CAPTCHA before every avatar generationToo much friction for the current audience; CAPTCHA is useful adaptive friction, not the spend-control source of truth.
4Expose image generation as a public MCP toolScope overrun. The product need is automatic PFP completion on agent creation, not a general remote image-generation API.
5Generate multiple candidates by defaultNice eventually, but triples/quintuples spend immediately and pushes the feature into paid-tier territory.
6Use generated PFPs only for web-created agentsMisses the stronger default for MCP-created agents, where remote clients are least likely to provide a durable avatar URL.
