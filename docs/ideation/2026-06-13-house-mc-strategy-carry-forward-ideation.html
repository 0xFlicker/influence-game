House MC Strategy Carry-Forward

How to upgrade The House from a short-tail narrator into a high-context producer that identifies alliances, names them, dramatizes them, and uses that work to ask better diary-room questions.

Date: 2026-06-13 Mode: repo-grounded Topic: house-mc-strategy-carry-forward
Focus: selected option 3, high-context House producer memory before diary sessions

Grounding Context

The current House is split across three responsibilities. In packages/engine/src/house-interviewer.ts, it assigns initial Mingle rooms, asks diary questions, and writes short gameplay summaries. The diary prompt is sharp, but it is still built from a compact DiaryRoomContext: recent public messages, a few player messages, prior diary answers, alive players, eliminations, and current council state.

House MC summaries are even more constrained. GameRunner calls generateGameplaySummary after Mingle and Council only, passing this.logger.transcript.slice(-15) or slice(-30). The method then slices again to the last 20 entries. That means the House cannot reliably carry forward alliances, patterns, dropped promises, or emerging factions across rounds, even if the underlying model can handle much larger context.

The simulator currently disables diary rooms by default with diaryRoomAfterPhases: []. Strategic reflection capture is also explicit via --strategic-reflections or INFLUENCE_SIM_STRATEGIC_REFLECTIONS=true. The user's goal to "optionally turn diary back on" and "always see House summaries" should be treated as two related but separate controls.

The strongest repo learning is the strategy observability spine: strategy improvements should become typed private evidence, not prompt folklore. The same pattern should apply to The House. If the House is identifying alliances and shaping diary questions, that work needs a structured private record so MCP and simulation artifacts can validate carry-forward instead of judging only the final recap prose.

Core diagnosis: The House is omniscient in tone but short-memory in implementation. The upgrade is not just "better recap prompt." It is a producer-memory layer that creates a structured House strategy packet before narration and diary interviews consume it.

1

House Strategy Bible Packet

Axis: House memory Confidence: 92% Complexity: Medium

Create a persistent, producer-private House packet updated between rounds. The packet should summarize named alliances, active tensions, broken promises, vote blocs, Mingle room discoveries, dramatic storylines, next diary angles, dropped threads to revisit, and open uncertainties. The next packet receives the previous packet as first-class input, so carry-forward is explicit rather than dependent on whatever recent transcript slice survives.

Description
Add a House method shaped like generateStrategicHousePacket(context). It consumes full or large-context evidence plus the previous House packet and returns structured JSON: alliance entities, conflict arcs, player trajectories, producer hypotheses, diary prompts to probe, and summary beats. Store the latest packet on GameRunner or a House context object for the live run, and emit it as a private/system agent_turn for simulation review.
Basis
direct: generateGameplaySummary currently receives only recent transcript tails, while DiaryRoomContext carries only a compact slice of public and player-specific messages. direct: repo learning says model-quality complaints should become typed private evidence flowing through prompts, phase runner, artifacts, MCP, and tests.
Rationale
This is the central fix for "carry forward was not guaranteed." The House cannot reliably name alliances across rounds if each summary starts from a short recent slice. A packet makes the House's own strategic memory inspectable and reusable by diary questions, recaps, and future MC work.
Downsides
Extra LLM call and a new private type contract. It also needs careful visibility discipline: this is producer/debug state, not public game truth and not agent knowledge.
2

Pre-Diary Producer Briefs

Axis: Diary quality Confidence: 88% Complexity: Medium

Before a diary session, have The House convert its strategy packet into a per-player producer brief. The diary question generator should receive that brief, not just raw recent messages. Each brief names the player's current role in the story, alliances to probe, contradictions to needle, secrets The House knows but should not reveal directly, and one or two sharp question angles.

Description
Add a structured intermediate step: buildDiaryProducerBrief(player, housePacket, context). The brief is private to the House prompt and simulation artifacts. It can include fields like storyRole, suspectedAllies, pressurePoints, recentContradictions, questionAngles, and doNotReveal.
Basis
direct: generateQuestion currently builds from a generic situation switch plus recent messages and prior diary entries. direct: the House prompt already asks for pointed, specific questions, but the available context is not rich enough to guarantee continuity.
Rationale
This separates producer analysis from wording. The model can first decide what matters, then ask one clean question. It also creates inspectable evidence when a diary question is great or misses the mark.
Downsides
One more internal object to maintain and test. If over-specified, it could make questions too formulaic, so the brief should supply angles, not final scripts.
3

Named Alliance Entities

Axis: Alliance detection Confidence: 85% Complexity: Medium

Make alliances first-class producer hypotheses. The House should identify groups, give them memorable showrunner names, track membership and confidence, cite evidence, and describe fracture lines. Names are for the audience and producer artifacts: "The Glass Table," "Quiet Majority," or "The Debt Circle" can dramatize social structure without exposing omniscient knowledge to players.

Description
Define a HouseAllianceEntity shape with id, name, members, confidence, evidence, status, fractureLines, publicCover, and lastUpdatedRound. The packet update can keep, merge, rename, or retire entities as evidence changes.
Basis
direct: the user explicitly wants The House identifying, naming, and dramatizing alliances. direct: existing agent reflections already track allies and threats individually, but no House-level global alliance object exists.
Rationale
A named entity gives continuity to recaps and diary sessions. It also avoids repeated generic language like "alliances are shifting" by letting The House say which relationship cluster is actually shifting and why.
Downsides
False positives are inevitable. The packet must label confidence and evidence so a producer can distinguish a real alliance from a dramatic but weak read.
4

Always-On Round MC Recaps

Axis: Observability Confidence: 82% Complexity: Low

Emit House MC summaries as normal simulation artifacts, not as something humans only get when a run is chatty. The recap should happen at stable round boundaries and should use the House packet, so the text becomes a visible audience layer over structured producer memory.

Description
Add a round-summary call after each completed round or after each elimination resolution. Keep phase mini-summaries optional if they are useful, but make the end-of-round recap default for simulation output. The summary should be logged through logSystem and serialized into transcript JSON, whether or not --chatty is printing live terminal output.
Basis
direct: GameRunner currently calls House MC summaries only after Mingle and Council. direct: the user wants House summaries always visible and suspects they only appear with --chatty.
Rationale
This gives every run a human-readable strategic spine. It also creates checkpoints for comparing House memory against actual later decisions.
Downsides
More tokens in every simulation unless controlled by config. Summary frequency should be explicit so release-validation runs can stay bounded when needed.
5

Diary and Summary Presets

Axis: Cost controls Confidence: 80% Complexity: Low

Expose diary-room and House-summary behavior as explicit simulator modes. This resolves the current ambiguity where diary is disabled for bounded local runs, strategic reflections are flag-gated, and House summaries are present but sparse.

Description
Add CLI/config presets such as --diary-room off|round|key-phases|all, --house-summaries off|round|phase, and a higher-level --producer-mode preset that enables House packets, round summaries, diary questions, and strategic reflections. Keep fast default behavior deliberate, but make the rich mode one flag instead of a pile of hidden toggles.
Basis
direct: buildSimulationConfig sets diaryRoomAfterPhases: [] and maxDiaryFollowUps: 0. direct: parseArgs already has strategic-reflection flags, proving this config layer is the right place for simulation validation controls.
Rationale
The user wants to turn diary back on but may not know the flag because there is not one. Presets make cost and behavior discoverable while avoiding accidental token explosions.
Downsides
CLI surface area grows. Tests need to protect default behavior and each preset.
6

Carry-Forward Validation Surface

Axis: Observability Confidence: 78% Complexity: Medium

Add MCP/search support and instrumentation counters for House packets: alliance entities created, retained, renamed, dropped, contradicted, or revived. The point is to measure whether House carry-forward actually survives from one summary to the next.

Description
Emit house-strategy-packet, house-diary-brief, and house-mc-summary records into the structured turn/progress artifacts. Add generic MCP searches if current log search is enough, or a thin query helper like "show house packets for session/game" if repeated analysis proves awkward. Instrument retention: previous packet alliance IDs appearing in the next packet, dropped-with-reason counts, and summary references to active alliances.
Basis
direct: game MCP and game-N-turns.jsonl are already the validation surface for Mingle intent, strategy packets, and strategic lenses. direct: the user specifically wants to validate House carry-forward across summaries.
Rationale
Without a validation surface, a better House prompt can still feel good while forgetting half the story. Retention metrics turn "did the House remember?" into something searchable.
Downsides
Some metrics will be approximate because House alliance entities are hypotheses, not canonical facts. The artifact should be reviewed as producer evidence, not treated as ground truth.
7

Big Evidence Bundle Builder

Axis: Context assembly Confidence: 76% Complexity: Medium

Build a dedicated House evidence bundle instead of passing raw transcript tails from callers. Since this House turn is allowed to spend more tokens, the bundle should include the full current round, recent prior rounds, all relevant private agent turns, canonical facts, diary answers, vote history, Mingle assignment diagnostics, and the previous House packet.

Description
Add a HouseContextBuilder or a method on the existing context layer that returns a large structured object for the House. The object should preserve evidence type and visibility: public transcript, private producer/debug turns, canonical outcomes, diary Q/A, and agent strategy packets. Do not optimize hard in v1. Make the evidence complete enough for strong House reasoning, then later add pruning if cost forces it.
Basis
direct: current callers pass this.logger.transcript.slice(...) directly into generateGameplaySummary. reasoned: prompt size optimization is not the bottleneck for the selected House MC upgrade because the user explicitly accepts large House contexts for better producer intelligence.
Rationale
The House needs the right raw material before prompt wording matters. A bundle builder also prevents future call sites from quietly starving the House again.
Downsides
Can become a dumping ground if not typed. The bundle should be large, but still structured enough that prompts can explain what each evidence class means.

Rejection Summary

# Idea Reason Rejected
1 Make House alliance reads canonical game events Wrong truth tier. House reads are producer hypotheses; canonical events should remain accepted board facts.
2 Share House alliance graph with player agents Hidden-information leak. The House can be omniscient for viewers and producers, but players should not receive global reads they did not earn.
3 Let The House rewrite agent Strategy Thread packets Breaks player agency. Agent packets should remain each player's private intent; House packets can analyze them but not edit them.
4 Enable full diary after every phase by default Too expensive and noisy as a blanket default. Better as an explicit rich simulation preset or diary mode.
5 Only improve generateGameplaySummary prompt copy Not enough. The main bug is missing durable House memory and thin evidence input, not just weak prose instructions.
6 Keep House summaries visible only through --chatty Already causing confusion. Summaries should be serialized artifacts and optionally printed live.
7 Infer alliances only from public transcript text Underuses the available producer/debug spine. Mingle intents, private turns, votes, and strategy packets are essential House evidence.
8 Optimize House context size immediately Premature for this slice. The user explicitly wants to spend context to maximize House knowledge first.
9 Make alliance names player-visible during the game Changes game mechanics and leaks analysis. Use names in producer/debug artifacts and audience recaps unless a future design chooses a public broadcast reveal model.
