Ideation: Named Alliances

Best direction: make alliances explicit consent artifacts first, then let The House schedule scarce sidecar-native huddles before Vote and Council. Acceptance records a social commitment; huddle time is producer-gated screen time, not an always-on private chat entitlement.

Grounding Context

The codebase already has a strong private-social substrate, but it does not yet have first-class alliance truth. Current games run through MINGLE, while Whisper names remain compatibility residue in older types and UI components. Mingle can log private room messages with recipients, roomId, and room metadata, and the engine already emits private agent-turn records such as Mingle intent. What is missing is a small durable object that says: these players explicitly proposed, accepted, countered, or declined a named pact.

The trust boundary matters as much as the feature. Production MCP is intentionally read-only for active games, public watch surfaces should not leak hidden social truth, and producer/debug views can inspect richer evidence than players or public viewers. The useful product shape is not "show everyone the alliance graph." It is controlled private truth plus later postgame explanation.

Refined recommendation: proposal/response ledger plus sidecar-native alliance huddles. The full-drama loop adds Mingle I for formation, House-scheduled pre-vote huddles, the existing post-vote Mingle for fallout, and House-scheduled pre-Council huddles for damage control.

Recommended Rule Changes

Direct proposal: Mingle creates or mutates named alliances; the durable Alliance sidecar holds state between phases; huddle windows are scarce, House-scheduled coordination scenes before Vote and Council, not always-on chat.
Current scope: keep the full-drama route and current token-maxing rules for now. Do not cap how many alliances a player may join until simulations prove the need; do give every agent its full alliance roster plus failed or closed alliance context. Votes remain public until a dedicated private vote reveal phase is worth designing.

R1. Full-Drama Round Loop

Rule
Use Lobby - Mingle I - Pre-Vote Alliance Huddles - Vote - Post-vote Mingle - Power/Reveal - Pre-Council Alliance Huddles - Council as the full-drama normal-round shape. Mingle I forms or mutates alliances. Pre-vote huddles coordinate action. Post-vote Mingle preserves fallout. Pre-Council huddles negotiate reality after pressure is visible.
Axis
Round cadence, huddle placement, post-vote drama.
Basis
direct: phase-machine.ts:253-328 currently routes lobby - vote - mingle - power - reveal - council; CONCEPTS.md:15-17 defines post-vote Mingle as the pressure/fallout window to preserve.
Rationale
Pre-vote coordination is a real new round-loop change, not a rename. This rule adds that coordination while keeping the existing vote-to-Council drama instead of flattening it.
Downside
This is the most expensive shape. Current token-maxing rules still apply until a dedicated alliance-huddle compression pass is designed.

R2. House-Scheduled Huddle Tickets

Rule
The House grants each active alliance 0, 1, or 2 huddle rounds per decision window. Default budget: max(2, floor(alivePlayers / 4)) huddle tickets per window, capped at 4. Score by decision relevance, conflict/tension, underdog flip potential, dominance interruption, recency/fatigue, and estimated token cost.
Axis
Producer/showrunner gate and token budget.
Basis
direct: mingle.ts:762-793 already asks The House to assign rooms and records private House rationale; CONCEPTS.md:99-105 describes House-owned producer strategy memory and producer briefs.
Rationale
The House should schedule attention like a showrunner, not certify loyalty. Scarce huddles save tokens and make fake/stale alliances cheap to ignore until they become dramatic.
Downside
House scheduling becomes part of gameplay. It needs producer-only reasons so weird choices are debuggable.

R3. Pass-Wise Huddle Execution

Rule
For each decision window, run huddles pass-wise: for pass 1..2, each selected alliance gets one huddle round. One huddle round means every live member may speak once. Do not let one alliance finish all discussion before others get a first round.
Axis
Huddle order and conversation beats.
Basis
direct: mingle.ts:802-836 already runs beat-based room turns; this adapts that rhythm to alliance-specific rooms.
Rationale
Pass-wise execution creates actual back-and-forth while avoiding runaway order advantage. Multi-alliance agents can carry tension between rooms without one group monopolizing the window.
Downside
Internal execution order can still matter. In fiction, huddles should be framed as same-window scenes, and prompts should not grant omniscient knowledge from earlier huddles.

R4. Counter Cap and Proposal Cooldown

Rule
Counters may rename, resize, narrow, or timebox an alliance, but each proposal lineage gets at most 2 counter rounds per formation window. Then it must resolve as accept, decline, defer, trial, or expire. Deferred proposals cannot consume huddle tickets until the next round unless House marks them urgent.
Axis
Proposal lifecycle and anti-stall rules.
Basis
reasoned: alliance negotiation is valuable only if it produces a state transition; unbounded counters turn Mingle into paperwork.
Rationale
Counterplay stays strategic, but every proposal reaches a visible next state before it starts consuming huddle budget.
Downside
A hard cap may prematurely kill a fun negotiation. Trial status gives a pressure valve.

R5. Ask, Plan, Receipt

Rule
Each huddle round produces a compact sidecar receipt: one ask, one plan, promises or protections, dissent, confidence, target/vote/Council posture, and any explicit leak or betrayal. Raw huddle transcript is evidence, not future prompt memory.
Axis
Durable memory and token control.
Basis
direct: Mingle already emits private agent-turn records and transcript entries; existing strategy-memory patterns favor compact structured context over raw hidden-history replay.
Rationale
Huddles need to matter later. Receipts make them durable without dumping transcripts into every future prompt.
Downside
Receipt schemas can get too clever. Keep the first version aggressively small.

R6. Universal-Alliance Closure

Rule
Before a round enters Mingle I ahead of a vote, any active alliance whose living membership equals all alive players is unstable and must close. The v1 behavior is to mark it closed in context before Mingle I and let agents react, re-propose smaller pacts, or pass during normal Mingle. Alliances with fewer than two alive members still auto-archive.
Axis
Universal alliance instability and endgame dissolution pressure.
Basis
reasoned: an alliance of everyone alive stops being a coalition and becomes a stalled game state. Closing it before Mingle I preserves social pressure without inventing a special resolution phase yet.
Rationale
Dissolution is social pressure, not a mechanical betrayal requirement. The closed alliance remains as context and receipt evidence even after the room stops getting oxygen.
Downside
Abrupt closure may feel producer-handed. If simulations need more ceremony, promote a special universal-alliance resolution phase from future work.

R7. Producer Reasons Ledger

Rule
Every scheduled or skipped huddle stores producer-only reasons such as swing_vote_pressure, fracture_risk, power_holder_member, recent_betrayal, underdog_flip, dominance_interruption, and fatigue_penalty.
Axis
House scheduling auditability.
Basis
direct: House room assignment already emits private rationale in Mingle; House Strategy Bible and producer briefs are private producer/debug state.
Rationale
This is where The House shines: it curates scenes, records why, and stays out of declaring loyalty as truth.
Downside
Reason labels can leak design intent if exposed. Keep them producer/debug-only during live games.

R8. Visibility and MCP Boundary

Rule
Alliance sidecar state is member-safe and producer-safe. Public live views may show coarse status only when intentionally revealed; they do not expose membership or huddle transcript. Votes remain public in the current rules. Production MCP can read scoped alliance summaries later, but it must not propose, accept, schedule, or dissolve active-match alliances.
Axis
Privacy, API, MCP, and public viewer boundaries.
Basis
direct: CONCEPTS.md:175 and game-mcp/server.ts:106-122 keep active-match actions out of user-facing MCP.
Rationale
The alliance feature is only fun if hidden coordination remains hidden. Tool convenience is not worth punching a spoiler hole in the game.
Downside
More projections and tests. The alternative is production confetti with a security badge.

1. Consent Artifact Ledger

Confidence 92% Complexity Medium
Description
During Mingle, a player can propose a named alliance with a name, invited members, a short charter or objective, and a response deadline. Responses are explicit artifacts: accept, decline, or counter. Once every invited member accepts compatible terms, acceptance is the activation event and an allianceId exists. The fact recorded is mutual consent, not true loyalty.
Axis
Alliance lifecycle and activation.
Basis
direct: packages/engine/src/canonical-events.ts:41-62 has no alliance event types; packages/engine/src/types.ts:358-378 has no alliance action; packages/engine/src/phases/mingle.ts:731-759 already has a bounded private Mingle intent seam. external: Diplomacy-style non-binding commitments matter because the agreement creates social debt without enforcing behavior.
Rationale
This gives the system one small truth object that prompts, replay, API, and postgame can reference without pretending House hypotheses or vote cohorts are confirmed alliances.
Downsides
Needs new event/schema semantics and deterministic response resolution. The product copy must be blunt that accepted alliance means "we agreed," not "we are loyal."

2. Alliance Sphere Huddle Windows

Confidence 74% Complexity High
Description
Accepted alliances get a durable sidecar-native sphere: private thread, notebook, receipts, and House-scheduled huddle windows. Mingle forms or mutates the alliance, but the alliance sidecar is the persistent home. Conversation opens only when The House grants pre-vote, pre-Council, activation, repair, or endgame-reckoning huddle time.
Axis
Private-room substrate and message/thread model.
Basis
direct: packages/engine/src/transcript-logger.ts:129-146 logs private Mingle messages with room context; packages/api/src/db/schema.ts:223-236 stores transcript scope, recipients, room_id, and metadata; mingle.ts:802-836 already has beat-based private room turns.
Rationale
Alliances need a place, not only a label. The product place should be the Alliance sidecar; the implementation can reuse Mingle's private-room machinery where sensible without making the concept a persistent Mingle room.
Downsides
This is a real gameplay-system expansion: huddle windows, House scheduling, receipts, and sidecar state all need explicit rules. The upside is that the rules are now crisp enough to plan.

3. Alliance Notebook Capsules

Confidence 88% Complexity Medium
Description
Each active alliance produces compact member-specific prompt context: name, members, obligations, current ask, last strain, promise debt, and revise trigger. Full transcript history stays in producer/debug evidence. The agent sees the sharp delta, not the gossip archive.
Axis
Durable alliance facts, prompt memory, summaries, and tightness.
Basis
direct: packages/engine/src/context-builder.ts:361-431 and packages/engine/src/agent.ts:3209-3371 already favor compact structured context and stale-fact override rules; strategic reflection artifacts are private/producer-safe rather than raw hidden reasoning replay.
Rationale
This is the leverage move: alliances only matter if agents remember them cheaply and usefully in later social decisions.
Downsides
The capsule can become stale or overconfident if it lacks "last confirmed" and revise-trigger fields. It also needs careful subject scoping.

4. Receipt-Vector Tightness

Confidence 86% Complexity Medium
Description
Model tightness as evidence dimensions: accepted commitments, repeated room contact, promises kept or broken, contradictions, rescue attempts, recency, and member-stated confidence. Avoid one opaque magic score, and never treat shared votes alone as confirmed alliance truth.
Axis
Durable alliance facts, prompt memory, summaries, and tightness.
Basis
direct: packages/api/src/game-mcp/read-model.ts:588-596 separates derived vote cohorts from confirmed alliances; packages/api/src/services/postgame-analysis.ts:314-345 already analyzes cohesion-like signals; docs/reasoning-transcript-observability.md:178-181 treats private decision evidence as first-class for analysis.
Rationale
Receipts make alliance health explainable. That helps prompts, producer inspection, and postgame storytelling without smuggling fake certainty into public play.
Downsides
Evidence weights can become pseudo-science if over-polished. Prefer visible dimensions and short explanations over a single authoritative grade.

5. Trust-Boundary Alliance Read Models

Confidence 90% Complexity Medium
Description
Derive separate member-safe, public-safe, and producer-safe projections from the same alliance facts. Members can see their own proposals, responses, active alliances, and evidence. Public live viewers see nothing private. Producers can inspect richer evidence. Production MCP stays read-only for active games.
Axis
API/MCP visibility and authorization boundaries.
Basis
direct: packages/api/src/services/mcp-scope-policy.ts:1-55 defines scoped access; packages/api/src/game-mcp/server.ts:106-122 explicitly bans active-match actions; packages/api/src/game-mcp/read-model.ts:273-303 resolves visibility modes.
Rationale
Alliance state is spoiler-prone by default. The architecture should make leakage difficult rather than relying on every future caller to remember what is private.
Downsides
More projections means more tests and more DTO discipline. That is still cheaper than cleaning up a hidden-state leak later.

6. House-Scheduled Alliance Sidecar

Confidence 76% Complexity Medium
Description
Expose pending proposals, active alliance cards, huddle schedules, notebook snippets, tensions, producer reasons, and room transcripts in a private sidecar or inspector surface. The center-stage watch theater stays about game flow, not alliance administration.
Axis
Frontend watch/private UI sidecar.
Basis
direct: packages/web/src/app/games/[slug]/components/match-watch-shell.tsx:145-168 and :510-555 show existing side rails and inspector affordances; packages/web/src/app/games/[slug]/game-viewer.tsx:870-1290 already separates mobile tabs and desktop stage/HUD structure.
Rationale
The UI should make alliances legible without turning MatchWatch into an admin dashboard. Sidecar placement also gives The House a visible producer/debug workspace for huddle tickets, reasons, and receipts.
Downsides

7. Counteroffer Lineage and Automatic Decay

Confidence 82% Complexity Medium
Description
Counteroffers create proposal lineages instead of editing the original. A pact can rename itself, change members, narrow purpose, set a one-round trial, or fork into a smaller pact. If an active alliance goes untouched or contradiction receipts pile up, it drifts toward fracturing and queues a recommit-or-exit moment for the next Mingle.
Axis
Alliance lifecycle and activation.
Basis
direct: packages/engine/src/game-runner.types.ts:469-480 already contains private lifecycle vocabulary such as forming, active, fracturing, and retired; packages/engine/src/house-interviewer.ts:877-897 tracks promises, tensions, and alliance pressure.
Rationale
Negotiation history is strategy evidence. Decay keeps alliance memory from fossilizing after eliminations, betrayals, or silence.
Downsides
This is too much for the first cut unless the proposal ledger is already solid. Decay rules must feel like social pressure, not hidden mechanical punishment.
