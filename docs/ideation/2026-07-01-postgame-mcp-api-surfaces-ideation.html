Grounding Context

Recommendation: add denormalized read-model DTOs on top of the canonical event stream and completed-results projection. Do not replace canonical events, do not make transcripts authoritative, and do not make private reasoning player-visible by accident.

The repo already has the core truth substrate this v0 needs. `packages/engine/src/completed-game-results.ts` builds a canonical-event-first completed-results read with winner, finalists, elimination order, per-round revealed facts, endgame eliminations, jury vote counts, jury ledger, player placements, vote patterns, and diagnostics. `packages/engine/src/revealed-round-facts.ts` already turns per-round canonical events into sanitized vote, power, Council, and player-status facts.

The production MCP surface in `packages/api/src/game-mcp/server.ts` already exposes scoped game inspection tools: `list_games`, `read_projection`, `read_round_facts`, `filter_events`, `player_timeline`, cognitive artifact reads, and producer-only trace/durable-run tools. That scope split is healthy. `games:read` is sanitized and subject/access controlled; `producer` exposes private trace evidence. The postgame gap is that current game reads are still too close to facts and events. An LLM can get the pieces, but still has to assemble the chessboard with oven mitts.

The completed-results product requirements in `docs/brainstorms/2026-06-26-completed-game-results-review-requirements.md` make two constraints especially relevant: canonical facts come first, and vote grouping is allowed only as legibility, not as definitive alliance truth. This proposal carries that into MCP: player-safe tools can say "these players voted together in these rounds" and "bloc evidence confidence: medium"; they should not say "confirmed secret alliance" unless producer-only evidence supports it.

External protocol guidance points the same direction. MCP tools are model-invoked functions with input schemas and optional output schemas, tool results can return `structuredContent`, and servers must enforce access controls and sanitize outputs. MCP resources are useful for app-driven context, but these postgame answers are better as tools because the model needs to ask specific questions. OpenAI Apps SDK guidance also favors explicit read-only annotations and structured outputs. In short: structured, read-only, compact, scoped. Very boring. Very correct.

Fixture note: `edge-smoke-dusk`, `Lilith Voss`, `Shadowtech`, and `Kestrel` were not found as checked-in fixture names. Acceptance should add a deterministic fixture or DB snapshot named `edge-smoke-dusk`; relying on a live row would make tests theatrical in the bad way.

I1. Shared Postgame Analysis Projection

Confidence 92%Complexity Medium

Build one shared `buildPostgameAnalysisProjection(...)` helper that takes trusted canonical events, persisted projection status, completed-results output, optional agent/profile rows, and optional producer evidence facets, then emits compact DTO-ready facts. Every v0 MCP tool and API endpoint should read from this helper instead of each tool re-deriving boot order, vote counts, placement, risk, and jury facts. This is the spine. Without it, the tools will slowly become six tiny truth engines, which is how software starts doing interpretive dance in production.

Axis
Whole-game postgame brief
Basis
direct: `buildCompletedGameResults` already rolls canonical events into public postgame facts; `ProductionGameMcpReadModel` already wraps game reads for MCP. The missing move is a shared postgame projection specialized for LLM payloads.
Rationale
The user's target calls need overlapping facts: player refs, placements, vote ledgers, majority alignment, jury votes, exposure pressure, and diagnostics. A shared helper keeps those facts consistent and keeps token-efficient DTOs cheap to add.
Downsides
Requires careful naming and tests so this does not become a second projection model. The helper must remain rebuildable from canonical facts and must not hide degradation.

I2. Agent Game Ledger

Confidence 88%Complexity Medium

Add `list_agent_games(agentId | agentName, limit?, cursor?, detailLevel?)` as the agent-centric entry point. Current MCP `list_games` is game-centric, which is fine for browsing but awkward for "what has Lilith Voss played?" The ledger should resolve an owned/visible agent, return completed and optionally active games, and include outcome facts directly in each row.

Axis
Agent game discovery
Basis
direct: STRATEGY emphasizes repeat joiners, agent improvement after games, and AI app compatibility. The current MCP surface has `list_agents` and `list_games`, but no joined, outcome-rich agent history.
Rationale
This is the cheapest way for an LLM to answer "Which games has this agent played?" and then choose one game to analyze.
Downsides
Agent-name resolution can be ambiguous. v0 should return `resolutionCandidates` when names collide instead of guessing.

I3. One-Call Game Brief

Confidence 91%Complexity Medium

Add `read_game_brief(gameIdOrSlug, detailLevel?, includeEvidence?)` as the primary "Analyze this game" tool. In standard detail, it should fit comfortably in one tool result while carrying enough structure for an LLM to cite winner, final vote, boot order, dominant empowered players, most exposed players, unanimous or near-unanimous votes, major eliminations, endgame sequence, and confidence diagnostics.

Axis
Whole-game postgame brief
Basis
direct: the user asked for ChatGPT to answer "Analyze edge-smoke-dusk" in one or two calls, and existing completed-results DTOs already contain the underlying winner, round, vote, elimination, and jury facts.
Rationale
This should be the top-level answer surface. Most users will not know whether they need projection, round facts, player timeline, or jury tools; `read_game_brief` should make that choice unnecessary for normal postgame analysis.
Downsides
It will be tempting to turn this into a generated recap. Do not. Keep summaries short, deterministic, and labeled when confidence is partial.

I4. Jury Breakdown

Confidence 90%Complexity Low

Add `read_jury_breakdown(gameIdOrSlug, includeEvidence?)`. This tool is intentionally narrow. It should answer "how did the jury vote break down?" with finalists, winner, vote counts, every juror's vote, eliminated round, and derivable relationship flags such as "voted for finalist who eliminated them" or "voted against prior vote partner." The flags must be derivation-labeled, not social clairvoyance.

Axis
Jury and vote accountability
Basis
direct: `CompletedGameResultsJury` already has finalists, ledger, vote counts, winner, and method. The missing data is juror placement/context and a small narrative-hint layer.
Rationale
Jury votes are the highest-value postgame detail. They are also easy for LLMs to botch if they have to infer from raw events.
Downsides
Relationship flags need conservative semantics. Prefer `unknown` over a dramatic but thin label.
{
  "schemaVersion": 1,
  "winner": { "name": "Lilith Voss" },
  "finalists": [{ "name": "Lilith Voss" }, { "name": "Kestrel" }],
  "voteCount": [{ "finalist": { "name": "Lilith Voss" }, "votes": 4 }, { "finalist": { "name": "Kestrel" }, "votes": 3 }],
  "jurors": [{
    "juror": { "name": "Shadowtech" },
    "eliminatedRound": 5,
    "votedFor": { "name": "Kestrel" },
    "relationshipToVote": {
      "votedForAlly": "unknown",
      "votedForBetrayer": "unknown",

I5. Player Game Summary

Confidence 89%Complexity Medium

Add `read_player_game_summary(gameIdOrSlug, player, detailLevel?, includeEvidence?)`. This should replace "call `player_timeline`, then rebuild a personality arc yourself" for normal postgame questions. It should return votes cast by round, votes received by round, empower/expose pressure, Council votes cast and received, powers used, shields received, majority alignment, nomination/at-risk moments, endgame/jury facts, and a compact `readableSummary`.

Axis
Player-level game arcs
Basis
direct: the current `player_timeline` returns canonical events mentioning a player. That is a search result, not a game arc. Completed-results model code already computes votes cast, votes received, tags, and placement-like facts for web cards.
Rationale
This directly answers "Why did this player win or lose?" and "What did a specific player do across the game?" with deterministic math before any narrative color.
Downsides
Majority alignment is derived, not canonical. It needs explicit definitions per vote type: aligned with empower winner, aligned with Council elimination, aligned with endgame elimination, aligned with final jury winner.
{
  "schemaVersion": 1,
  "player": { "id": "p_lilith", "name": "Lilith Voss" },
  "outcome": { "placement": 1, "status": "winner", "eliminatedRound": null, "won": true },
  "voting": {
    "votesCastByRound": [],
    "empowerVotesReceivedByRound": [],
    "exposeVotesReceivedByRound": [],
    "councilVotesCast": [],
    "councilVotesReceived": [],
    "alignmentWithMajorityByRound": [{ "round": 1, "aligned": true, "basis": "council_elimination_target" }]
  },
  "risk": { "timesNominated": 0, "atRiskRounds": [] },
  "powers": { "used": [], "shieldsReceived": [] },
  "endgameAndJury": { "finalist": true, "juryVotesReceived": 4 },

I6. Turning Points and Vote-Bloc Evidence

Confidence 84%Complexity Medium

Add `read_game_turning_points(gameIdOrSlug, detailLevel?, includeEvidence?)` with the enum set from the prompt: `power_shift`, `majority_consolidation`, `alliance_member_cut`, `threat_removed`, `jury_split`, `endgame_pivot`, and `near_miss`. The v0 should use deterministic signals: repeated empowerment, a player suddenly receiving pressure, a close Council vote, a finalist cutting a prior vote partner, endgame eliminations, or a split final jury. It should also include vote-bloc evidence, but label it as derived vote cohesion rather than confirmed alliance membership.

Axis
Turning points and bloc evidence
Basis
reasoned: "Major turning point" is not a single canonical event type. It is a thresholded derivation over vote deltas, power concentration, eliminations, risk, and final-jury consequences. That derivation can be deterministic if it exposes its evidence.
Rationale
This is the right place to answer "what alliance or voting bloc controlled the game?" without inventing secret social contracts. The tool can report who voted together, when their cohesion mattered, and when it broke.
Downsides
The scoring rules need tests and conservative labels. Weak heuristics here would be worse than no endpoint because LLMs will launder them into certainty.
{
  "schemaVersion": 1,
  "turningPoints": [{
    "round": 2,
    "type": "power_shift",
    "playersInvolved": [{ "name": "Shadowtech" }],
    "description": "Shadowtech was empowered again early, concentrating power before the midgame.",
    "evidence": [{ "kind": "empower_tally", "round": 2, "sequence": 42 }],
    "confidence": "high"
  }],
  "voteBlocEvidence": [{
    "rounds": [1, 2, 3],
    "players": [{ "name": "Lilith Voss" }, { "name": "..." }],
    "cohesion": 0.72,
    "controlledOutcomes": ["round_3_council"],
    "label": "derived_vote_cohort",
    "confidence": "medium"
  }],
  "diagnostics": []

I7. Producer Game Analysis

Confidence 82%Complexity High

Add producer-only `read_producer_game_analysis(gameIdOrSlug, detailLevel?, includeEvidence?, maxBytes?)`. This is the one place v0 can use reasoning, thinking, strategy artifacts, private trace facets, vote graphs, trust graphs, House strategy packets, and discrepancy analysis. It should still start from canonical facts and then attach private interpretation. Producer analysis is not a shortcut around the scoreboard.

Axis
Producer-only strategy overlay
Basis
direct: the repo already has `producer` scope, private trace manifest/content/search tools, cognitive artifact access, and docs that distinguish producer private trace data from player-visible facts.
Rationale
Prompt and engine tuning need richer analysis than players should see: strategy pivots, betrayal moments, public/private mismatches, jury-management mistakes, and model behavior observations.
Downsides
High blast radius if it leaks into `games:read`. Tool inventory tests and response redaction tests are non-negotiable.

Recommended V0 Catalog

Implement these as MCP tools first, backed by shared API/read-model methods. Add REST parity where the web UI or external clients need it, but keep the DTO contract shared.

`list_agent_games`

  • Scope: `agents:read` plus game visibility, or `producer`.
  • Inputs: `agentId | agentName`, `limit`, `cursor`, `detailLevel`.
  • Returns: compact game rows with outcome, placement, final vote, finalists, jury vote count, rating delta when available.
  • API: `GET /api/agents/:agentId/games?limit=&cursor=&detailLevel=` for authenticated agent owners.

`read_game_brief`

  • Scope: `games:read` for accessible games, `producer` for global read.
  • Inputs: `gameIdOrSlug`, `detailLevel`, `includeEvidence`.
  • Returns: winner, finalists, final vote, boot order, round/player counts, dominant empowered players, exposed players, major eliminations, endgame sequence, diagnostics.
  • API: `GET /api/games/:idOrSlug/postgame/brief` or fold into `GET /api/games/:idOrSlug/results?view=brief` for public-by-URL completed results.

`read_jury_breakdown`

  • Scope: `games:read` or `producer`.
  • Inputs: `gameIdOrSlug`, `includeEvidence`.
  • Returns: finalists, winner, vote counts, per-juror vote, eliminated round, derivable relationship flags, narrative hints.
  • API: `GET /api/games/:idOrSlug/postgame/jury`.

`read_player_game_summary`

  • Scope: `games:read` or `producer`.
  • Inputs: `gameIdOrSlug`, `player`, `detailLevel`, `includeEvidence`.
  • Returns: placement, status, vote ledger summaries, votes received, majority alignment, risk moments, powers, endgame/jury facts, readable summary.
  • API: `GET /api/games/:idOrSlug/postgame/players/:playerId/summary`.

`read_game_turning_points`

  • Scope: `games:read` or `producer`.
  • Inputs: `gameIdOrSlug`, `detailLevel`, `includeEvidence`.
  • Returns: enum-typed turning points, players involved, evidence fields, short generated-safe descriptions, vote-bloc evidence with confidence.
  • API: `GET /api/games/:idOrSlug/postgame/turning-points`.

`read_producer_game_analysis`

  • Scope: `producer` only.
  • Inputs: `gameIdOrSlug`, `detailLevel`, `includeEvidence`, `maxBytes`.
  • Returns: private strategy pivots, public/private discrepancies, betrayal moments, threat/jury management, grades, model behavior observations, debugging notes.
  • API: `GET /api/producer/games/:idOrSlug/postgame/analysis` with producer auth.

Shared DTO Rules

RuleV0 Contract
SchemaEvery payload includes `schemaVersion: 1` and stable player/game IDs plus display names.
Detail level`brief` omits per-round arrays where possible; `standard` is default; `full` adds bounded supporting arrays but still avoids raw logs.
Evidence`includeEvidence=false` by default. When true, include compact event pointers: `{ kind, round, sequence, eventType }`, not raw envelopes.
DiagnosticsAlways include unavailable/partial facts as diagnostics with codes, severity, and messages. Missing facts are not vibes invitations.
Player-safe boundaryNo source pointers, raw payload envelopes, private trace metadata, `thinking`, `reasoningContext`, storage keys, or producer-only artifacts.
Producer boundaryProducer tool may include private evidence summaries and references, but still anchors claims to canonical outcomes first.
MCP outputReturn `structuredContent` plus a compact JSON `content[0].text` compatibility copy; add output schemas for core DTOs.
