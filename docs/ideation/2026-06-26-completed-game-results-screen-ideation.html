Ideation: Completed Game Results Screen

Focus: restore the completed-game choice between watching an unspoiled replay and peeking at results, then make the results screen explain what happened across the whole game.

Grounding Context

Product context. Influence is an AI social-strategy spectator loop. The current strategy names "Completed games watched" as a key metric and frames "Replay and sharing" as a track for making completed games easier to watch, understand, and share.

Current completed-game entry behavior. The game page fetches the full transcript when a game is completed or cancelled, then initializes the replay index at the end. The route decision sends completed games with transcript data straight into replay mode through MatchWatchShell. That explains the current user-visible problem: the only natural way to reach the result is to consume the whole replay surface first.

Available structured truth. The durable projection already summarizes vote state, including empower votes, expose votes, council votes, endgame elimination votes, jury votes, empowered player, candidates, power action, accepted eliminations, and winner. Canonical events cover the event-level details that a results screen needs: power candidate resolution, council elimination, player eliminated, endgame elimination, jury vote, and winner determined.

Privacy and evidence boundary. Existing docs are explicit that canonical events are accepted board facts, while private agent-turn records and thinking explain strategy but do not prove votes, eliminations, powers, or winners. The results screen should use canonical projection as the fact layer and use transcript/thinking/strategy as optional color only when the source boundary is clear.

External context. Survivor season pages show how a compact voting-history matrix can carry an entire social-strategy season: eliminated player, vote count, and each voter by round. Recent social-deduction research also emphasizes structured memory, social graph analysis, transcript analysis, and role/vote inference as useful lenses for multi-agent social games.

Topic Axes

  • Entry and spoiler control: how a completed game lets a viewer choose "watch unspoiled" or "see results".
  • Canonical game facts: winner, elimination order, round outcomes, powers, votes, and jury result.
  • Vote and alliance legibility: compact patterns that help viewers infer blocs, flips, and shared targets.
  • Agent-level review and learning: per-agent paths, decisions, thinking/strategy snippets, and owner-safe improvement hooks.

1. Results Entry Gate

Entry and spoiler control Conf 88% Cx Medium
Description
When a completed game route loads, show a compact choice screen before revealing spoilers: "Watch replay without results" and "See results". Let query params deep-link to either mode, such as ?view=replay or ?view=results, so list cards, dashboard modules, and shared links can intentionally choose the destination.
Basis
direct: game-viewer.tsx already detects completed/cancelled games, fetches the transcript, and treats them as replay. match-watch-model.ts classifies completed games with messages as mode: "replay".
Rationale
This restores the prior choice behavior without disturbing the replay theater. It also protects the viewer who wants to watch the whole match from accidental winner exposure.
Downsides
Requires new route/view state and careful link behavior so "Replay" never flashes result data during hydration.

2. Canonical Results Atlas

Canonical game facts Conf 86% Cx Medium
Description
Add a results read model that returns one product-shaped object for completed games: winner, finalists, elimination order, per-round vote ledgers, power outcomes, council outcomes, endgame elimination votes, jury votes, source status, and diagnostics. The screen should render from this result atlas rather than mining transcript prose in the browser.
Basis
direct: DurableVoteStateSummary already includes empower, expose, council, endgame, and jury votes. summarizeCanonicalProjection already emits accepted outcomes and winner. The current GET /api/games/:id returns only winner-level detail plus watchState.
Rationale
This is the foundation that makes every richer visual honest. It also lets older or degraded games show a clear "best available" state rather than pretending all result dimensions exist.
Downsides
Some projection data is current-state oriented, so a full per-round results atlas may need to fold over canonical events directly rather than use only the final summary.

3. Vote Matrix with Coalition Stripes

Vote and alliance legibility Conf 82% Cx Medium
Description
Render a Survivor-style vote history matrix: rows are agents, columns are rounds or decision moments, cells show empower/expose/council/endgame/jury votes, and row ordering clusters agents with similar voting records. Use soft background stripes or brackets to show inferred blocs for a round without naming them as confirmed alliances.
Basis
external: Survivor voting-history tables compactly show eliminated player, vote count, and each voter by episode. direct: canonical events expose the same raw ingredients for Influence votes and juries.
Rationale
This directly answers "who voted for who" and gives viewers alliance hints by alignment rather than by speculative copy. It can fit above the fold on desktop and scroll horizontally on mobile.
Downsides
Dense tables can become hard to read on mobile. Needs careful cell labels, sticky first column, and a fallback list view.

4. Elimination Timeline

Canonical game facts Conf 84% Cx Medium
Description
Show a compact vertical timeline of every elimination: round, phase, eliminated agent, method, candidates or finalists, vote split, empowered tiebreaker if relevant, and a one-line "why this mattered" beat. The final row becomes the winner announcement with jury vote count or final method.
Basis
direct: canonical event types include council.elimination_resolved, player.eliminated, endgame.elimination_resolved, and jury.winner_determined, each with structured method/tally fields.
Rationale
Elimination order is the lowest-friction answer to "what happened" and gives the vote matrix a narrative spine. It also avoids making users scrub the replay for major state changes.
Downsides
The "why this mattered" line should be derived conservatively from structured facts or House summaries. A generated explanation could overclaim causality.

5. Alliance Graph with Confidence Labels

Vote and alliance legibility Conf 64% Cx High
Description
Create an inferred relationship map that connects agents based on shared votes, repeated room contact, mutual protection, co-targeting, and strategy-card language. Edges must be labeled as evidence-backed hints, such as "voted together twice" or "shared Mingle room R3", not as definitive alliances unless the game has a typed alliance source later.
Basis
direct: existing docs warn that canonical facts prove votes and eliminations, while thinking and strategy explain posture. The agent strategy observability spine already separates public transcript, private agent turns, and canonical events.
Rationale
This is the highest-ceiling way to help viewers infer groups, because it makes coalition structure visible without requiring the viewer to mentally compare every vote row.
Downsides
High risk of overclaiming. Needs explicit confidence labels, evidence chips, and a conservative v1 that starts with vote similarity before adding transcript or cognitive signals.

6. Agent Result Cards

Agent-level review and learning Conf 70% Cx Medium
Description
Give each agent a compact postgame card: finish placement, elimination round or winner status, votes cast, votes received, shared-vote alignment, power/council/jury involvement, and selected thinking or strategy snippets. For owned agents, the card can later branch into "improve this agent" when product support exists.
Basis
direct: public watch intelligence already returns selected-agent thinking, whitelisted strategy fields, visible transcript thinking, and revealed fact receipts for current watch context. Strategy docs say agent improvement depends on seeing why an agent acted, not just what happened.
Rationale
It turns the results screen from a spectator recap into the product's agent-development loop. It also gives viewers a natural drill-down when the overview raises "why did they do that?"
Downsides
Needs owner/public visibility rules to stay crisp. Public results can show facts and safe thinking; owner-only reasoning remains outside this screen unless a separate authenticated path is designed.

7. Final Scene Results Link

Entry and spoiler control Conf 90% Cx Low
Description
When replay reaches the final scene, replace the dead-end pause with a small final-results card: winner, final vote or method, and a clear link to "View full results". This is the companion to the entry gate: watch-first viewers remain unspoiled until the replay ends, then get a deeper postgame path.
Basis
direct: DramaticReplayViewer already stops playback when the last scene completes. The websocket path already has a game_over event that triggers winner-announced audio.
Rationale
This fixes the current "you watched it all, now what?" moment and gives the richer results screen a natural CTA without changing replay pacing.
Downsides
Small change, not sufficient by itself. It only shines if the destination is the richer results atlas.
