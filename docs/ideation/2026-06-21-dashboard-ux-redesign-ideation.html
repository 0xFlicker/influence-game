Dashboard UX Redesign

A ranked idea set for making `/dashboard` feel less like a pile of useful objects and more like a personal command center for the agent-owner loop.

Date: 2026-06-21 Topic: dashboard-ux-redesign Focus: UX redesign of `/dashboard` Mode: repo-grounded 40 raw ideas, 7 survivors

Grounding Context

Influence is currently framed as an agent development and spectator loop: users create and tune persistent competitors, watch games expose behavior, and eventually improve agents after games. The signed-in dashboard is authenticated, sits in a single `max-w-4xl` column, and currently stacks account stats, a full open-games browser, game history, and a saved-agent preview.

The page has useful ingredients, but the composition mixes four jobs: personal status, open-game discovery, replay/history review, and saved-agent inventory. Existing dedicated surfaces already exist for `/games`, `/games/free`, `/dashboard/agents`, and `/dashboard/profile`, so the dashboard redesign should clarify what the user should do next rather than replicate every focused page.

External UX anchors: NN/g's visibility-of-system-status heuristic supports making current status and next steps obvious; dashboard pattern research frames dashboards as curated at-a-glance lenses rather than all-purpose data dumps.

IDEA 1

Mission-Control Overview

Axis: Next action and personal status Confidence: 86% Complexity: Medium

Turn the top of `/dashboard` into a state-aware overview that answers, in order: what is happening with my agents, what can I do next, and what should I review? The page would keep the existing ingredients but recompose them into a compact personal status band, one primary next action, and three short modules: active/live, recent result, and agent bench.

Basis

direct: `dashboard-content.tsx:318-371` stacks Open Games, Your History, and Your Agents; `games-browser.tsx:493-554` brings full filters into the dashboard; external: NN/g status visibility says users need current system state to decide next steps.

Rationale

This directly addresses the user's "too much stuff" reaction without discarding useful surfaces. It gives the dashboard a job: orient the signed-in owner and route them to the right action.

Downsides

Needs a small personal-status composition layer and careful priority rules so the "one next action" is helpful rather than bossy.

IDEA 2

Owner Loop Tabs

Axis: Cross-page navigation Confidence: 80% Complexity: Medium

Organize the dashboard around four owner intents: Prepare, Play, Review, Improve. Prepare shows saved agents and create/edit entry points; Play shows joinable/open or queue status; Review shows recent history and replay; Improve highlights post-game agent edits when available.

Basis

direct: `STRATEGY.md` frames Influence as users creating and tuning persistent competitors, then watching games expose behavior; `dashboard-content.tsx:352-370` already links to agent creation/management; `dashboard-content.tsx:46-95` already links history to replay.

Rationale

The labels map to the product's actual loop, not generic account-dashboard categories. This gives collapsible/navigation structure a meaning that can survive future features.

Downsides

"Improve" may be thin until post-game reasoning/edit flows exist, so the tab may need an empty or disabled state at first.

IDEA 3

Open Games Preview, Not Full Browser

Axis: Open-game discovery and join flow Confidence: 82% Complexity: Low

Replace the full embedded `GamesBrowser` on the dashboard with a curated preview: a few joinable games, a live game to watch if present, and a single "Browse all games" link. Keep filters, status tabs, tier select, and long list behavior on `/games`.

Basis

direct: `/dashboard` embeds `GamesBrowser` with `compact={false}` at `dashboard-content.tsx:328-331`; `GamesBrowser` renders search, status tabs, tier and track selects at `games-browser.tsx:493-554`; `/games` already exists as the broader browse surface.

Rationale

This is the smallest high-leverage decluttering move. It preserves discovery while letting the dashboard stay personal and action-oriented.

Downsides

If open-game volume is tiny, a preview may feel redundant with `/games`; it needs smart empty states and a clear fallback link.

IDEA 4

Progressive Dashboard States

Axis: Next action and personal status Confidence: 76% Complexity: Medium

Make the dashboard adapt to owner maturity. A new user sees "create an agent" and "join the free queue"; an active competitor sees queue/live/open-game status; a returning owner sees recent results, replay, and agent improvement prompts. Advanced lists stay available, but they are not all front-loaded.

Basis

direct: current empty states already distinguish no history and no agents at `dashboard-content.tsx:47-51` and `dashboard-content.tsx:150-162`; `/games/free` has richer signed-out, queued, no-agent, and selected-agent states at `free-game-content.tsx:124-220`; external: dashboard research highlights tailoring to user needs as desirable and feasible.

Rationale

The dashboard can feel simpler without hiding power-user affordances permanently. The user only sees the complexity that matches their current relationship to the game.

Downsides

Requires clear state precedence and regression coverage so users do not lose access to expected sections when data changes.

IDEA 5

Agent Bench Snapshot

Axis: Saved-agent inventory and management Confidence: 78% Complexity: Low-Medium

Reframe saved agents as a playable bench rather than a mini inventory list. Show the top few active agents with performance, persona, and the next relevant action: queue, join, edit, or manage. The full CRUD page remains `/dashboard/agents`.

Basis

direct: the dashboard previews only three agents unless searching at `dashboard-content.tsx:189-222`; the saved-agent page already handles create/edit/delete at `agents-content.tsx:82-182`; strategy tracks "Agents created" and "Agent edits after game" as product metrics.

Rationale

This makes agents feel like the user's core objects, not a sidebar afterthought. It also attaches cleanly to the soft-delete contract without requiring a new management page.

Downsides

Needs careful copy so "bench" or equivalent product language feels natural for social-strategy agents, not sports cosplay unless that voice is intentional.

IDEA 6

Recent Result to Improve CTA

Axis: History, replay, and learning loop Confidence: 74% Complexity: Medium

Add a post-game card that turns the latest result into a learning prompt: replay the game, inspect how the agent did, and edit the agent. It should appear above or beside history rather than requiring the user to infer improvement from a table row.

Basis

direct: history rows currently offer a `Replay` link at `dashboard-content.tsx:82-88`; strategy explicitly lists "Agent edits after game" as a planned metric and connects reasoning access to improvement.

Rationale

This is the dashboard move most aligned with Influence's long-term differentiation: games are not just outcomes, they are evidence for improving the agent.

Downsides

May need additional API shape to map results cleanly back to editable agent profiles, especially when profiles are later soft-deleted or edited after a game.

IDEA 7

Dashboard Local Nav With Section State

Axis: Cross-page navigation Confidence: 68% Complexity: Medium

Add a compact local nav or segmented control inside `/dashboard` for Overview, Games, History, and Agents, with remembered section state on the client. This directly addresses the "too much stuff" feeling while keeping current sections reachable.

Basis

direct: the top nav links to Dashboard and Profile for signed-in users but not Agents at `nav.tsx:55-64`; the dashboard's three major sections are vertically stacked at `dashboard-content.tsx:318-371`; user context explicitly suggested collapsible sections or nav.

Rationale

This is the most literal response to the observed problem and can compose with stronger ideas above. It becomes more valuable if paired with Mission-Control Overview or Owner Loop Tabs.

Downsides

By itself, this mostly organizes clutter; it does not decide what matters. It should be treated as a supporting interaction pattern, not the whole redesign.
