What the code says today

Grounding Context

The current implementation expresses authorization through two MCP resource profiles: /mcp with scope=games for player-facing access, and /mcp/producer with scope=mcp plus the producer role for admin/developer tools. That split leaks into OAuth metadata, consent copy, MCP tool descriptors, dispatch checks, database constraints, setup instructions, README text, and concepts docs.

Current resource model

packages/api/src/services/mcp-oauth.ts maps profiles to endpoint paths and issues one exact profile scope. Resource path is carrying capability meaning.

Current consent model

authorize-client.tsx previews one grant and submits approve, deny, or cancel for the original request. Users cannot trim write-like access today.

Current tool model

Agent read, agent write, game read, queue enrollment, and producer trace tools are split by profile buckets rather than granular required scopes.

Current storage model

Client, authorization-code, access-token, and refresh-token rows constrain scopes to old names. Multi-scope grants need real set semantics, not string folklore.

Old surface Problem Refresh target
/mcp + games Reads games, reads agents, writes agents, and queue mutations are advertised as one generic game permission. Use one /mcp resource and explicit games:read, agents:read, and agents:write scopes.
/mcp/producer + mcp Admin access is encoded as a second endpoint and an opaque scope name. Delete the producer endpoint and use producer as a role-gated capability on the canonical endpoint.
All-or-nothing consent AI apps may ask broadly, but users cannot reduce grants before approving. Render an editable permission sheet with default-selected grantable scopes and selectable write scopes.

Prose is authoritative: endpoint collapse removes URL-shaped authorization semantics; tokens carry the capability set.

Exploration coverage

Topic Axes

These five axes kept the ideation from only solving the route rename while missing the consent, provider, and enforcement consequences.

Axis 1

Scope and resource model. What the canonical endpoint, scope set, producer capability, and token lifetime rules should become.

Axis 2

OAuth consent and user agency. How a user sees, edits, blocks, and approves the scopes an AI app requested.

Axis 3

Tool inventory and enforcement. Which MCP tools require read, write, game, or producer access, and where denial happens.

Axis 4

Provider/client compatibility. How broad AI app requests, dynamic registration, metadata, and omitted resource indicators behave.

Axis 5

Documentation and setup coherence. How setup screens, README, concepts, tests, and examples stop advertising stale nouns.

Ranked Ideas

The ranked set intentionally favors ideas that compose into one cutover. The best implementation path is not seven unrelated features; it is a scope contract, a registry, an editable consent UI, and a validation matrix that keep each other honest.

I1. Single Endpoint Scope-Set Contract

Replace the two-profile resource split with one canonical /mcp resource.

Rank 1
Confidence: High Complexity: Medium Axis: Scope and resource model
Description
Authorization issues a normalized set of granted scopes from agents:read, agents:write, games:read, and producer. The producer scope is grantable only when the logged-in user currently has the producer role.
Why it matters
This is the actual model refresh. It removes URL-shaped authorization semantics and makes the token say what the client can do.
Evidence and tradeoffs

Direct evidence: packages/api/src/services/mcp-oauth.ts currently maps auth profiles to endpoint paths, and the user explicitly requested a single endpoint with no backwards compatibility.

Tradeoff: schema, metadata, route registration, docs, tests, and client setup need to move together. Existing producer clients pointed at /mcp/producer intentionally break.

I2. Registry-Driven Scope Matrix

Make one typed registry define scopes, copy, role gates, dependencies, refresh policy, and tool requirements.

Rank 2
Confidence: High Complexity: Medium Axis: Tool inventory and enforcement
Description
Add a typed scope registry and generate protected-resource metadata, consent rows, descriptor security schemes, dispatch gates, docs/test fixtures, and audit labels from that registry.
Why it matters
Granular scopes only work if metadata, consent, tool discovery, and enforcement share one source of truth.
Evidence and tradeoffs

Direct evidence: packages/api/src/game-mcp/server.ts currently builds descriptors and tool buckets from one profile-level scope. Agent reads and writes sit under the same old games scope.

Tradeoff: this adds a small abstraction early. It buys back clarity by preventing scope string drift across server, web, docs, and tests.

I3. Editable Consent Permission Sheet

Let the user approve a selected subset instead of rubber-stamping the app's requested wishlist.

Rank 3
Confidence: High Complexity: Medium Axis: OAuth consent and user agency
Description
Change inspect preview to return requested, grantable, default-selected, blocked, and dependency-linked scopes. Render checkboxes grouped as Agents, Games, and Developer. Normal users do not see grantable producer controls, and users can uncheck write access before approving.
Why it matters
AI apps can ask for everything, but normal users grant only the meaningful subset they are comfortable delegating.

Prose is authoritative: approval submits the selected scope set, and the API rejects empty or dependency-invalid selections safely.

Evidence and tradeoffs

Direct evidence: the current authorize client submits approve, deny, or cancel for the original request and renders one raw scope row. Editable consent needs API support, not only frontend checkboxes.

Tradeoff: the redirect-safe error path becomes more important because a user can now approve a subset that invalidates dependencies.

I4. Honest Agent/Game Tool Split

Make MCP tool availability match the delegated permission, especially for agent writes.

Rank 4
Confidence: High Complexity: Medium Axis: Tool inventory and enforcement
Description
Map list_agents, get_agent, search_agents, and archetype reads to agents:read. Map create_agent and update_agent to agents:write with an agents:read dependency. Map game projection, event, facts, and cognitive reads to games:read. Producer trace/global tools require producer.
Why it matters
This is where the new scopes become product permissions instead of pretty OAuth wallpaper.
Evidence and tradeoffs

Direct evidence: the current server routes list, search, get, create, and update agent tools in one non-producer bucket. Docs currently list agent management under scope=games.

Tradeoff: queue tools need an explicit decision. Treat join_queue and leave_queue as agents:write pre-match enrollment, or hide/defer them until a future write scope exists. Do not leave state mutation under games:read.

I5. Multi-Scope Token Storage and Refresh Policy

Store normalized granted scopes as a set while still emitting OAuth's space-delimited scope string.

Rank 5
Confidence: Medium Complexity: Large Axis: Scope and resource model
Description
Store normalized granted scopes on clients, authorization codes, access tokens, and refresh tokens. Refresh tokens are issued only for non-producer grants; any grant containing producer stays short-lived and browser-authorized.
Why it matters
The storage layer has to understand sets, dependencies, and refresh eligibility. Otherwise multi-scope authorization turns into fragile string parsing.
Evidence and tradeoffs

Direct evidence: packages/api/src/db/schema.ts constrains current client, code, access-token, and refresh-token scopes to the old vocabulary.

Tradeoff: this is the highest-migration item. Since backwards compatibility is explicitly out of scope, it can be a clean constraint replacement instead of a bridge.

I6. Broad Ask, Narrow Grant Provider Compatibility

Let provider-packaged AI apps request broadly while the authorization flow grants narrowly.

Rank 6
Confidence: Medium Complexity: Small Axis: Provider/client compatibility
Description
Advertise all supported scopes in authorization-server metadata and allow dynamic clients to register broad supported sets. Consent filters by user and selection. tools/list reflects only granted scopes. With one /mcp resource, omitted resource resolves to the canonical resource.
Why it matters
This keeps provider-packaged AI apps working when they ask for everything, without giving everything.
Evidence and tradeoffs

Direct evidence: current docs already tolerate broad client registration because some clients register every advertised scope. External OAuth and MCP guidance expects supported scopes and resource metadata to be discoverable while server-side authorization still decides what the token actually receives.

Tradeoff: public metadata can mention producer, but the consent layer must make clear that producer is not grantable for normal accounts.

I7. No-Compatibility Noun Purge and Validation Matrix

Move constants, docs, setup, tests, and concepts together so the old names do not keep teaching the old model.

Rank 7
Confidence: High Complexity: Medium Axis: Documentation and setup coherence
Description
Replace old scope names and endpoint messaging across constants, migrations, setup commands, consent copy, server instructions, docs, concepts, and tests. Add a table-driven scope matrix for metadata, preview, issuance, tool discovery, allowed and denied calls, refresh behavior, producer role revocation, and audit fields.
Why it matters
Scope refreshes fail when code changes but docs and tests keep advertising the old contract.
Evidence and tradeoffs

Direct evidence: README, docs/game-mcp-production-oauth.md, CONCEPTS.md, /get-mcp, and existing tests still advertise games, mcp, and /mcp/producer.

Tradeoff: this is boring work, which is why it matters. Auth bugs adore stale docs.
