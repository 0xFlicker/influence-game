Codebase Context

Grounding Context

Binding product contract

Privy remains a permanent, first-class login and signup path. Existing Privy identities, wallets, ownership, roles, and history are preserved. Conventional email/password is additive; linking is explicit; migration is optional; Privy is not a legacy lane.

Today the frontend already offers Privy email and wallet login, but Privy email is OTP/passwordless rather than a conventional password. The browser exchanges a Privy token for an Influence-owned seven-day JWT; downstream product API routes consume that internal session, while /api/auth/login is the provider-specific seam. MCP OAuth is separate: the browser session authenticates only the authorization/consent entry, after which MCP authorization codes, access tokens, refresh families, introspection, and resource authorization have their own lifecycle.

The dangerous coupling is identity, not the password form. Newer rows use the Privy subject as users.id, a key referenced throughout ownership and OAuth records. Email is private and non-unique. A second provider therefore needs a linked-credential model and explicit collision policy before it needs a polished modal.

OpenAI does not require conventional passwords. It requires the existing OAuth 2.1/MCP authorization behavior for protected data and actions. That boundary already handles exact callbacks, PKCE, resources, scopes, consent, refresh families, and current role checks; it should remain stable while app login expands.

1Layered Identity Sandwich

Keep direct Privy email/wallet login and signup permanently available. Add a managed conventional-password provider as another credential authority, map both providers to one stable Influence account, and continue issuing the same Influence browser session consumed by product APIs. That session authenticates the MCP authorization/consent entry only; MCP codes, access tokens, refresh families, and resource authorization remain independent artifacts. Privy remains responsible for its linked identities and wallet continuity; the password provider owns password custody and recovery.

Axis
Identity authority and provider topology
Basis
direct: Only the upstream login exchange is Privy-specific; downstream product middleware already validates an Influence JWT, and the MCP authorization entry separately requires that app session. external: Privy supports external JWT/OIDC providers while explicitly not offering conventional passwords.
Rationale
It preserves every working Privy and wallet behavior while offloading the exact security burden the team does not want.
Downsides
Two vendors, non-trivial identity mapping, and provider-specific federation semantics that must be proven before selection.

2Safe Linking Protocol

Make provider subjects linked credentials rather than owners. A matching email or wallet creates a collision case, not an automatic merge. Linking requires live proof of the current account and incoming credential, preserves all existing ownership references, and emits an auditable conservation receipt showing that agents, games, competition history, roles, and OAuth grants were neither orphaned nor transferred accidentally.

Axis
Account linking, migration, and canonical ownership
Basis
direct: Email is non-unique; current login resolves wallet first and then Privy subject; users.id is referenced by many durable ownership records. The public-identity migration test already treats ownership-reference equality as an invariant.
Rationale
A successful login into an empty duplicate account—or an incorrect merge—is more damaging than a rejected login. This makes preservation mechanical.
Downsides
Requires explicit collision UX, support tooling, audit storage, and a difficult policy for two already-populated accounts.
Confidence
95%
Complexity

3Resumable Auth Journey

Build the requested wrapper as a provider-neutral auth coordinator, not a cosmetic chooser. It owns sign-in, explicit signup, invite interruption, email verification, password reset, credential linking, Influence-session hydration, expiry, logout, and return-to-intent. Direct Privy and managed-password paths remain equally visible; product surfaces request an outcome and stop importing provider SDK state.

Axis
Unified sign-in, signup, recovery, and session UX
Basis
direct: Direct usePrivy() and login calls span navigation, gates, permissions, gameplay, queueing, and MCP consent; signup is implicit; invite and expiry behavior are provider-coupled.
Rationale
Two first-class paths need one state machine or the product will accumulate duplicate spinners, lost actions, and provider-specific logout bugs.
Downsides
Large frontend seam; intent preservation and OAuth detours need careful server/browser boundaries. A server-side handoff ticket should be added only if real flows prove ordinary validated state cannot survive.
Confidence
90%
Complexity
High

4OAuth Entry Parity

When ChatGPT sends a user to the Influence OAuth authorization page, render the same expanded auth wrapper used elsewhere. The user can sign in or create an account through either managed email/password or Privy email/wallet, complete any required verification or recovery, and return to the exact pending authorization request before reviewing consent. Both paths resolve to the same Influence account/session at the authorization entry; the existing MCP authorization-code, token, refresh, and resource lifecycle remains separate and unchanged.

Axis
OpenAI OAuth compatibility, security, and privacy
Basis
direct: The current MCP authorization page already gates consent behind an Influence browser session and currently invokes Privy directly. The OAuth request separately binds client, callback, resource, requested scopes, state, and PKCE inputs, so authentication can expand without changing what is ultimately authorized.
Rationale
This is the submission-critical acceptance path: a reviewer arriving from ChatGPT can choose either supported account path, including signup, and still reach the same scoped consent screen without losing their OAuth request.
Downsides
Verification, password reset, invite redemption, popup cancellation, refresh, and multi-tab flows must preserve the original OAuth request safely. Password recovery should revoke affected login sessions without silently disconnecting unrelated Privy credentials or existing MCP grants.
Confidence
95%
Complexity
Medium–High

5Progressive Trust Account

Let users who choose conventional passwords participate normally without wallet literacy, while Privy users continue to receive the full first-class experience. Existing role assignments and access remain intact: a direct Privy wallet login satisfies the current privileged path, and no migration is required. Treat wallet linking as an added capability for password-authenticated accounts; require fresh possession only when linking or changing a role-bearing wallet, or when a password-only session first attempts privileged access.

Axis
Wallet continuity, RBAC, and privileged access
Basis
direct: Walletless users can already hold valid sessions and own ordinary resources, while roles and producer access are wallet-address based. reasoned: stored address association is not equivalent to fresh possession proof.
Rationale
This removes wallet friction from adoption without weakening the existing high-privilege trust boundary or interrupting existing Privy administrators and producers.
Downsides
Requires a real step-up ceremony and careful messaging; it must not make direct Privy users feel demoted or make password users feel incomplete.
Confidence
78%
Complexity
Medium

6Additive Managed-Provider Bake-Off

Evaluate Clerk, Auth0, Stytch, and Supabase only as additive conventional-password authorities beside permanent Privy support. Score the failure journeys first: reset, changed email, MFA, session compromise, account-linking collision, federation into or coexistence with Privy, export, and support burden. The happy-path modal is table stakes, not the selection event; every scored capability must point to provider documentation or a demonstrated spike.

Axis
Identity authority and provider topology
Basis
external: Official documentation establishes Clerk account linking and migration/export, Auth0 database migration and explicit linking, Stytch consumer auth plus crypto-wallet auth, and Supabase password auth, identity linking, and a queryable Postgres identity store. Pricing and support burden remain time-sensitive bake-off inputs rather than fixed conclusions.
Rationale
The provider choice is consequential but should follow the architecture. A narrow bake-off prevents vendor marketing from quietly redefining ownership or MCP authorization.
Downsides
A fair spike costs time, and provider capabilities/pricing drift. Privy federation semantics still need direct validation for each serious finalist.
Confidence
87%
Complexity
Low–Medium

7Provider Exit Before Adoption

Require a credible identity-porting story before selecting the password provider. The team should document, provider by provider, whether subjects, verified email state, password hashes or migration credentials, MFA factors, and lockout state are self-service exportable, support-mediated, or unavailable—then prove that exported credentials can be remapped without changing the Influence account that owns agents, history, or OpenAI grants. Privy remains an independent permanent login and recovery rail throughout.

Axis
Account linking, migration, and canonical ownership
Basis
external: Clerk documents user migration and export, Auth0 documents bulk/trickle migration plus support-reviewed password-hash and MFA-secret export, and Stytch documents a support-mediated password-hash export path; Supabase keeps user/identity records in the project’s Postgres database. reasoned: offloading password security should not create a second permanent ownership trap.
Rationale
Designing exit while leverage is high is cheaper than reverse-engineering identity continuity during an incident or pricing shock.
Downsides
May exclude otherwise attractive providers and cannot eliminate all lock-in around MFA or risk signals.
Confidence
74%
