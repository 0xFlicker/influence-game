Codebase Context

Grounding Context

The repo already has the skeleton for private evidence: `Private evidence manifest` is a named domain concept, `game_evidence_manifests` stores private storage pointers and redaction/access metadata, and `game_evidence_manifest_reads` records read attempts. The existing `game-evidence` service validates that manifests point at the private Linode evidence bucket under `evidence/<gameId>/` and rejects the public profile-picture bucket/prefix, but it does not yet write raw trace content.

Access control is also partly present. `readEvidenceManifest` enforces producer/admin-style permissions and audits denied, expired, redacted, and allowed outcomes. The admin durable-run route deliberately exposes only evidence counts and provider summaries, not manifest IDs, bucket keys, raw prompts, raw responses, thinking, or `reasoningContext`.

The current MCP server is read-only and useful, but it reads local simulation artifacts from disk rather than the API/Postgres durable run state. The active viewer path is similarly split: durable canonical events have per-game sequence numbers and a replayable read model, while live transcript/viewer updates are still mostly in-memory websocket deltas with no durable cursor. Websocket payload sanitization strips private reasoning today, which is the right privacy direction, but catch-up still needs a database-backed observable stream.

Core invariant Raw prompts, model responses, thinking, native reasoning contexts, and tool arguments should become more durable for producers, not more public. Canonical game events, public viewer transcript, and private evidence need linked boundaries, not a shared stream.

Critiqued Direction Set

Ranked Ideas

The strongest path is to build the private evidence writer first, because it creates the trace files, pointer metadata, and boundary vocabulary that the MCP and catch-up work can safely reference.

  • I1. Private Evidence Segment Writer + Envelope v1
  • I2. API-Gated Trace MCP
  • I3. Manifest-Guided Reasoning Trace Search
  • I4. Durable Observable Catch-Up Cursor
  • I5. Triple-Ledger Visibility Contract
I1.

Private Evidence Segment Writer + Envelope v1

Axis: Private evidence capture Confidence: 88% Complexity: Medium

Add an API-owned writer that receives bounded evidence records, writes JSONL/JSON blobs to the configured private bucket, computes hash/size/count metadata, then creates `game_evidence_manifests` rows containing only storage pointers, sanitized counts, source pointers, and boundary facets. The envelope should capture full prompt, raw model response, emitted thinking, native `reasoningContext`, tool args, action name, actor, phase, round, owner epoch, and canonical event boundary.

Basis
direct: `game_evidence_manifests` already has storage pointer, source pointer, retention, access, and redaction fields; `game-evidence` already validates private bucket/key discipline; durable-run tests already assert raw prompts, responses, thinking, and `reasoningContext` stay out of read models.
Rationale
This turns the existing manifest model into a real evidence system while preserving the intended privacy boundary. It also gives later MCP/search work a stable artifact shape to read.
Downsides
Content write plus Postgres manifest insert is not a true atomic transaction, so the writer needs explicit degraded/write-failed handling, retries, and hash validation. Envelope versioning matters immediately because raw trace files will outlive code shape changes.
I2.

API-Gated Trace MCP

Axis: Admin/producer MCP Confidence: 84% Complexity: Medium

Build the producer MCP as a read-only API client with producer/admin credentials, not as a direct database or storage client. In that context, the clear tool split is `list_durable_runs`, `inspect_durable_run`, `list_manifests`, `read_content`, and `search_reasoning_traces`: manifests are the DB index, and content is the raw JSON/JSONL trace payload for a selected manifest. The tools should call API routes that reuse `requireAdminRead`, `readEvidenceManifest`, content-read authorization, purpose logging, output bounds, and rate limits.

Basis
direct: The current MCP is explicitly local and file-backed; `readEvidenceManifest` already enforces the permission/read-audit path the user wants reused. external: The MCP tools spec requires access controls, input validation, output sanitization, rate limiting, and audit logging.
Rationale
This preserves one policy path. It also means the MCP can be safer than a privileged local debug script: every sensitive read gets an API-level authorization decision and an audit row.
Downsides
The API must grow the missing endpoints first. Tool ergonomics will depend on crisp pagination, bounded content output, and purpose strings; otherwise the MCP becomes either too weak or too noisy.
I3.

Manifest-Guided Reasoning Trace Search

Axis: Admin/producer MCP Confidence: 78% Complexity: Medium

Make `search_reasoning_traces` use manifest metadata to find the right trace files, then read matching content through the same authorized API path as `read_content`. The manifest side handles actor, action name, phase, round, canonical boundary, evidence type, envelope version, record counts, token counts, and optional line offsets; the content side returns the actual matching JSON/JSONL records with manifest IDs and positions.

Basis
direct: The manifest table already has `evidenceType`, `sourcePointers`, and `metadata`; the admin durable-run response is count-only today. reasoned: A trace search tool without a manifest index would either scan too much content or be too slow/noisy to trust.
Rationale
This gives producers a useful investigative surface while keeping Postgres limited to pointer and sanitized summary data. It also creates high-quality MCP answers: "here are the matching trace records and where they came from" rather than a vague storage listing.
Downsides
Search quality depends on the writer capturing the right boundary/action fields. Early versions should keep outputs bounded by count/bytes, but the happy path should still be actual trace content, not a fake summary layer.
I4.

Durable Observable Catch-Up Cursor

Axis: Public durable catch-up Confidence: 82% Complexity: High

Persist a public-ish observable stream with an opaque sequence cursor for viewer catch-up: public transcript messages, phase updates, eliminations, game-over signals, and projection hints. Websocket delivery becomes a wake-up mechanism; reconnecting clients call a DB-backed catch-up endpoint with after=<cursor> and receive sanitized entries after that cursor.

Basis
direct: Durable canonical events already have per-game sequence validation, while current live websocket payloads have no cursor and active catch-up comes from an in-memory runner snapshot. external: SSE's `Last-Event-ID` pattern shows the value of opaque reconnect cursors for incremental catch-up.
Rationale
This is the preparation needed for web clients: a client can reconnect, ask for missed public observable state from the DB, and remain insulated from private evidence.
Downsides
This is the broadest survivor. It needs careful mapping between canonical event sequence, public observable cursor, and existing transcript rows so the UI does not double-append or reorder messages.
I5.

Triple-Ledger Visibility Contract

Axis: Visibility boundaries Confidence: 86% Complexity: Medium

Make the architecture explicit as three linked ledgers: canonical events for accepted game truth, public observable transcript/state for viewers, and private evidence for producer inspection. Each ledger has its own cursor and visibility rules, but records can cite each other through source pointers and canonical event boundaries.

Basis
direct: Current concepts already distinguish canonical events, projection/read models, checkpoints, and private evidence manifests; websocket sanitization already strips reasoning fields. reasoned: Collapsing these into one stream would either leak private reasoning or make public catch-up depend on non-public artifacts.
Rationale
This is the mental model that keeps the other work coherent. It names which data is replay truth, which data is viewer UX, and which data is producer-only evidence.
Downsides
It is partly documentation and type discipline, so it can feel abstract unless paired with tests that prove raw evidence cannot appear in public catch-up or durable-run summaries.
