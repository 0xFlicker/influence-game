Ideation

Simulation/API Statefulness Unification

The strongest direction is not "make the server stateless" in the abstract. It is to make each game run a resumable, single-writer workflow whose accepted domain facts, checkpoints, public projections, and raw LLM evidence have separate durable ledgers. Simulation JSONL and API Postgres/S3 storage should become adapters over the same runner emission contract.

Codebase Context

Grounding Context

  • The current risk is explicitly state loss. The statefulness plan says running games hold state in process memory with no mid-game persistence, and API games still lack a durable event table, XState snapshots, runner/agent checkpoints, and resume after process loss.
  • Simulation already has the better artifact spine. Simulator runs write progress, turns, transcripts, and accepted-domain canonical event JSONL; the docs say to replay game-N-events.jsonl through replayCanonicalEvents(...) instead of parsing transcript prose as board state.
  • API lifecycle is still process-local and terminal-persisted. The API uses an in-memory activeGames map, creates new GameRunner(...), streams to WebSockets, then batch-inserts transcript/results after runner.run() completes or writes partial transcript on error.
  • Canonical event identity is close but not unified. GameState can accept a gameId option, but GameRunner currently constructs new GameState(...) without the API games.id; this makes the first production event-store decision an identity decision.
  • The public viewer is doing too much inference. WebSocket snapshots derive phase from the last transcript entry, and structured game facts are parsed with regex helpers. The engine projection already has typed canonical facts and ordered replay guards.
  • Linode S3 exists, but the current helper is not the raw-evidence store. The storage helper is for profile-picture uploads and sends ACL: "public-read". Raw prompts, model responses, reasoning context, and producer records need a private object-storage boundary with Postgres manifests.

Decomposition

Topic Axes

  • Canonical event truth and run identity One accepted event stream per game, with API and simulator emitting equivalent facts.
  • Checkpoint/hydration and single-writer lifecycle Pause safely at known boundaries, resume honestly, and keep XState mutations sequential.
  • Raw LLM evidence and private debug storage Preserve prompts, responses, reasoning, and producer artifacts without exposing them to UX surfaces.
  • Viewer/public projection and catch-up contract Let the UX read typed public facts and sequence cursors rather than transcript prose and process-local WebSockets.
  • Operations/storage migration on Linode/Postgres Use the existing always-on Postgres/Linode shape first, then decide whether heavier workflow infrastructure is warranted.

I1. Top pick

Unified Run Artifact Sink Matrix

Conf 86% Complexity High
Description
Define one runner emission surface for canonical events, transcript entries, agent turns, progress, checkpoints, and outbound viewer notifications. The simulator adapter writes JSONL/text artifacts; the API adapter writes ordered Postgres rows, private object evidence, and outbox/public projection rows. The important move is not a giant abstraction first; it is a shared contract for what a game run emits.
Axis
Canonical event truth and run identity; Operations/storage migration on Linode/Postgres.
Basis
direct: Simulator attachProgressLogger already wires canonical events, stream events, progress, and agent turns into artifact writers. API lifecycle currently only wires the stream listener to ViewerEventPacer and terminal DB writes. external: Event sourcing prior art separates the authoritative event store from projections and downstream distribution.
Rationale
This is the cleanest unification idea because it makes simulation and API games equivalent consumers of one engine emission protocol. It also gives a parity harness: export API rows back into simulator-shaped artifacts and ask the existing MCP/replay tooling the same questions.
Downsides
The name can invite over-abstraction. Keep it as a small interface plus acceptance tests, not a new runtime framework. The API sink must be fail-closed for authoritative events; the current listener pattern is not enough while listener errors can be swallowed.
Confidence
86% - strong direct basis, high leverage, and it directly answers the simulation/API split.
Complexity
High - touches engine emission boundaries, API persistence, simulator artifacts, and validation.
I2.

Postgres Workflow Kernel Before Temporal

Conf 84% Complexity High
Description
Build the first durable workflow kernel in the existing database: game_events, game_checkpoints, game_leases, game_outbox, and game_evidence_manifests. Keep Linode/Postgres as the operational base while modeling the same boundaries a Temporal-style system would eventually require.
Axis
Operations/storage migration on Linode/Postgres.
Basis
direct: The current schema has game status, players, transcripts, results, and memories but no suspended status or event/checkpoint/outbox tables. external: Event sourcing guidance explicitly allows a general-purpose relational database as an event store if the application builds stream ordering, optimistic concurrency, and snapshots.
Rationale
This keeps the migration close to the infrastructure already running. It gives you ordered facts, checkpoint heads, single-writer ownership, and transactional public delivery before adding a queue, Redis, or workflow engine.
Downsides
The team must build some event-store ergonomics itself: unique sequences, replay guards, idempotent consumers, schema evolution, and compaction or snapshot policy. That is still smaller than introducing Temporal before the game event model is proven.
Confidence
84% - strong fit with current storage and with the user's preference to stay on Linode if viable.
Complexity
High - mostly schema and lifecycle work, but it becomes the durability foundation for everything else.
I3.

Canonical Run Identity Adapter

Conf 88% Complexity Medium
Description
Make API games.id the required engine/canonical gameId before the first roster event. Every persisted canonical event should be keyed by (game_id, sequence), and the append path should reject events whose envelope game id does not match the API game row.
Axis
Canonical event truth and run identity.
Basis
direct: GameState already supports options.gameId, but GameRunner constructs new GameState(...) without passing the API id. The API starts a game by loading gameId from Postgres, then constructs new GameRunner(...).
Rationale
Durable event storage is brittle until identity is settled. This is the first small step that makes simulator events, API rows, projection cursors, object manifests, and debug artifacts all address the same run.
Downsides
It forces tests and fixtures to care about game ids earlier. That is a good cost, but it will expose any code that assumed generated internal ids were harmless.
Confidence
88% - narrow, directly supported, and prerequisite to a production event table.
Complexity
Medium - small surface, but it affects event compatibility and snapshot parity.
I4.

Checkpoint Capsule With Projection Cursor

Conf 82% Complexity High
Description
Persist phase-boundary checkpoint capsules keyed to last_event_sequence. A capsule should include the XState persisted snapshot, runner data, token usage, non-replayable private continuity state, and the public/domain projection cursor. First ship forensic checkpoints that can be diffed against live state; only then claim fromCheckpoint() resume.
Axis
Checkpoint/hydration and single-writer lifecycle.
Basis
direct: The statefulness plan already names GameState, canonical event cursor, XState snapshot, GameRunner accumulated data, and TokenTracker usage as checkpoint inputs. external: XState exposes persisted actor snapshots and restore-by-snapshot semantics; Temporal's replay model is useful as a mental model for recording outside-world results rather than recomputing them.
Rationale
This answers the user's pause/resume question honestly: yes, a singleton game can stop safely and continue later, but only at persisted boundaries. It also keeps canonical events as truth and snapshots as a resume optimization, not a replacement event store.
Downsides
Mid-phase resume remains a later problem. In-flight LLM calls and raw provider conversations should not be promised as recoverable; accepted sub-results can be recorded, pending actors can be resumed with recap, and unfinished work can be retried.
Confidence
82% - strongly aligned with the plan, but hydration details need careful proof.
Complexity
High - requires engine serializers, API transaction boundaries, and resume validation.
I5.

Owner Epoch Lease

Conf 80% Complexity Medium
Description
Make the current singleton requirement explicit: one durable owner epoch mutates a game at a time. Before processing, a worker acquires a Postgres lease/advisory lock or lease row, records an owner epoch, and stamps accepted events/checkpoints with that epoch. Parallel LLM calls can happen inside the owner, but accepted state updates append sequentially.
Axis
Checkpoint/hydration and single-writer lifecycle.
Basis
direct: The statefulness plan says horizontal scaling is impossible today because activeGames is process-local, WebSocket pub/sub is per-process, and there is no distributed lock for game ownership. It recommends starting with Postgres advisory locks.
Rationale
This makes the user's intuition precise: event storage gives safe stop/restart and durable catch-up, not safe parallel mutation of the same XState actor. The lease is how the system says "parallel outside, sequential inside" without hand-waving.
Downsides
Leases introduce stuck-owner and timeout policy. Epoch stamping helps reject stale writers, but the recovery path needs conservative states like suspended and explicit operator-visible readiness.
Confidence
80% - direct plan fit and high operational clarity.
Complexity
Medium - the concept is simple, but every append/checkpoint path must enforce it consistently.
I6.

Private Evidence Object Ledger

Conf 83% Complexity Medium
Description
Use private Linode Object Storage for full raw evidence: original prompt, model response, tool-call result, reasoning context, normalized agent turn, and source packet references. Postgres stores manifests: object key, content hash, event/source pointer, visibility, model, token usage, and redaction/export status.
Axis
Raw LLM evidence and private debug storage.
Basis
direct: Simulation docs treat game-N-turns.jsonl and hidden producer records as debug artifacts, not player-visible speech. WebSocket sanitization strips thinking and reasoningContext, and drops agent_turn. The current S3 helper uploads public-read PFP files, so this needs a separate private helper.
Rationale
This matches the user's proposal: Linode S3 for full raw event logs and Postgres for UX-facing events. The split preserves forensic debugging without turning the relational database into a giant hidden-reasoning blob store.
Downsides
Private evidence storage needs retention, encryption/ACL policy, redaction policy, and a producer-only access path. It also needs a clear rule: raw evidence explains accepted events but never becomes public game truth.
Confidence
83% - strongly grounded and a clean fit for object storage.
Complexity
Medium - storage plumbing is tractable, but privacy/access boundaries matter.
I7.

Public Projection and Cursor Wake-Up Contract

Conf 81% Complexity High
Description
Create public projection/event endpoints keyed by sequence cursor. The viewer asks for all public material after cursor N and receives typed phase, vote, room, shield, elimination, jury, winner, and transcript records. WebSocket becomes a wake-up that says "new public material exists after N"; a durable outbox handles delivery and pacing.
Axis
Viewer/public projection and catch-up contract.
Basis
direct: The current viewer derives snapshot phase from the last transcript entry and parses structured game facts with regex helpers. game-projection.ts already enforces same-game and contiguous-sequence replay for canonical events. external: CQRS/event-sourcing patterns use materialized views for query/UI surfaces.
Rationale
This improves user experience before full engine resume exists. Viewers can refresh, reconnect, or move across API instances and catch up from durable public projections instead of hoping the current process still has the right WebSocket queue.
Downsides
The public projection is not free: it needs idempotent reducers, versioning, privacy filters, and migration from current transcript-derived UI logic. It should ship after the canonical event identity and event sink are stable.
Confidence
81% - strong direct evidence and high UX resilience payoff.
Complexity
High - spans API contracts, viewer state, projection reducers, and delivery semantics.

Cuts

Rejection Summary

# Idea Reason Rejected or Nested
1 API flight recorder / canonical event sink Sound, but nested inside I1. It must be fail-closed durable append, not just the current listener hook.
2 Event-store golden path parity harness Sound, but better as the acceptance test for I1 than a standalone architecture direction.
3 Forensic checkpoints before resume Sound, nested into I4 as the migration sequence so the repo does not overclaim crash-safe resume.
4 Phase accumulator resume Weak for first ranking. Good future design spike for mid-phase resumability, but current code lacks a durable accumulator/inbox abstraction.
5 Suspended is first-class Sound, but a lifecycle/status requirement nested under I2/I4 rather than a top-level idea.
6 Raw evidence manifests instead of raw DB logs Sound, nested under I6 as the Postgres side of the private evidence ledger.
7 Agent continuity record Weak unless narrowed to existing typed strategy/reflection fields; defer broader commitments/social-debt serialization.
8 Cursor wake-up WebSocket Sound, nested under I7. It is a delivery contract consequence, not the whole idea.
9 Durable outbox after event commit Sound, nested under I7 and I2. It belongs after event append/projection commit rather than before them.
10 Runtime readiness refs Useful migration telemetry, but below the meeting-test as a standalone idea; include as readiness gating in downstream planning.
11 Temporal now Rejected for timing. Temporal is useful prior art, but the repo should prove the Postgres event/checkpoint kernel first.
12 S3 as primary event store Rejected. Object storage is a good raw-evidence/archive layer, not the ordered UX/domain event store.
13 Parallel XState updates Rejected. Parallel LLM calls are fine, but accepted XState/game-state mutations still need one ordered writer.

No topic axis ended with zero survivors. The ranked set deliberately spreads across identity, storage kernel, checkpoint/resume, private evidence, single-writer ownership, and public viewer catch-up.
