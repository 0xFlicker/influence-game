Ideation: games Scope MCP OAuth Hardening

Top recommendation: make /mcp the user-facing MCP resource, starting with scope=games for canonical game/player/agent access only. Move the existing privileged developer/global surface, including private trace tools, to /mcp/producer with today's scope=mcp meaning.

Date: 2026-06-19 Topic: games-scope-mcp-oauth-hardening Mode: repo-grounded Focus: resource-based user MCP authorization for Influence games

Grounding Context

The current production OAuth path is deliberately singleton-scoped. packages/api/src/services/mcp-oauth.ts defines MCP_OAUTH_SCOPE = "mcp", validates authorization scope with "scope must be exactly mcp", persists that scope into authorization codes and access tokens, and client registration also accepts only mcp. The schema reinforces this with check constraints on clients, authorization codes, and access tokens.

The deployed MCP route validates the bearer token before dispatch, but packages/api/src/routes/mcp.ts then calls server.handle(validation.request) without passing the validated auth context. packages/api/src/game-mcp/server.ts therefore describes and serves a global developer inspection server, and packages/api/src/game-mcp/read-model.ts defaults event reads to producer visibility.

User/resource seams exist but are not currently attached to MCP auth: games.createdById, game_players.userId, agent_profiles.userId, and /api/player/games already define joined games as the authenticated player's history. Private trace manifests have actor/source-pointer metadata, but game_evidence_manifests.access_scope currently permits only producer_admin, and PrivateTraceReadModel.readContent falls back to a fixed producer accessor.

External protocol grounding: MCP authorization expects protected-resource metadata, scope guidance, resource indicators in authorization and token requests, bearer tokens on every request, and resource/audience validation before data is returned. Codex also prefers server-advertised scopes during OAuth login, so scope discovery shape matters for this design.

Recommended Scope Contract

/mcp is the user-facing MCP resource. Its first scope is games, described as "access your games via MCP." A valid games token authorizes a connected MCP client to read Influence game resources attributable to the authenticated subject, not the global Production Game MCP corpus.

  • Game-level access: games the subject created through games.createdById or joined through game_players.userId.
  • Player/agent access: player rows owned by the subject, plus player rows using an agentProfileId whose profile belongs to the subject through agent_profiles.userId.
  • No trace data: user-facing games access does not expose private trace content, trace manifests, reasoning search, or trace metadata.
  • Not granted: producer/global corpus access, private traces, developer evidence tools, mutation tools, admin/user-management tools, or normal app API authentication.

/mcp/producer is the privileged maintainer/developer resource. It keeps today's mcp role + scope=mcp global access contract and carries producer evidence/private trace tooling.

Consent copy should be concrete: "Allow this MCP client to read games you created or joined and your player/agent records in those games. This does not grant maintainer access, developer evidence access, or private trace access."

1. Trust-Boundary Route Split

Axis: scope contractConfidence: 88%Complexity: Medium

Description: Make /mcp the canonical user-facing MCP protected resource, initially challenging for scope=games. Move the current developer/global MCP to /mcp/producer, where protected-resource metadata and WWW-Authenticate challenge for scope=mcp. Do this as the new truth, without a compatibility alias or deprecation window for producer access at /mcp.

Basis: direct: existing metadata advertises only ["mcp"]; direct: current docs state scope=mcp grants global access; external: MCP clients use challenge/metadata scope guidance and resource indicators for audience binding.

Rationale: This segments by trust boundary, not by every product resource. /mcp can grow from games to agents later without multiplying endpoints, while /mcp/producer keeps privileged tools and global corpus access isolated under a different resource audience.

Downsides: Requires moving the existing producer route/docs/tests to /mcp/producer. The OAuth service needs a scope/resource pairing model instead of a singleton constant.

2. Live Subject Resource-Claim Resolver

Axis: resource claimsConfidence: 86%Complexity: Medium

Description: Add a service such as resolveMcpGamesClaims(db, userId) that computes accessible game IDs, owned player IDs, and owned agent profile IDs from live DB state. The token stores subject, scope, resource, client, purpose, and expiry; claims are resolved per request or per JSON-RPC call.

Basis: direct: the schema already has games.createdById, game_players.userId, and agent_profiles.userId; direct: /api/player/games already treats joined player rows as user game history.

Rationale: Live claims reflect newly joined/created games without refresh tokens or token reissue. They also keep user-facing game access separate from producer-only private trace policy.

Downsides: Requires careful query/test coverage and likely per-request caching to avoid repeated joins inside multi-tool sessions.

3. Auth Context as a First-Class MCP Input

Axis: auth propagationConfidence: 84%Complexity: Medium

Description: Change the MCP route/server boundary from server.handle(request) to server.handle(request, authContext). Make the context discriminated: { kind: "developer_mcp", ... } versus { kind: "games_subject", userId, claims, ... }. The user-facing read model should require that context for every list/read/filter call.

Basis: direct: /mcp already validates tokens before dispatch and audits subject/client/resource, but the JSON-RPC server/read model does not receive that context; direct: current read methods default to producer visibility.

Rationale: Authorization cannot be reliable if it lives outside the method that chooses games and visibility. Passing context first makes every future tool addition answer "what authority is this running under?" at compile time.

Downsides: Touches tests and call signatures across the MCP server/read-model seam.

4. Profile-Specific Tools and Resources

Axis: tool partitioningConfidence: 78%Complexity: Medium

Description: Treat developer MCP and user games MCP as two authorization profiles. The developer profile keeps existing tools and descriptions, including explicit private trace tools. The games profile exposes user-scoped descriptors, for example list_games returning only accessible games and event filters that do not allow producer visibility. It does not list trace tools.

Basis: direct: current server instructions say scope=mcp grants global access and tool descriptors carry OAuth scope mcp; external: Codex reads server instructions and prefers server-advertised scopes.

Rationale: Tool discovery is part of authorization UX. A user-facing token should not even see developer evidence or private trace wording in this phase.

Downsides: Requires either separate server factories or scope-aware descriptor generation. Reusing the same tool names is possible, but descriptions must be unambiguous.

5. Trace Access Stays Producer-Only

Axis: trace boundaryConfidence: 88%Complexity: Low

Description: Keep private trace tools, trace manifests, reasoning search, and raw trace content exclusively on /mcp/producer behind scope=mcp. The user-facing /mcp + scope=games surface should not expose trace data or trace metadata in this phase.

Basis: direct: current private trace manifests are written with accessScope: "producer_admin"; direct: existing docs intentionally put raw private trace content behind the global producer MCP gate.

Rationale: This avoids a storage/model-policy migration while still making the user-facing MCP useful for canonical game access. Future user-visible thinking or strategy resources should be a separate derived artifact, not a direct read of the raw producer trace blob.

Downsides: User-facing MCP will not answer questions that require private reasoning evidence until a later derived-resource design exists.

6. Defer Evidence Accessor Upgrade

Axis: trace boundaryConfidence: 84%Complexity: Low

Description: Do not upgrade EvidenceAccessor for user subjects in this phase. Keep the current producer-oriented evidence path intact for /mcp/producer, and keep user-facing games tools on canonical game/player/agent read models.

Basis: direct: EvidenceAccessor currently grants private evidence through producer/admin roles; direct: private trace reads already fall back to a fixed producer accessor.

Rationale: If games does not expose private traces, the accessor migration is not on the critical path. Deferring it keeps this iteration focused on route/resource/scope split and user game authorization.

Downsides: Future user-facing thinking/strategy access will still need either a derived resource model or a real policy-subject accessor.

7. Atomic Cutover With Matrix Tests

Axis: tests and cutoverConfidence: 82%Complexity: Low-Medium

Description: Ship the route/scope/resource split as one atomic cutover: /mcp becomes the user-facing games resource and /mcp/producer becomes the privileged mcp resource. Inside that one change, implement the pieces in dependency order: scope/resource metadata and DB constraints, auth context propagation, user-scoped canonical game reads, and producer trace retention on /mcp/producer. The test matrix should prove wrong-scope/wrong-resource tokens fail before read models, mcp behavior stays producer-only, games only sees owned/created/joined resources, and games cannot list or read trace data.

Basis: direct: existing docs and tests already emphasize wrong-resource/wrong-scope failures before dispatch and Codex-first OAuth smoke checks.

Rationale: There is no product need for a multi-release migration. The important discipline is final-state matrix coverage, because authorization bugs have high blast radius and the old producer defaults must not leak into the new user endpoint.

Downsides: The migration and app code need to deploy together cleanly, and existing producer clients pointed at old /mcp will stop working by design.

How games and mcp Coexist

  • /mcp is the user-facing MCP resource. scope=games is the first user-facing scope and grants only the subject's allowed game/player/agent resources.
  • /mcp/producer is the privileged developer/global resource. scope=mcp keeps today's meaning: mcp role + valid token grants global access to wired producer MCP tools.
  • Do not make mcp imply games on the user endpoint in the first phase. Maintainers who want the user-scoped view can authorize games as themselves; maintainers who want global inspection use /mcp/producer.
  • Prefer exactly one scope per issued MCP access token in this phase. Dynamic client registration and authorization may accept supported scope sets such as games mcp for client compatibility, but the resource indicator must select one exact grant scope and unsupported or wrong scope/resource combinations must fail.
  • Both scopes can share the OAuth code/token storage pattern, but scope/resource validation must happen before JSON-RPC dispatch and must pass the resulting auth profile into the server.

First Code Boundaries

OAuth service/schema
Replace singleton scope typing with an explicit scope union, update DB constraints to allow mcp and games, and validate exact scope/resource pairings: /mcp + games, /mcp/producer + mcp.
Metadata/routes
Move producer metadata and route handling to /mcp/producer. Put user-facing protected-resource metadata, challenges, and JSON-RPC handling at /mcp.
Bearer validation
Return a discriminated auth context with scope, subject, client, resource, and profile. Validate exact resource/scope pairing.
MCP dispatch
Pass auth context into JSON-RPC handling and audit scope/profile/tool outcomes.
Read model
Add auth-aware game resolution, list filtering, event visibility enforcement, and player/agent ownership checks.
Evidence/trace
Keep trace tools producer-only and exclude trace metadata from the user-facing /mcp profile.
Web consent
Change the authorization page to render scope-specific grant copy and denial messages; games should not say "MCP Role Required."
Tests
Add final-state scope/resource matrix tests for the atomic cutover.

Intentionally Deferred

  • User-facing trace/thinking representation: raw full trace, reasoning-only, strategy packets, summaries, redacted derived views, and prompt removal policy.
  • Refresh tokens, long-lived sessions, DPoP, machine-to-machine grants, or a general third-party OAuth app platform.
  • Mutation tools or game lifecycle control through MCP.
  • Public ChatGPT app listing, iframe UI, or broad external tester policy.
  • Reinterpreting existing scope=mcp tokens as user-scoped or retrofitting per-user filters into the producer endpoint.
  • Compatibility aliases or deprecation ceremony for privileged producer access at old /mcp.
