Strategic Upgrade Slice

A focused idea set for turning private strategic intent from a receipt trail into a useful in-game notebook without making every turn feel like paperwork.

Grounding Context

The current strategy system already has live private agent memory, scheduled strategic reflection, prompt rendering, and producer/debug artifacts. It does not yet have a first-class moment-of-action notebook. The gap is most visible when an agent deliberately breaks from a prior strategy and needs to carry forward "I did this because X; next I should Y."

Current live memory includes allies, threats, notes, round history, power actions, `lastReflection`, and `strategyPacket`, but no agenda/task list. `getContinuityCapsule()` already exposes `commitments: []`, which is a useful empty hook for future private strategic loops.

Prompt construction keeps `## Your Recent Decisions`, `## Past Vote History`, `## Strategy Thread`, and `## Strategic Assessment` as distinct sections under `## Your Memory`. That separation preserves the public/private boundary, but current decision entries lack an obvious "when this decision was made and why it changed my plan" shape.

The action tool surface currently requires `strategyPacketUse` and `strategyPacketUseRationale` across many strict schemas. Those fields classify whether a packet was followed, revised, ignored, or deferred, but they do not update strategy state by themselves. The model call helper is also one-tool oriented: it sends a single tool, sets `parallel_tool_calls: false` in named mode, and consumes only the first `tool_calls[0]` result.

External tool-calling prior art supports multiple function calls in one model turn and recommends code handle arrays of tool calls, but also notes that `parallel_tool_calls: false` forces zero or one tool and that strict mode has schema requirements. That makes multi-tool strategy updates feasible, but not the smallest compatibility-safe first slice.

1Anchored Strategy Receipts

Add one nullable structured object to existing action outputs: an anchored private receipt that records the action's strategic meaning. It should be short and typed: `anchor`, `decision`, `reason`, `nextIntention`, `changedPlan`, and `reflectionRequest`. This turns `strategyPacketUseRationale` from a loose label into a note that can drive later play.

Confidence 88% Complexity Medium Top pick
Axis
Moment-of-action capture
Basis
direct: Existing strict schemas already require `strategyPacketUse` fields on Mingle intent, Mingle turn, votes, candidate selection, power, and council actions; `StrategyPacketUseMarker` only stores revision, use classification, and rationale.
Rationale
The receipt is the smallest move that gives a future reflection concrete material: not just "revised", but "after Vera dodged, I exposed her because Jace became useful cover; next repair with Jace."
Downsides
Every touched action schema grows. The field must be bounded and nullable to avoid turning every utterance into strategy paperwork.

2Private Strategic Loops

Introduce a bounded private agenda list for unresolved strategic loops. These are not chores; they are soft intentions such as "test whether Jace is protecting Nyx", "repair with Mira after exposing her ally", or "if empowered, protect Vera to collect debt." Status values should support `open`, `in_progress`, `done`, `abandoned`, and `superseded`.

Confidence 82% Complexity Medium
Axis
Mutable strategic agenda
Basis
direct: Agent memory currently has notes and strategy packets but no agenda object, while `PlayerContinuityCapsule` already contains `commitments: string[]` as private continuity state.
Rationale
Open loops give social strategy a durable shape: agents remember what they meant to probe, repair, exploit, or avoid. This is more gameplay-driving than a single current target posture.
Downsides
Too many tasks will make agents mechanical. Limit visible prompt entries and reconcile/prune through reflection.

3Reflection Request Gate

Let each action optionally request a heavier strategy reconciliation: `reflectionRequest: "none" | "soon" | "now"`. `now` runs a targeted reflection immediately after the action if the phase can tolerate it; `soon` batches at the next phase boundary. Automatic triggers can promote a request when `changedPlan` is true or when the action contradicts the current Strategy Thread.

Confidence 80% Complexity Medium
Axis
Reflection cadence
Basis
direct: Strategic reflections are already separate memory updates, run through `DiaryRoom.runStrategicReflections`, and currently fire at fixed vote boundaries.
Rationale
This preserves the old cost concern while giving agents a path to say "this was a strategic rupture; update me before I forget why."
Downsides
Immediate reflections can increase latency in hot phases. The runner needs clear guardrails for when `now` is downgraded to `soon`.

4Receipt-to-Reflection Weave

Build strategic reflection prompts from recent public/private context plus structured receipts, not raw hidden thinking. The reflection sees what the agent said or did, what private anchor it attached, and which loops were opened or closed. It updates the Strategy Thread and agenda from that safer, smaller input.

Confidence 76% Complexity Medium
Axis
Reflection cadence
Basis
direct: Docs require hidden thinking to remain producer/debug material and keep Strategy Thread summaries as the compact safe state reinserted into prompts.
Rationale
The "dumber" thinking-weave idea has the right spirit, but a typed receipt is safer than feeding freeform hidden reasoning back into strategy prompts.
Downsides
Requires a new context builder slice so reflection can see the right receipts without bloating every prompt.

5Strategy Prompt Ledger

Keep `## Your Recent Decisions` separate from transcript summaries, but upgrade it into an anchored ledger. Each entry should carry a time/event anchor, private decision, reason, and next intention. A separate `## Your Open Strategic Loops` section can then show only active agenda items.

Confidence 74% Complexity Low
Axis
Carry-forward prompt shape
Basis
direct: `buildRecentDecisionsSection` already renders a separate section and warns that recorded decisions are not instructions to repeat them.
Rationale
This answers the timing concern without leaking private motive into conversation summaries. The anchor makes causality obvious: when the decision happened, what triggered it, and what the agent intended next.
Downsides
If rendered too long, it competes with Strategy Thread and Strategic Assessment. Cap entries by relevance and recency.

6Multi-Tool Notebook Pilot

After the inline receipt path proves value, pilot an explicit side tool such as `update_strategy_notebook`. The model would receive the phase action tool and the notebook tool together, and the runner would process both. Keep this as a second-stage architecture option, not the first slice.

Confidence 68% Complexity High
Axis
Tool-call architecture
Basis
external: OpenAI function-calling docs say responses can include zero, one, or multiple calls and describe iterating over `tool_calls`; the same docs say `parallel_tool_calls: false` prevents multiple calls. direct: this repo currently sends one tool and reads only `tool_calls[0]`.
Rationale
Separate action and notebook tools are conceptually clean and would let the strategy update feel like a real agent affordance. The cost is call-loop complexity and provider compatibility.
Downsides
Local OpenAI-compatible providers may be less reliable with multi-tool strict schemas. Partial success semantics also get harder: action succeeds, notebook fails, or vice versa.

7Continuity-Ready Agenda Schema

Design receipts and strategic loops so they can later flow into continuity capsules and `MemoryStore`, but do not persist them as the first implementation goal. The first slice should be honest live-run memory plus simulation artifacts; the schema should simply avoid painting durability work into a corner.

Confidence 66% Complexity Medium
Axis
Mutable strategic agenda
Basis
direct: `MemoryStore` currently persists only ally, threat, note, vote history, and reflection records. Docs state Strategy Thread packets are live-run private state and not written to `MemoryStore`, while continuity capsules are the future private snapshot lane.
Rationale
Task-like strategy loops become much more valuable when games can survive restart, but implementing persistence first would repeat the old overclaiming risk.
Downsides
Some schema choices may change when runtime snapshot hydration matures. Keep v1 type names explicit and migration-friendly.

Rejection Summary

#IdeaReason Rejected
1Receipt-only minimalismToo close to the current `strategyPacketUseRationale` weakness; it classifies a pivot but does not create durable enough structure for play.
2Always-on reflection after every actionDirectly fights the old cost/noise finding and risks making every turn over-optimized. Better as a trigger-gated path.
3House-mediated strategy coachingBreaks player agency and private knowledge boundaries. House can analyze player behavior, but should not rewrite a player's private strategy.
4MemoryStore-first strategy persistenceValuable later, but too entangled with broader checkpoint/hydration work for the next slice.
5Inline private notes in conversation summarySolves timing at the cost of blurring transcript facts, public social surface, and private motive. The anchored ledger gives the same benefit more safely.
6Replace Strategy Thread entirelyPremature. The packet already works as compact posture; the missing piece is moment-of-action receipts and open loops feeding it.
