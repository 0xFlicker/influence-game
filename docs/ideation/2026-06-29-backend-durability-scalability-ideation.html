Ideation: Backend Durability and Scalability

Updated around a higher impact bar: durable resume work should make one real API-backed game survive a supported process restart boundary and reach completed results. The June 30 continuation narrows the next expansion to endgame phase-boundary coverage, while keeping accumulator-heavy states fail-closed until the missing runtime input is real.

Grounding Context

Influence already has the inspection substrate: owner epochs, canonical API events, event-boundary checkpoints, Runtime Snapshot v1, hydration passports, evidence manifests, and durable watch summaries. At the start of this pass, that substrate had not yet changed the game outcome after process loss: a real checkpoint could earn hydration_candidate, while resumeAvailable remained false and active execution was not crash-safe.

The live failure mode is process-local execution. GameRunner still owns the phase actor, game state, agents, diary room, token tracker, House context, Mingle inbox, elimination order, and accusations in memory. game-lifecycle.ts still tracks active games in a process-local activeGames map, but supported suspended runs can now be claimed by startup recovery instead of remaining only inspectable.

The useful seam is narrower than full crash-safe sessions but more valuable than another readiness label: the runner already writes phase-boundary checkpoints after durable event flush, and an existing DB-backed test already aborts after such a checkpoint. The next slice should make that class of interruption continue, not merely explain why it cannot.

Impact bar

A candidate survives only if its output can be judged by game behavior: same game ID, same durable event stream, fresh process/owner, post-restart events, watch continuity, and completed results. Proof artifacts can exist only as support for that outcome.

Idea 1. One-Boundary Resume To Completion

Axis
Resume outcome
Confidence
86%
Complexity
High

Description

Ship one executable restart boundary: start an owner-backed API game, let it reach one whitelisted phase-boundary checkpoint, interrupt the process, recover the same game under a fresh owner, append new canonical events, and reach normal completed results. The deliverable is not a new checkpoint verdict; it is one class of interrupted game no longer dying.

Basis

direct: AGENTS.md says a server reset can corrupt active games; docs/statefulness-plan.md says no resume exists; GameRunner already writes phase-boundary checkpoints after durable event flush; the passport test already aborts after a real candidate checkpoint and asserts resumeAvailable is false.

Rationale

This is the first slice that changes the game outcome. It turns weeks of readiness scaffolding into an observable durability improvement: one interruption class no longer kills the run. It also forces every supporting piece to be judged by whether the game continues.

Downsides

It is high blast-radius for a first resume slice. The scope must stay deliberately narrow: one boundary, one recovery path, one real kill/restart acceptance test, and no claims about mid-phase model calls or arbitrary phase recovery.

Idea 2. Supported Boundary Whitelist

Axis
Supported boundary scope
Confidence
84%
Complexity
Medium

Description

Define the v1 resume surface as a small whitelist, not a general promise. A checkpoint is executable only when its kind, phase actor coordinate, event head, owner epoch, accumulator state, and no-pending-effects evidence match a supported boundary class. Everything else remains non-resumable.

Basis

direct: Existing checkpoint storage already binds game ID, owner epoch, checkpoint kind, event sequence, event head hash, and projection hash. The runner already records actor witness and accumulator registry evidence at phase-boundary checkpoints.

Rationale

The whitelist is what keeps the impact slice small enough to ship. It lets the team say "this exact boundary resumes" instead of reopening the proof rosebush around every phase and every in-flight model effect.

Downsides

The first version will cover only a fraction of interruptions. That is acceptable only if the covered class genuinely resumes to completion and the unsupported classes fail closed without new terminology churn.

Idea 3. Minimal Runner Hydrator

Axis
Runner reconstruction
Confidence
78%
Complexity
High

Description

Add the smallest executable hydration path for the supported boundary. It should reconstruct enough GameRunner state to continue: accepted game state/projection, phase actor coordinate, agent/House continuity where required, token cursor, and written checkpoint/event cursors. It should not attempt generic mid-phase recovery.

Basis

direct: GameRunner owns many long-lived private fields, while the checkpoint payload already persists runtime snapshot, transcript cursor, token cursor, continuity capsules, event boundary, and projection evidence. The missing piece is not another field label; it is code that loads a valid boundary and runs forward.

Rationale

This is the core technical move behind the outcome. Without an executable hydrator, all recovery commands and readiness gates still terminate in "cannot resume." The implementation should be boring and bounded, not elegant.

Downsides

This may expose that one "quiet" boundary still needs more runner state than expected. If so, the slice should reduce the supported boundary or persist one missing runtime input only when it is required for the game to continue.

Idea 4. Startup Recovery Coordinator

Axis
Ownership and recovery operation
Confidence
80%
Complexity
Medium

Description

Create a configured API-startup recovery coordinator that finds eligible suspended or startup-orphaned games, validates the latest supported checkpoint, claims a fresh owner epoch, and starts the hydrated runner. Keep operator/admin controls as a later backstop, not the primary v1 recovery path.

Basis

direct: startup cleanup currently marks old active owners as startup_orphaned; ownership code already issues owner epochs and active checks; event append code already locks the owner row and rejects stale or non-contiguous commits.

Rationale

This fits the current deployment reality: the API process is the only worker today, and it can safely recover eligible games on startup when configured. The behavior improves the game without asking an admin to babysit interrupted sessions.

Downsides

Startup recovery needs strict support predicates and bounded failure semantics. If the checkpoint is unsupported or hydration fails, the game should remain suspended and inspectable.

Idea 5. Kill/Restart Smoke Harness

Axis
Verification by interruption
Confidence
88%
Complexity
Medium

Description

Replace "checkpoint proves candidate" as the headline test with a restart smoke: create an API-backed game, interrupt after the supported checkpoint, start a fresh process or recovery harness, recover the same game, and assert post-restart canonical events plus completed results.

Basis

direct: the existing checkpoint passport test already drives a real GameRunner to a candidate phase-boundary checkpoint, aborts it, and verifies that resume remains unavailable. That is almost the failing test for the new behavior.

Rationale

This is how the team avoids sliding back into proof-only work. A slice is not done because a validator passes. It is done when the kill/restart scenario that previously stranded the game now completes it.

Downsides

The harness can become expensive or flaky if it depends on live model calls. The first test should use deterministic agents and the real DB path; later smoke coverage can graduate to fuller model-backed games.

Idea 6. Resume Watch And Results Continuity

Axis
Resume outcome
Confidence
76%
Complexity
Medium

Description

Make the acceptance test include the surfaces the game actually uses: after recovery, the watch state advances past the pre-restart prefix, completed results exist for the same game, and no second game/run identity appears to players. This is not a separate UI feature; it is the product-facing proof that resume worked.

Basis

direct: lifecycle completion already persists completed results and refreshes watch summaries; watch summaries are durable projections rather than only process-local WebSocket state. Resume should use those surfaces as outcome checks.

Rationale

A backend-only resume that appends events but leaves the viewer or results confused is not game durability. This idea keeps the implementation accountable to what the user and friends will actually experience.

Downsides

It adds acceptance scope. Keep it to verification of existing watch/results behavior, not redesigning the viewer or inventing new recovery labels.

Idea 7. Deploy Drain Plus Resume

Axis
Deployment usefulness
Confidence
72%
Complexity
Medium

Description

Once phase-boundary startup recovery is trustworthy, use the same mechanism for planned restarts: stop an active game at the next supported boundary, exit the process, then recover after deploy. This makes deployment safer without claiming arbitrary crash or spot recovery.

Basis

direct: GameRunner.abort() already signals a stop at the next phase boundary, and activeGames is process-local. Planned deploys are the most controllable restart class, so they are the first operational place the resume path should pay off.

Rationale

This gives immediate operational value after the core slice lands: active games no longer force a choice between waiting indefinitely and risking corruption during deploys.

Downsides

It depends on the first manual recovery path being trustworthy. If built too early, it becomes graceful shutdown theater rather than restart durability.

Implementation Recut

The one-boundary slice proved the startup recovery seam but left the product impact too narrow: a game only resumes if interruption happens after Introduction and before Round 1 starts. The next useful implementation slice should remove that artificial wall and support persisted normal-round phase boundaries wherever the checkpoint evidence says the runner is between phase effects.

The strongest implementation direction is not a new proof label. It is a generic phase-boundary resume contract for persisted actor coordinates, a selector that accepts the latest event-head checkpoint only when it is implemented-safe, API startup recovery acting as today's worker, and a DB-backed interruption matrix proving the same game completes after several real phase stops.

Idea 8. Normal-Round Boundary Resume Matrix

Axis
Supported boundary scope
Confidence
82%
Complexity
High

Description

Generalize the supported recovery target from one lobby checkpoint to persisted normal-round phase boundaries: pre-round lobby, then vote, mingle, power, and reveal. Recovery should accept the latest checkpoint only when it is at the event head and implemented-safe, hydrate to that actor coordinate, append post-restart events, and complete the same game.

Basis

direct: before this pass, game-recovery.ts and game-durable-run.ts rejected any checkpoint whose actor coordinate was not lobby and any event log containing round.started. GameRunner writes Runtime Snapshot v1 and transcript replay for every phase-boundary checkpoint.

Rationale

This is the first slice that matches the likely failure distribution. A process interruption usually lands somewhere inside a round, so normal-round boundary resume changes real deploy and restart risk instead of only protecting the first few seconds of a game.

Downsides

Normal phases carry more phase-machine context than the intro boundary. The slice should fail closed for endgame and accumulator-heavy boundaries until those states have specific hydration support.

Idea 9. Actor Coordinate Hydration Contract

Axis
Runner reconstruction
Confidence
78%
Complexity
High

Description

Replace post_intro_pre_lobby with a versioned phase_boundary resume input that carries the target actor coordinate, canonical event prefix, transcript replay, token cursor, and House continuity. The runner should hydrate the phase actor deterministically to that coordinate before entering the existing phase loop.

Basis

direct: GameState.fromCanonicalEvents(), TranscriptLogger.seed(), TokenTracker.loadCursor(), and House continuity already exist. The missing piece is phase actor restoration beyond the one hard-coded two-transition path in runGameLoop().

Rationale

The product value is unlocked by an engine contract, not by API-side optimism. Once the runner can resume from a coordinate, the recovery selector can widen without lying.

Downsides

Hydration must avoid replaying phase side effects. The phase actor can be advanced with machine events derived from canonical truth, but the game phase runners must not re-run work that already emitted canonical events.

Idea 10. Startup Reconciler As Today's Worker

Axis
Ownership and recovery operation
Confidence
84%
Complexity
Medium

Description

Keep recovery automatic on API startup when configured. The single API process can scan suspended/startup-orphaned games, claim a fresh owner for the latest implemented-safe event-head checkpoint, and run the recovered game itself. Future worker fleets can reuse this service after the single-container path is trustworthy.

Basis

direct: current deployments have an API process and a web process, and the previous slice already added a startup recovery gate. Owner epochs and append-time owner checks already fence stale writers in the durable path.

Rationale

An admin-gated resume flow does not fit the product goal. If the API can recover safely under today's deployment constraints, it should do so without making an operator babysit interrupted games.

Downsides

Startup recovery should cap attempts and leave failed games inspectable. V1 does not need multi-worker leases or spot-fleet coordination, but it must not spin forever on a bad checkpoint.

Endgame Resume Expansion

The generic normal-round recut has now landed far enough that the next product question is no longer "can startup recovery work at all?" It can. The sharper question is which later phase-boundary coordinates can be added without turning resumeAvailable back into wishful thinking with a nicer jacket.

The current support set is lobby, vote, mingle, power, reveal, and reckoning_lobby. The low-risk endgame states mostly need explicit phase-actor hydration, prerequisite checks, and DB-backed same-game completion tests. tribunal_defense is different: it needs the runner's _currentAccusations map, which is still an in-memory accumulator rather than canonical state.

Continuation filter

Enable only coordinates that can resume from the latest event-head checkpoint and complete the same game under startup recovery. Anything that needs _currentAccusations must first persist or reconstruct that accumulator through a typed runtime contract.

After the staged endgame coordinate expansion, Accusation Capsule V1 is the next necessary TODO: tribunal_defense stays blocked until _currentAccusations has structured durable support.

Idea 11. Reckoning Pair Resume Tranche

Axis
Cheap endgame coordinates
Confidence
86%
Complexity
Medium

Description

Add reckoning_plea and reckoning_vote as the next supported phase-boundary coordinates after reckoning_lobby. The slice should add explicit phase actor hydration steps, endgame-stage and alive-count prerequisites, and DB-backed interruption/recovery tests for both coordinates.

Basis

direct: docs/statefulness-plan.md marks reckoning_plea / reckoning_vote as likely small because canonical GameState already carries alive players and endgame state after reckoning_lobby. packages/engine/src/game-runner.ts already dispatches both phase runners in the main loop.

Rationale

This is the cleanest next real expansion. It adds meaningful late-game restart coverage without introducing a new accumulator persistence problem.

Downsides

The value is bounded to the four-player endgame transition. It still leaves Tribunal, Judgment, mid-phase interruption, and historical repair unsupported.

Idea 12. Tribunal Entry Resume Tranche

Axis
Cheap endgame coordinates
Confidence
80%
Complexity
Medium

Description

Add tribunal_lobby and tribunal_accusation as a second low-risk tranche, gated on three alive players, the correct endgame stage, and an empty currentAccusations accumulator at the boundary. Keep tribunal_defense out of the tranche.

Basis

direct: docs/statefulness-plan.md says tribunal_lobby / tribunal_accusation should be similar to reckoning_lobby if accumulators are empty. The runner already has separate branches for tribunal_lobby and tribunal_accusation.

Rationale

This expands into the three-player endgame while preserving the important boundary: entering the accusation phase is cheap, resuming defense after accusations is not.

Downsides

It depends on the checkpoint accumulator registry proving currentAccusations is empty. If a boundary lands after accusation collection, this tranche must reject rather than guess.

Idea 13. Judgment Finale Resume Tranche

Axis
Cheap endgame coordinates
Confidence
74%
Complexity
Medium

Description

Add judgment_opening, judgment_jury_questions, judgment_closing, and judgment_jury_vote as a finale tranche after the earlier endgame slices. Each coordinate should assert finalist/jury prerequisites and complete through the normal jury-result path.

Basis

direct: docs/statefulness-plan.md classifies Judgment coordinates as likely small-to-medium because GameState and transcript replay carry most context. The runner already contains distinct Judgment phase branches.

Rationale

Judgment is where a crash hurts viewer trust the most: the story is basically complete, but the ending can still vanish. This tranche protects the finale without solving every earlier Tribunal accumulator state first.

Downsides

It is broader than the Reckoning and Tribunal-entry tranches. Jury/finalist prerequisites must be explicit so the runner does not hydrate into a finale-shaped state with missing finale facts.

Idea 14. Accusation Capsule V1

Axis
Accusation accumulator durability
Confidence
78%
Complexity
High

Description

Persist _currentAccusations as a typed Runtime Snapshot accumulator capsule for the boundary between tribunal_accusation and tribunal_defense. Recovery should hydrate the map directly and enable tribunal_defense only when the capsule validates player IDs, accuser IDs, accusation text, and the checkpoint boundary.

Basis

direct: docs/statefulness-plan.md marks tribunal_defense blocked because it needs the runner's _currentAccusations map. runTribunalAccusation populates that map, and runTribunalDefense reads it to prompt accused players.

Rationale

This is the first real accumulator beyond Mingle inbox. Solving it as a versioned capsule creates a repeatable pattern for future accumulator-heavy boundaries without making transcript prose authoritative.

Downsides

It is more invasive than simply adding actor coordinates. The capsule needs privacy and compatibility discipline because it stores social content, even if that content is public accusation text.

Idea 15. Post-Defense Accusation Drain

Accusation accumulator durability

72%
Complexity
Low

Description

Audit whether tribunal_vote can safely ignore _currentAccusations, then clear or mark the accumulator drained after runTribunalDefense completes. If true, tribunal_vote can become a cheap supported boundary without persisting accusation state.

Basis

direct: docs/statefulness-plan.md says tribunal_vote does not need _currentAccusations but needs an accumulator audit. The runner calls runTribunalVote without passing the accusations map.

Idea 16. Endgame Hydrator Ladder

Axis
Hydrator and selector correctness
Confidence
77%
Complexity
Medium

Description

Refactor hydratePhaseActorForResume into a declarative ladder of actor-coordinate steps before adding several endgame states. Each step should name prerequisite canonical events, alive-player expectations, endgame-stage checks, and the phase-machine events needed to reach the coordinate without re-running phase effects.

Basis

direct: the current hydrator manually advances through normal-round states and special-cases reckoning_lobby. evaluateSupportedRecovery separately validates prerequisites from canonical events and rebuilt GameState.

Rationale

Endgame support will add enough coordinates that a readable ladder becomes cheaper than another pile of bespoke branches. The point is not architecture astronautics; it is making each supported coordinate inspectable in code.

Downsides

If the immediate slice adds only the Reckoning pair, this may be premature. It earns its keep when two or more endgame tranches are added close together.

Idea 17. Coordinate Recovery Matrix Generator

Axis
Verification by interruption
Confidence
84%
Complexity
Medium

Description

Generalize the DB-backed recovery matrix so every exported supported coordinate gets an interrupt, startup-recover, contiguous-event, completed-result test. Unsupported coordinates should be generated as negative cases with explicit skip reasons such as unsupported actor coordinate or unsafe accumulator registry.

Basis

direct: game-recovery.test.ts already matrix-tests supported normal coordinates and separately proves reckoning_lobby. The startup-recovery learning says every new coordinate needs this same DB-backed proof before it can advertise resumeAvailable.

Rationale

This keeps the exported coordinate union, hydrator, recovery predicate, durable inspection, and acceptance proof locked together. Without it, support will drift one innocent string literal at a time. Classic clown-car engineering, best not.

Downsides

The matrix can get slow because DB-backed recovery tests touch real persistence. Keep it focused and run the known DB suites sequentially when local Postgres lock contention appears.

Rejection Summary

# Idea Reason Rejected
1 Terminal Redrive First Slice Repairs a narrow terminal-materialization gap, but does not solve the more likely interruption point: mid-run process loss before the game is over.
2 More Hydration Passport Stamps Below the new bar. The repo already has candidate/readiness machinery; more proof without continuation is not game impact.
3 Resume Proof Ladder As A Standalone Slice Useful vocabulary has become wheel-spinning unless attached to a working restart-and-complete path.
4 Phase-Step Kernel First Potentially right long-term, but too abstract before one current checkpoint can resume a game.
5 Full Workflow Platform Migration Too expensive and likely to defer the real question: can this game continue from this repo's existing durable boundary?
6 Redis Pub/Sub Or Horizontal Scaling First Improves process topology and freshness, but a fresh worker still cannot continue execution.
7 Generic Serverless Function Port Changes hosting shape without durable execution semantics. It can make state loss more frequent rather than less.
8 Full Arbitrary Mid-Phase Resume Too broad for the next slice and likely to tangle with in-flight model calls, partial accumulators, and duplicate effects.
9 Retry Current Phase From Start Tempting shortcut, but unsafe for social facts and model effects unless every phase action is already idempotent.
10 Unbounded Startup Auto-Recover Automatic startup recovery is useful only behind strict support predicates and bounded failure behavior; a broad best-effort scan would hide unsafe checkpoints.
11 Mock-Only Resume Demo Useful during development but not sufficient. The accepted output must be an API-backed game path with durable DB state.
12 Manual DB Surgery Runbook Helpful support material, but it does not make the backend more durable or the game continue.
13 Endgame Boundary Coverage In The Same Slice Valuable later, but the current accumulator and accusation state shapes make it a riskier first expansion than normal-round phase boundaries.
14 Backfill Old Interrupted Games First May help support work, but it does not improve future process interruption behavior unless generic recovery lands first.
15 Serverless Workflow Rewrite Could be a later deployment target, but it replaces the hosting layer before the current runner can resume from its own durable checkpoints.
16 Structured Accusation Reconstruction First Interesting but weaker than a typed accumulator capsule unless structured agent-turn evidence becomes recovery-authoritative; public transcript prose stays out of game-state truth.
17 Restart Soak As A Separate Slice Useful validation, but it should be folded into the coordinate recovery matrix rather than becoming another proof-only deliverable.
18 One-Predicate Resume Truth As A New Idea Already the current architecture: durable inspection and startup recovery share the implemented support predicate. Preserve it as a constraint, not a fresh direction.
19 Historical Recovery Triage View Useful follow-up for support and prioritization, but it does not itself expand which active games resume after a restart.
20 Full Endgame Resume In One Slice Overclaims the surface. tribunal_defense has a real accumulator blocker, so the safe path is staged coordinate support followed immediately by Accusation Capsule V1 / full accumulator support.
