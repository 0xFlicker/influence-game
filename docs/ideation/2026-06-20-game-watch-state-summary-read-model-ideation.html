Codebase Context

Grounding Context

  • The list route pays detail cost per visible game. GET /api/games loads visible game rows, then maps every row through buildGameWatchState(db, game) before returning summary fields and summarizeWatchState. Evidence: packages/api/src/routes/games.ts:190.
  • Full watch-state build does the expensive work the list does not need. buildGameWatchState loads player identities and terminal results, reads persisted events, replays projection, builds player rows, then only later can the route drop players. Evidence: game-watch-state.ts:150.
  • The event reader validates the whole ordered log. getPersistedGameEvents selects rows from game_events ordered by sequence, validates metadata, recomputes hashes, and returns a trusted prefix. Evidence: game-event-read-model.ts:222.
  • The projection reader replays every trusted event in memory. getPersistedGameProjection starts from createEmptyProjection and calls applyCanonicalEvent for each trusted event. Evidence: game-projection-read-model.ts:174.
  • There is already a clean write-side seam. appendDurableEventsAndPublishWatchState wraps durable append and watch-state publish, and persistCompletedGame owns terminal result/status creation. Evidence: game-lifecycle.ts:181 and game-lifecycle.ts:426.
  • The existing tests define the correctness bar. Route tests expect list/detail to return durable round, phase, counts, winner, projection status, and cursor fields, while service tests verify degraded logs and private-field stripping. Evidence: games-api.test.ts:451 and game-watch-state.test.ts:24.
  • Prior local guidance supports rebuildable summaries, not private leakage. The durable architecture separates canonical game events as accepted board facts from private agent-turn evidence and replayable projections. Evidence: docs/solutions/architecture-patterns/agent-strategy-observability-spine.md:40.
Recommended interpretation: the first watch-state slice deliberately deferred materialization. This review finding is the trigger to add it, but only for compact list summaries. Detail reads and websocket catch-up should keep full replay/validation semantics.

Surface Map

Topic Axes

  • List API shape Preserve current public list payload fields and public-by-URL watch behavior while removing per-row full replay.
  • Summary storage and refresh lifecycle Decide what compact row exists, where it is updated, and which status/player/result transitions must refresh it.
  • Migration and stale-summary handling Make existing games correct after deploy and make missing/stale rows visible without moving replay back into the list route.
  • Future append/checkpoint fit Keep the design compatible with incremental durable append, checkpoint boundaries, and potential projection caching without forcing that larger rewrite now.

1

Materialized game_watch_state_summaries

Summary storageConf 88%Cx Medium
Description
Add a compact persisted table keyed by game_id that stores the viewer-safe GameWatchState summary without players. The list route batch reads these rows for all visible games and keeps the existing top-level fields by projecting from the summary row.
Basis
direct: The current list route calls buildGameWatchState for every visible row, while buildGameWatchState reads and replays the durable event log before returning fields that the list then summarizes.
Rationale
This fixes the reviewed complexity class directly: list reads become O(number of listed games) over games, kernel-health rows, and compact summary rows. Full correctness stays in the detail/watch path.
Downsides
Adds a table, migration, refresh service, backfill story, and stale-row discipline. The first version may still refresh from full replay on writes until incremental projection caching exists.
2

Summary refresh service as the boundary

LifecycleConf 84%Cx Medium
Description
Create a service such as game-watch-state-summary.ts with refreshGameWatchStateSummary, getGameWatchStateSummariesByGameIds, and shared mappers between persisted rows and the list response shape. Use it after durable append, terminal completion, cancellation/suspension, game creation, joins, and fill operations.
Basis
direct: The lifecycle already has a durable append plus publish seam, and completion already centralizes result/status writes. These are better update points than the HTTP list route.
Rationale
A single service keeps the table from becoming a second, divergent watch contract. It also gives tests a clean place to prove summaries exclude private evidence and match current watch-state scalar fields.
Downsides
Care is needed around transactions. A summary refresh failure should not corrupt canonical truth, but it should be observable and repaired rather than silently leaving stale list data.
3

Projection cache as a later foundation

Future fitConf 62%Cx High
Description
Instead of only storing watch summaries, add a broader projection cache or snapshot table keyed by game_id and event head, then derive GameWatchState summaries from that. This could eventually support incremental append and checkpoint-aligned inspection.
Basis
reasoned: The current cost comes from replaying canonical events into projection. Persisting a reusable projection head attacks the shared root for list, detail, MCP, and future checkpoint work.
Rationale
This is the best long-term infrastructure direction if replay cost becomes a system-wide bottleneck.
Downsides
Too broad for this review fix. It would require larger schema choices, stronger projection-versioning semantics, and extra privacy review if a full projection object is stored.
4

Lazy repair only as compatibility glue

MigrationConf 55%Cx Low
Description
Permit targeted summary repair when a row is missing, but do not make cache-on-read the normal design. The deployment path should backfill existing games before relying on the list route, and regular writes should keep summaries current.
Basis
reasoned: A missing-row fallback is useful for compatibility, but if the list route repairs every visible row on a cold request, it recreates the exact O(all events for all games) behavior under a new name.
Rationale
This keeps the system robust during migration without letting lazy caching become the architecture.
Downsides
The fallback policy needs a hard guardrail. Tests should prove the happy-path list route does not call event replay when summaries exist.

Recommendation

Recommended Approach

Add a first-class game_watch_state_summaries table keyed by game_id. Store only viewer-safe list/detail summary fields: schema version, game status, source, current round, current phase, max rounds, counts, event cursor, projection availability/status/counts/diagnostics, final state, winner fields, and timestamps. Do not store players, raw event envelopes, source pointers, event hashes, owner epochs, checkpoint payloads, private traces, thinking, or reasoningContext.

The normal list route should batch load summaries for returned game IDs and never call buildGameWatchState for each row. GET /api/games/:id and websocket catch-up can keep using full buildGameWatchState so detail correctness is not capped by a stale or compact summary. If a summary is missing, detail and catch-up can repair by full build; the list path should rely on backfilled/write-maintained rows and avoid broad request-time replay.

First implementation can refresh summaries by calling the existing full watch-state builder on write-side transitions. That moves the work off the list route and keeps the code small. The row schema should still carry event-head metadata so a later ce-work session can replace refresh internals with an incremental projection cache without changing the list API again.

Follow-Up

Recommended Implementation Plan

  1. Schema and migration. Add gameWatchStateSummaries to packages/api/src/db/schema.ts, add packages/api/drizzle/0017_game_watch_state_summaries.sql plus Drizzle meta, and add the table to the test truncation list. Use game_id as the primary key and foreign key to games.id.
  2. Summary service. Add a service that can convert GameWatchState to a persisted summary row and reconstruct the list route's watchState summary. Keep the mapper shared with route code so the public contract does not fork.
  3. Write-side refreshes. Call the summary refresh after game creation, join/fill roster changes, start/stop status changes, durable event append, suspension/cancellation, and terminal completion. Put the durable append call near appendDurableEventsAndPublishWatchState; put terminal refresh immediately after persistCompletedGame succeeds and before publishing final watch state.
  4. Backfill compatibility. Add a one-time backfill command or deploy step that replays existing games once and writes summaries. The list route should not become the backfill worker. Missing summaries after deploy should be logged as repair debt, not silently repaired by replaying every visible log.
  5. List route rewrite. Replace the Promise.all(rows.map(buildGameWatchState)) path in GET /api/games with one batch summary read keyed by returned game IDs. Preserve the current response shape, status filtering, hidden-game exclusion, game-number assignment, kernel-health merge, public access, and winner/count/cursor fields.
  6. Tests. Add route tests proving the list route returns correct round, phase, counts, winner, source, projection status, and cursor from summaries. Add a regression that would fail if list replayed game_events, for example by corrupting event rows after summary write and showing list still uses the summary while detail degrades through full replay, or by injecting a watch-state builder dependency that throws on list use.
  7. Privacy and stale-row tests. Extend watch-state tests so persisted summaries do not contain private/prod evidence fields, and add service tests for missing summary, stale schema version, terminal fallback, and degraded projection summaries.
  8. Docs. Update the June 20 watch-state plan or a short docs note to say materialization is now part of the list-read path, while detail/watch state remains replay-validated and public watching remains public-by-URL.
