What changed since July 10

Grounding Context

The product thesis is still an improvement loop.STRATEGY.md says owners create and tune persistent competitors, then watch them reason, fail, and improve.
Agent Seasons machinery is now real.Immutable revisions, current and game-effective revision pointers, change magnitude, rating recalibration, and per-revision performance are implemented in agent-revisions.ts, revision-policy.ts, and the season analysis UI.
The remaining revision gap is narrative.Owners can compare performance by revision, but they still cannot easily see what changed, why it changed, what evidence motivated it, or whether the next game validated the hypothesis.
The refactor queue's top is stale.R1 still asks for an API simulation harness even though api-simulate.ts and bun run simulate:api exist. R3 also predates dedicated postgame MCP reads and needs narrowing.
CI has no visible security baseline.ci.yml covers install, types, lint, tests, and container smoke, but not dependency review, security-focused SAST, secret scanning, or container scanning.
The product-specific security work is mostly ordinary tests.Privy/JWT, OAuth and MCP scopes, owner-only cognitive artifacts, worker credentials, and private traces create boundaries that generic scanners cannot exercise. Several are already covered in isolation; the remaining work is to inventory the high-consequence promises and close the gaps.
Terminal settlement is the sharp reliability seam.R7 records that a finished in-memory result can be lost if persistence fails; the rolled-back transaction prevents partial awards, but there is no durable result capsule to retry.
Werewolf is a separate game, not a mode flag.The current config, state machine, events, projections, recovery coordinates, results, watcher, and season semantics are Influence-specific. The venue can host another format; the engine cannot pretend it is the same ruleset.

External evidence reinforces three cautions: agent improvement works best as trajectory → critique → persistent lesson → next attempt; agent security requires explicit authorization and token-boundary tests; and hidden-role games are useful agent benchmarks precisely because belief, deception, and partial information are hard. None of those findings require turning the repo into a framework-shaped pumpkin.

Decisive sequence

Clean the map, add guardrails, then close the loop.

The best next product build remains the owner improvement loop. Before starting it, spend one bounded engineering pass pruning stale R1/R3 queue entries, adding the missing automated scans, and completing a small trust-boundary contract suite. Then build the Agent Experiment Loop. Make terminal-result escrow and fault injection the reliability gate before widening gameplay. Validate House Coach cheaply through MCP, and only then run a one-night Werewolf-inspired simulation spike.

This is a sequence, not five simultaneous projects. Complexity is expensive; spend it where evidence has already accumulated.

1The Agent Experiment Loop

Turn the original Debrief Loop into a complete owner experiment: show three to five evidence-backed pivotal moments, draft a small agent change and plain-language patch note, require owner approval, mark the new revision as unmeasured, and compare its next games against the prior revision. Film Room is the evidence layer; shipped Agent Seasons machinery is the measurement layer.

Confidence 96%Complexity MediumAxis: Owner improvement

The hypothesis must remain a hypothesis. Social games are noisy; the product should show evidence without cosplaying as a laboratory.

Basis
direct + external: The product strategy names improvement as the loop; the repo now has revision lineage, per-revision performance, replay, public facts, and owner-scoped cognitive artifacts. Reflection research supports persisting evidence-grounded lessons into later attempts.
Rationale
This makes the expensive systems already shipped cohere into one reason to return: understand the agent, change it deliberately, and learn whether the change helped.
Downsides
Generated coaching can invent causality. Every lesson needs provenance, explicit uncertainty, small owner-controlled edits, and a clean separation between canonical fact, private cognition, and House-authored interpretation.

2Trust-Boundary Contract Tests

Add the boring automated baseline—dependency review, security-focused SAST, secret scanning, and container scanning—then inventory roughly 12–20 high-consequence authorization promises and exercise them as table-driven Bun tests at the existing policy, Hono route, MCP, and response-serialization seams. The matrix is test data, not a new policy engine.

direct + external: CI lacks visible automated security checks, while existing tests already cover much of the product contract: owner/participant/producer artifact access, MCP token boundaries, worker-token rejection, and read-versus-mutate admin permissions.
Rationale

Downsides
A large or supposedly exhaustive matrix would manufacture confidence and duplicate policy logic. Keep the cases curated, drive the real handlers and database-backed policies, and add a row when a new sensitive boundary appears. Treat the minimal R4 trace-retention/purge slice as a separate remediation.

3Terminal Result Escrow

Persist the exact terminal game result before settlement, represent settlement as pending or complete, and make retries idempotent by game ID. Add a narrow producer recovery action that completes results, points, ratings, receipts, and profile counters without replaying gameplay.

Confidence 94%Complexity HighAxis: Reliability
Basis
direct: R7 identifies the concrete failure: gameplay can finish in memory and settlement can roll back safely, yet the exact result is not retained as retryable work and terminal checkpoints are not startup-recoverable.
Rationale
This closes the most consequential statefulness gap without pretending the entire runtime is crash-safe. It preserves the one thing that cannot be reconstructed honestly: the specific outcome the agents produced.
Downsides
Requires a careful persistence boundary, explicit state transitions, idempotency across every award path, and migration/recovery diagnostics. Half-escrow is worse than no escrow because it manufactures confidence.

4MCP-First House Coach

Validate owner Q&A through the existing Games MCP before building a large chat surface: “Why did Nyx trust Echo?”, “Where did she abandon her plan?”, and “What small change should I test next?” Answers must cite public facts and authorized owner artifacts, label House inference, and respect current scopes.

Confidence 88%Complexity LowAxis: Owner improvement
Basis
direct: The repo already exposes bounded games, postgame, and cognitive read models through MCP. The immediate audience already uses connected AI tools, so the workflow can be tested without committing to a new in-product conversation stack.
Rationale
This cheaply discovers the questions owners actually ask and the evidence required to answer them. Those lessons can shape the fixed debrief before the interface becomes permanent.
Downsides
A coach can become a charming liar or a scope-leak oracle. Keep retrieval bounded, require citations, test authorization failures, and do not let conversational confidence outrun evidence.

5Kill the House at Every Door

Build a focused fault-injection league around phase transitions, checkpoint writes, terminal-result capture, and settlement. Kill or fail the process at named boundaries, restart it, and assert either an exact safe recovery or a legible operator failure state.

Confidence 86%Complexity MediumAxis: Reliability
Basis
direct + reasoned: Statefulness is the known operational risk, recovery coordinates already exist, and R7 supplies a high-value terminal seam. Deterministic failure injection converts vague crash anxiety into falsifiable invariants.
Rationale
It gives Terminal Result Escrow teeth and makes future runtime changes earn the phrase “recoverable.” The suite should be small, named after real failure doors, and run where its database/runtime cost is acceptable.
Downsides
Fault suites become slow and flaky when they test every possible line. Start with a boundary matrix and the failures that would corrupt trust, not chaos-engineering cosplay.

6One Night at The House

Run an invitation-only, simulation-only hidden-role game inspired by one-night Werewolf: one short cycle, asymmetric roles, private observations, public claims, and a final vote. Instrument belief changes, deception, rule adherence, and viewer legibility. Build no public selector and promise no reusable ruleset framework.

Confidence 82%Complexity MediumAxis: Multi-game exploration
Basis
direct + external: The House can plausibly host multiple formats, but current engine contracts are Influence-specific. Werewolf Arena, AvalonBench, and CSP4SDG show that hidden-role games expose agent reasoning, social deduction, and deception failures especially well.
Rationale
A one-night spike answers the product question—“is this watchable and strategically interesting?”—before architecture answers a question nobody earned the right to ask.
Downsides
Role balance, private knowledge, belief tracking, and deception are hard. If the simulation is fun, promotion still requires an explicit ruleset boundary across config, actions, events, projections, recovery, results, and UI.

7The Patch Reel

Let an owner curate a short, provenance-safe before/after story for one agent revision: the old tendency, the evidence that motivated a change, the owner-approved patch note, and a few later moments that illustrate what happened next. Share the canonical game/replay surface; describe outcomes as observations, never causal proof.

Confidence 78%Complexity MediumAxis: Sharing and audience
Basis
direct + reasoned: House Highlights, replay, canonical game links, revision lineage, and season comparison now exist. The original Premiere Night idea becomes more product-specific when the social artifact is an agent's evolution rather than another watch-room mechanic.
Rationale
“Here is how my agent changed” is a better Influence story than “a video finished rendering.” It connects owner craft, persistent identity, public drama, and return play.
Downsides
Before/after edits invite cherry-picking and false causality. Keep the owner in control, preserve source links, use observational language, and reuse the existing replay/trailer share dialog instead of spawning another share surface.
