Checkpoint Hydration

Move from forensic checkpoint capsules toward hydrateable checkpoint candidates without claiming crash-safe resume. The next slice should make every runtime input explicit, validator-derived, and inspectable before any GameRunner.fromCheckpoint() path exists.

Repo Grounding

Grounding Context

  • The current checkpoint is intentionally forensic. GameCheckpointCapsule records projection summary, transcript entry count, missing inputs, and hydrateable: false. The API persists that posture with a degraded reason rather than pretending the checkpoint can restore a runner.
  • The durable event/read-model lane created an inspection surface, not resume. The API can persist canonical events, validate owner/event/projection alignment, and summarize durable-run state. It still has no runtime hydration path.
  • The missing runtime inputs are concrete. The next slice must account for XState actor position, phase-local accumulators, pending LLM/effect work, agent memory and strategy packets, House producer state, transcript/outbox cursors, token/cost cursors, and worker ownership.
  • Canonical events are board truth, but not the whole runtime. Replaying canonical events can rebuild accepted game facts, but it cannot reconstruct private beliefs, partial phase collections, stream delivery state, token accounting, or in-flight model effects by itself.
  • Private evidence must stay private evidence. Raw prompts, raw responses, hidden reasoning, Strategy Thread state, and House producer reads may support debugging or continuity, but they should not become public transcript or canonical event truth.
  • The safest next move is validator-derived readiness. A hydrateable claim should be earned by checks over stored events, projection hashes, runtime subcontracts, cursor reconciliation, and privacy boundaries, not set directly by the writer.

Surface Map

Topic Axes

  • Runtime Snapshot Contract Naming the subcontracts that make up a future hydrateable snapshot, without collapsing projection truth, actor state, and private producer continuity into one JSON blob.
  • Phase-Boundary Safety Proving a checkpoint is taken at a quiet barrier where no old-process LLM call, phase accumulator, or actor entry action can later mutate state.
  • Agent And House Continuity State Preserving strategic memory and producer context while keeping beliefs, reasoning, and House reads private and non-canonical.
  • Token, Transcript, And Cursor State Turning entry counts and terminal accounting into durable boundaries that survive process loss and reconcile with model effects.
  • API Checkpoint Storage And Validation Making checkpoint readiness visible and validator-derived in the admin/read-model layer before a resume action exists.

Idea 1

Hydration Passport With Promotion Certificate

API Checkpoint Storage And Validation Confidence 88% Complexity Medium

Description: Replace boolean hydrateability with a typed readiness passport. Each checkpoint reports per-subsystem status for event/projection, actor state, phase accumulators, agent/House continuity, transcript cursor, token cursor, owner epoch, and privacy boundary. Status values should distinguish absent, captured, validated, stale, malformed, private-reference-only, forbidden, and forensic-only. A promotion validator, not the writer, derives whether a checkpoint is still forensic, a candidate, rejected, or eventually hydrateable.

Basis
direct: checkpoint capsules already enumerate missing hydration inputs; game_checkpoints stores hydrateable, hydrationStatus, snapshot, transcript cursor, token cursor, and degraded reason; the durable-run read model already fails closed when stored hydrateability is malformed.
Rationale
This creates progress without a resume promise. Future slices can remove one named blocker at a time, and the admin surface can answer which subsystem blocks hydration today.
Downsides
The status vocabulary can sprawl. Keep v1 tied to the current missing input list and require every new status to have a validator, not just display copy.
Idea 2

Quiet-Barrier Phase Boundary Certificate

Phase-Boundary Safety Confidence 84% Complexity Medium

Description: Only treat a phase-boundary checkpoint as a hydration candidate when the runner proves a quiet barrier: durable events are flushed, no pending LLM call or retry can still commit, no phase-local collection remains uncommitted, and actor entry/exit effects have receipts. The certificate should explicitly cover boundary effects that must not run twice, such as round increments, lobby/reckoning entry actions, durable flush completion, checkpoint write completion, and stream-buffer release status.

Basis
direct: current durable checkpoints are written around phase boundaries, while the runtime still lists pendingLlmCalls and phase accumulators as missing. Several phases collect LLM/model outputs in parallel before committing accepted results. external: XState persisted snapshots restore actor internals, but actions are not re-executed and invocations restart.
Rationale
The important safety property is not "a checkpoint was written after a handler"; it is "nothing from the old process can later commit, and boundary side effects will not be duplicated or skipped."
Downsides
Some phases may fail the certificate at first. That is useful signal, but the UI must call those checkpoints forensic rather than broken.
Idea 3

Runtime Snapshot Manifest

Runtime Snapshot Contract Confidence 82% Complexity Medium

Description: Define the hydrateable runtime snapshot as a manifest of subcontracts before serializing a larger blob. The manifest should name which systems are replayed, serialized, omitted, or blocked: canonical projection replay, XState actor value/context, runner phase accumulators, private agent memory, House producer continuity, transcript/outbox cursor, token/cost cursor, and owner epoch.

Basis
direct: getStateSnapshot() is viewer-shaped, not resume-shaped. GameCheckpointCapsule names missing runtime inputs but does not define their serialized contracts. Runner private fields, XState context, agent state, House state, logger state, and token tracker state live in separate trust domains.
Rationale
This prevents "hydrateable checkpoint" from becoming an unreviewable bag of JSON and lets reviewers reason about projection truth, actor progress, and private continuity independently.
Downsides
A manifest can become architecture paperwork if it is not enforced. Pair each row with a validator and tests before calling it part of readiness.
Idea 4

Private Continuity Capsule

Agent And House Continuity State Confidence 80% Complexity Medium-High

Description: Persist private continuity capsules for agents and House. The capsule would include strategy packet revision, prior revision links, last reflection summary, scrubbed ally/threat/notes state, House Strategy Bible revision, producer brief references, privacy labels, redacted summaries, source pointers, and hashes. It hydrates "what this agent or House believed," while canonical replay remains the only source of "what happened."

Basis
direct: Strategy Thread packets, reflections, agent notes, and House state are currently live/private runtime state. Private agent-turn records and producer evidence are intentionally not public broadcast or canonical board facts.
Rationale
Watchable strategy depends on continuity after restart, but promoting hidden beliefs or raw reasoning into canonical events would corrupt the privacy and truth model.
Downsides
This is the easiest area to over-capture. The capsule should store revision metadata and scrubbed continuity fields, not raw prompt logs or hidden reasoning as resume truth.
Idea 5

Durable Transcript / Outbox Cursor

Token, Transcript, And Cursor State Confidence 78% Complexity Medium

Description: Replace the current transcript entry-count cursor with a durable boundary over transcript rows, private turn rows, outbox delivery state, source pointers, and canonical event sequence. A checkpoint should validate that the transcript cursor, persisted rows, stream buffer, and replay/export ordering agree before transcript state contributes to readiness.

Basis
direct: checkpoint capsules currently record transcriptCursor.entries. Checkpoint writes can happen before stream-buffer flushes, and terminal transcript exports can depend on persisted ordering that is not the same as in-memory logger length.
Rationale
Viewer continuity is part of game continuity. A checkpoint whose transcript cursor points at memory that never reached durable storage is more misleading than a checkpoint that honestly says transcript continuity is absent.
Downsides
This may require choosing or adding durable transcript/outbox rows earlier than the pure runner snapshot work would otherwise demand.
Idea 6

Token Ledger First Hydrateable Component

Token, Transcript, And Cursor State Confidence 76% Complexity Low-Medium

Description: Make token/cost state the first small runtime subsystem with a versioned snapshot/fromJSON contract. The checkpoint should capture monotonic per-source totals, House tracker wiring, token deltas linked to accepted model-effect receipts, and a validator that can prove the cursor did not move backward or duplicate model costs.

Basis
direct: TokenTracker already exposes aggregate usage and merge behavior. Terminal API persistence stores totals and per-action usage, but checkpoint tokenCostCursor is currently null or undefined.
Rationale
Token state is low blast-radius compared with actor or agent hydration, and it gives the passport a first genuine green check while making expensive model effects easier to reconcile.
Downsides
It is not enough for resume by itself. The implementation must keep the UI language clear: token ledger readiness is one stamp, not hydrateable game readiness.
Idea 7

Replay-Verified Snapshot Witness

Runtime Snapshot Contract Confidence 74% Complexity Medium

Description: Build a read-only witness that loads a candidate capsule, replays canonical events to the checkpoint boundary, compares projection hash and summary expectations, and identifies which mutable runtime fields are replay-derived versus runtime-only. It never creates a live runner and never exposes a resume action.

Basis
direct: canonical replay and projection hashes already exist, while mutable GameState and runner internals do not have a hydrate partner. external: event-sourcing snapshots are acceleration overlays; the event stream remains the source of truth.
Rationale
This gives richer snapshots a safety rule: the snapshot can accelerate or annotate, but it cannot override replayed canonical truth.
Downsides
The witness can look like a dry-run resume if named carelessly. Keep it read-only and phrase failures as contract gaps, not runtime recovery failures.
