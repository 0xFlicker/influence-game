OAuth Gate for Global Game MCP Access

A narrowed V0: add an mcp role, allow only that role to authorize the mcp OAuth scope via authorization code plus PKCE, and treat a valid mcp token as global Game MCP access for testing and validation.

Grounding Context

Codebase Context. The initial ideation over-weighted eventual access-control shape: per-agent grants, per-game resources, consent ledgers, and private trace boundaries. The corrected first slice is narrower and more useful: create a working OAuth 2.1-style loop for the Game MCP where possession of a role-bound mcp scope grants intentionally global MCP access.

Game MCP is the target

The repo's Game MCP is a read-only query surface over simulation artifacts. It lists sessions and games, rebuilds projections from event logs, searches logs, reads timelines, and exposes artifact resource URIs.

RBAC already has the hook

The API already seeds roles and permissions, assigns roles by wallet address, and includes roles and permissions in the backend session JWT. A new mcp role fits the existing RBAC shape.

The scope is deliberately global

For V0, scope=mcp is not a row-level or agent-level grant. If the user has the mcp role, they can authorize the scope; a token with that scope can access the Game MCP globally.

Trace and private-agent boundaries wait

Trace MCP, private content, per-agent consent, and per-game grants remain future cuts. They should not block the first OAuth validation loop.

The security posture is intentionally narrow rather than elaborate: only role-bound users can mint the scope, the scope name is honest about its global authority, and the implementation is for trusted testing and validation before any broader user-facing delegation model exists.

V0 Scope Boundary

  • In: seed a new mcp role.
  • In: expose OAuth authorization-code plus PKCE for scope=mcp.
  • In: authorize scope=mcp only when the logged-in user has the mcp role.
  • In: make a valid mcp token grant global Game MCP access.
  • Out: per-agent, per-game, row-level, or private trace authorization.
  • Out: dynamic client registration, browser refresh tokens, and external multi-client policy.

Idea 1.

mcp Role plus mcp Scope as the V0 Contract

Axis
Role-gated OAuth entry
Confidence
92%
Complexity
Medium
Description
Add a system role named mcp. A logged-in user with that role may authorize exactly one initial OAuth scope, mcp. The scope is intentionally global: a token bearing it means the holder can use the Game MCP without per-game, per-agent, or private-boundary checks in this pass.
Basis
direct:packages/api/src/db/rbac-seed.ts already seeds role and permission records, and address_roles already binds wallet addresses to roles. direct:The Game MCP is documented as read-only over local simulation artifacts, which makes a global maintainer/testing scope a reasonable V0.
Rationale
This is the strongest correction because it turns OAuth from a broad product authorization system into a focused gate for one trusted user role. It gives the implementation a simple invariant: role check at authorization time, global Game MCP authority at resource time.
Downsides
The scope is powerful by design. The UI and docs must call it global access, and assigning the mcp role should remain deliberate. This is not suitable as a public user delegation model without later narrowing work.

Idea 2.

Minimal PKCE Authorization Server for First-Party MCP

Axis
Token validation and session separation
Confidence
84%
Complexity
Medium
Description
Implement the smallest authorization-code plus PKCE surface needed for a first-party MCP client: authorization endpoint, code storage, token exchange, token validation, and a static client configuration. The authorize step uses the existing app session to identify the user and checks the mcp role before issuing a code for scope=mcp.
Basis
direct:The current login flow already turns Privy identity into a backend session, so authorization can reuse the app session as the human-auth prerequisite. external:OAuth 2.1 draft -15 centers authorization code plus PKCE for browser/public-client flows.
Rationale
This keeps the work standards-shaped while avoiding a full product authorization platform. No refresh tokens, no dynamic client registration, and no consent matrix are needed for the first Game MCP validation loop.
Downsides
Even a minimal flow needs careful code expiry, redirect URI checks, PKCE verifier validation, and token signing/verification tests. The payoff is a real OAuth loop instead of a custom API key.

Idea 3.

Protected HTTP Game MCP Resource Server

Axis
Game MCP resource exposure
Confidence
80%
Complexity
High
Description
Expose the Game MCP through a protected HTTP transport or API-backed wrapper that validates bearer tokens for scope=mcp. Once validated, the request gets the same read-only global capabilities as the current local Game MCP: sessions, games, projections, log search, timelines, linked records, and artifact resources.
Basis
direct:packages/engine/src/game-mcp/server.ts already implements the JSON-RPC tool/resource behavior over a read model. external:The MCP 2025-06-18 authorization spec uses OAuth 2.1-style authorization and protected resource metadata for HTTP transports.
Rationale
The OAuth flow only matters if a real MCP client can use the token. Protecting a Game MCP HTTP surface is the narrow end-to-end test: authorize, exchange, connect, list/read/search.
Downsides
This is the largest implementation slice because the current Game MCP is local stdio/corpus oriented. The wrapper should reuse the existing read model and avoid changing the underlying MCP query semantics.

Idea 4.

Seeded Maintainer Smoke Loop

Axis
Developer validation loop
Confidence
86%
Complexity
Low
Description
Add an explicit smoke path that seeds or assigns the mcp role to a maintainer address, runs the OAuth PKCE flow, obtains a token with scope=mcp, and verifies Game MCP calls such as initialize, list sessions, list games, and read projection. The negative path should prove a non-role user cannot authorize the scope.
Basis
direct:The repo already treats Game MCP smoke and current-session checks as a first-class validation pattern. reasoned:A global test scope is acceptable only if role assignment and denied authorization are both easy to verify.
Rationale
This turns the slice into a concrete validation loop instead of an abstract auth implementation. It also matches the stated use: one trusted maintainer testing and validating the Game MCP.
Downsides
Local OAuth testing can be fiddly because redirects, ports, and browser session state all matter. The smoke should document exact local URLs and expected denial points.

Idea 5.

Explicit Fine-Grained Grant Deferral

Axis
Explicit deferral of fine-grained grants
Confidence
88%
Complexity
Low
Description
Write the V0 docs and code comments so the global nature of scope=mcp is impossible to miss. Per-agent grants, per-game row boundaries, private trace scopes, consent receipts, and revocation dashboards are future work, not hidden requirements of this slice.
Basis
direct:The repo already separates Game MCP, Trace MCP, public/canonical game state, and producer-visible artifacts in documentation. reasoned:Making the deferral explicit prevents the first OAuth implementation from being dragged back into broad access-control design.
Rationale
The biggest risk is scope creep, not insufficient security architecture. This idea protects the narrow loop by documenting what the token can do, who can mint it, and why that is intentionally enough for V0.
Downsides
The deferral can look blunt to a future reviewer. The fix is to label it as a maintainer/testing boundary and pair it with role-gated issuance plus a denial test.
