Ideation

The House Presents Influence

Ranked directions for repositioning thehouse.game as the venue while keeping Influence as the first and only available game in this pass.

Date: 2026-06-30 Topic: the-house-influence-rebrand Mode: repo-grounded Output: html

Grounding Context

Codebase Context

Influence is a Bun and TypeScript monorepo with engine, API, and Next.js web packages. The homepage is currently just Nav plus HomepageHero, so visitor-facing rebrand copy can land without a sweeping page rebuild.

Product Context

The strategy doc frames Influence as an agent-development and spectator loop for crypto-native AI/game tinkerers. The user asked for the top-level mental model: The House -> all games -> Influence.

Current Friction

Root metadata, nav, game cards, creation, and queue surfaces still behave as if the only brand is Influence. That is fine for the old domain, but unclear when a visitor lands on thehouse.game.

MCP Boundary

The user-facing MCP split is already strong: /mcp with scope=games supports rules, owned agents, supported queues, and game inspection; /mcp/producer remains producer-only.

Past learnings: production MCP role/resource split docs warn that scope=mcp must not be reinterpreted as player access. Agent-management/queue planning also says queue and rule vocabulary should be LLM-legible, server-owned, and honest about current support.

External context: platform examples support the same direction: Jackbox keeps a top-level brand with individual game titles; Board Game Arena separates platform navigation from game pages; Blood on the Clocktower separates rule/how-to/storyteller resources; Among Us keeps the game hook crisp while rules live elsewhere.

Topic Axes

  • Domain and home framing. How a visitor understands The House and Influence in the first viewport, nav, and metadata.
  • Games list identity and discovery. How lists, cards, filters, and empty states identify Influence as the current game.
  • Create and queue entry. How game creation and the free queue communicate that the selected/entered game is Influence.
  • Rules and docs IA. How Influence rules sit under The House without confusing the in-game House moderator role.
  • MCP and agent-management consolidation. How agent, queue, and MCP setup information forms one coherent pre-match story.

Ranked Ideas

1. Venue Pass 2. Games List 3. Create + Queue 4. Rulebook 5. Agent Gateway 6. Identity Constant 7. No Empty Shelves
1

The House Venue Pass

Confidence 92%Complexity Medium

A coherent first implementation pass: update root metadata, nav/hero copy, CTA labels, games-list labels, creation copy, free-queue copy, and rules framing around "The House presents Influence". Keep internals and routes stable so this is a visitor-facing mental-model pass, not a codebase rename.

Axis
Domain and home framing
Basis
direct: root page is only Nav plus HomepageHero; metadata and nav say only "Influence"; the user asked for tangible first-pass changes without a sweeping homepage.
Rationale
This resolves the domain confusion immediately while preserving Influence as the actual product.
Downsides
Touches several text surfaces and tests; needs a tight scope checklist to avoid accidental platform work.
2

Influence-Stamped Games List

Confidence 88%Complexity Low

Retitle or frame /games as "All games" and label every current row/card as Influence. Search, empty states, counts, and filters should understand "Influence" as the only available game title.

Axis
Games list identity and discovery
Basis
direct: current cards show Game #, status, free badge, and phase, but no game identity; the search haystack omits "Influence".
Rationale
This makes the "all games -> Influence" hierarchy real at the exact moment viewers browse games.
Downsides
A one-game "All games" page can feel overbuilt unless the copy stays restrained and avoids empty future-game shelves.
3

Influence-Selected Create And Queue Entry

Confidence 87%Complexity Medium

Add a read-only "Game: Influence" ruleset card on game creation, then rename free-queue UI and display copy to "Influence Daily Queue" / "Join Influence Queue". Keep wire values such as daily-free and open-game unchanged.

Axis
Create and queue entry
Basis
direct: creation starts at player/model parameters with no game selection; free queue copy says generic "Join Queue" and service labels say "Daily Free".
Rationale
Players and agent owners should know they are creating or entering Influence, not a generic game slot.
Downsides
Service display-label changes may affect tests or MCP snapshots; compatibility values should remain stable.
4

Influence Rulebook Under The House

Confidence 84%Complexity Medium

Reframe /rules as a rulebook area under The House with a single active Influence section. Clarify that The House is the venue brand and, inside Influence, the moderator role; fix obvious stale rules wording while there.

Axis
Rules and docs IA
Basis
direct: rules already define The House as moderator and contain deeply Influence-specific phases; markdown rules still contain legacy "Whisper" wording.
Rationale
This prevents the brand container from colliding with game-role terminology while grouping Influence rules as one game under the venue.
Downsides
A full structured-rules refactor is attractive but should remain separate from this first pass.
5

House Agent Gateway

Confidence 81%Complexity Medium

Consolidate user-facing MCP, agent, and queue copy around one story: The House lets your agents enter Influence and lets AI tools manage pre-match prep. Keep /mcp as scope=games, keep /mcp/producer internal, and update /get-mcp, the dashboard card, and docs headings without changing protocol names.

Axis
MCP and agent-management consolidation
Basis
direct: production MCP docs and server instructions already expose rules, owned agents, supported queues, and games under scope=games, with no active-match actions.
Rationale
The rebrand can turn a set of technically correct surfaces into one understandable product lane for agent owners.
Downsides
Copy must avoid making "The House MCP" sound like a new protocol or non-Influence data source.
6

Presentation-Layer Game Identity Constant

Confidence 76%Complexity Low

Introduce a tiny presentation-level identity object for the current game: id: "influence", title "Influence", venue "The House". Thread it through UI copy/search/tests instead of scattering strings, without adding a DB ruleset table.

Axis
Games list identity and discovery
Basis
reasoned: UI needs game identity now, but persistent multi-game architecture and per-game agent strategy are explicitly out of scope.
Rationale
This buys consistency and future readability without paying for a real multi-game model before there is a second game.
Downsides
Could become faux architecture if it grows beyond display copy.
7

No Empty Shelves Principle

Confidence 90%Complexity Low

Use this as the rebrand guardrail: make room for future games through hierarchy and naming, not disabled Werewolf/Mafia/Salem tiles, fake roadmap cards, dead links, or schema migrations. The visitor should see what works today.

Axis
Domain and home framing
Basis
direct: the user explicitly scoped future social deduction games to "making space" and excluded actually adding other games.
Rationale
This keeps the pass honest: the new brand becomes a venue, not theater around unavailable products.
Downsides
It is a principle more than a feature, so it should ride alongside implementation rather than stand alone.
