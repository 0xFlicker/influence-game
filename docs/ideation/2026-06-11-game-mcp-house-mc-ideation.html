Game MCP, Replayable Events, and House MC

A ranked direction set for turning live and completed simulation artifacts into a queryable MCP surface, then tightening the scope toward a canonical event stream that can rebuild game state without creating a parallel phase machine.

Grounding Context

Codebase Context

The simulator already produces the core raw material. Each game writes clean structured turn records to game-N-turns.jsonl, lightweight progress events to game-N-progress.jsonl, a full transcript/result bundle to game-N.json, and a human-readable transcript to game-N.txt. The docs explicitly tell humans and scripts to use the turn JSONL for structured analysis rather than parsing colored terminal output.

The strongest existing seam is AgentTurnEvent: round, phase, action, actor, visibility, normalized response, optional thinking, optional reasoningContext, scope, text, recipients, room id, anonymity, and display order. The simulator attaches a stream listener, writes turn events to the turns JSONL, and writes progress/transcript-preview events to the progress JSONL.

The House MC currently gets recent transcript slices directly and emits freeform summaries after Mingle and Council. Meanwhile, agent prompt construction contains richer state than the turn log preserves: memory, notes, vote history, strategic assessment, recent public messages, private Mingle messages, and room context. Strategic reflections already produce useful strategy objects, but simulator config currently disables them to keep validation runs bounded.

Statefulness is a separate risk. A live MCP can tail and query logs during a run, but it should not be described as crash-safe recovery. The existing statefulness plan is explicit that mid-game process loss still kills active games until checkpoint/resume work lands.

External MCP fit. MCP maps neatly to this problem: resources can expose sessions, logs, transcripts, and derived fact files; tools can provide filtered search and domain queries; stdio is the simplest local transport; Streamable HTTP becomes useful later for multiple live clients or server-sent updates. Official MCP guidance also makes security and access boundaries first-class, which matters because hidden reasoning and Mingle privacy should not leak into player-visible surfaces.

Rerun: Event Stream Scope

Seven New Survivors

The first pass treated the MCP as a local production-booth sidecar over the logs that already exist. The rerun keeps that useful read-only direction, but changes the center of gravity: the strongest next scope is a canonical domain event stream under GameState, with MCP indexes as projections over that stream. XState should remain the phase authority and resume cursor; the reducer should rebuild board/domain state only.

R1

Canonical Game Event Spine Under GameState

Conf 88% Cx Medium

Introduce a typed GameEvent stream for accepted domain facts at the exact points where the game mutates: round start, room allocation, vote cast, empower tally resolved, power action set, shield granted, council candidates chosen, council vote cast, player eliminated, endgame stage set, jury vote cast, and winner determined. The key is that events represent accepted outcomes, not model suggestions or transcript text. This lets a reducer rebuild board state without re-implementing phase flow.

Axis
Canonical event spine and schema
Basis
direct: phase-machine.ts:4 says XState sequences phases while game logic lives in GameState; game-state.ts:29 owns the mutable domain fields.
Rationale
It avoids the parallel-state-machine trap by placing the event spine at the current domain authority rather than outside the engine.
Downsides
Requires discipline around event vocabulary and versioning. Random/fallback outcomes must be recorded as accepted facts, not recomputed during replay.
R2

Append-Then-Apply GameState Facade

Conf 82% Cx Medium

Add a small evented facade around GameState so phase modules call intentful methods like recordVote, resolveEmpowered, grantShield, and eliminatePlayer through append-then-apply. The first implementation can delegate to existing GameState internals after appending, but the event reducer and live mutation path should share one apply function as soon as practical.

Axis
Reducer and projection correctness
Basis
direct: phase modules currently call mutators directly, for example runVotePhase records votes and runPowerPhase calls setPowerAction then determineCandidates.
Rationale
This is a contained migration path: the engine keeps running normally while the canonical log is created at the mutation source.
Downsides
There is a temporary dual path risk until direct mutator calls are removed or guarded by tests.
R3

Projection Parity Test Harness

Conf 86% Cx Medium

Build tests that replay canonical events into a fresh reducer projection and compare the result against GameRunner.getStateSnapshot() at phase boundaries and at game end. Start with deterministic unit scenarios, then run a short full-game fixture. This makes the reducer prove it is a read model of the engine, not an alternate engine with flattering assumptions.

Axis
Reducer and projection correctness
Basis
direct: existing stream-listener tests already capture getStateSnapshot() during a run, but do not replay events or compare projections.
Rationale
The parity harness is the safety rail that lets the event stream become more authoritative over time.
Downsides
Snapshot comparison needs stable normalization so timestamps, transcript ordering, and non-state observability fields do not create noise.
R4

Dual Timeline: Domain Events Plus XState Snapshot

Conf 84% Cx Medium

Persist two complementary records: the domain event log rebuilds GameState, while the XState persisted snapshot records where the phase actor is and which transition context it owns. On resume, rebuild or load the domain projection, restore the XState actor from its snapshot, and reconnect runner extras like transcript, Mingle inbox, diary entries, token usage, and recap context.

Axis
XState and resume boundary
Basis
direct: docs/statefulness-plan.md:80 already calls for serializing GameState, XState snapshot, runner data, and token usage; external: XState docs support actor persisted snapshots and restore.
Rationale
This answers the redesign question: the event stream does not prime XState from scratch; it resumes beside XState and feeds domain projections.
Downsides
Mid-phase resume remains hard because in-flight LLM calls and per-agent conversation state are not fully recoverable.
R5

Event-Backed MCP Read Models

Conf 80% Cx Medium

Reframe the Game MCP as a read-model host over canonical events and derived projections: transcript view, player timeline, relationship graph, vote ledger, room conversations, strategy snapshots, and FTS search. Existing JSONL sidecar indexing can still ship first, but the durable target is that every MCP answer can point back to a canonical event, not only a transcript line.

Axis
MCP read-model query surface
Basis
direct: GameStreamEvent and AgentTurnEvent already expose rich observability fields; external: MCP resources and tools are designed for application-specific context and callable structured queries.
Rationale
The House can learn from the same projection infrastructure as post-run analysis, instead of getting a bespoke prompt-stuffing path.
Downsides
Early projections may lag the canonical event schema. Keep derived facts rebuildable and citation-backed.
R6

Source Pointers and Visibility on Every Event

Conf 90% Cx Low

Put source pointers and visibility on the canonical event envelope from day one: sequence number, timestamp, round, phase, actor, causation/correlation IDs, visibility tier, source file/line for simulator artifacts when applicable, and payload version. The visibility tier should feed MCP access modes like producer, audience, and player-visible.

Axis
MCP read-model query surface
Basis
direct: reasoning observability docs require hidden thinking to stay separate from player-visible output, and existing turn events already carry visibility/scope fields.
Rationale
It keeps House and analysis queries powerful without accidentally giving active players omniscient access.
Downsides
Visibility mistakes are subtle; tests should cover player-visible, audience, and producer queries before live use.
R7

Simulator-to-API Event Log Migration Ladder

Conf 78% Cx Medium

Sequence adoption in three steps: first write canonical event JSONL from simulator runs, then persist the same events for API games in Postgres, then let API snapshots/checkpoints be projections over the event stream. This keeps the first implementation local and observable while making it a real springboard toward stateless app servers.

Axis
Simulator and API migration
Basis
direct: simulator artifacts are already local JSONL, while API games still depend on an in-memory activeGames map and transcript persistence drops full AgentTurnEvent.response / reasoningContext.
Rationale
This makes the stateless-backend path incremental: stateless servers become plausible only after events are durable for API games too.
Downsides
Requires schema/migration work and careful ownership rules so two API instances cannot append conflicting events for the same game.

I1

Local Production-Booth MCP

Conf 90% Cx Medium

Build the first game MCP as a local, read-only sidecar over existing simulation artifacts. It lists sessions and games, exposes logs/transcripts as MCP resources, and provides narrow query tools over the current batch directory without changing the game loop. This is the best first direction because it creates a real connection point for the House or another agent while keeping the simulator semantics stable.

Axis
MCP API and query grammar
Basis
direct: simulate.ts writes game-N-turns.jsonl and game-N-progress.jsonl; AgentTurnEvent already has queryable fields.
Rationale
A sidecar lets you connect external agents immediately while keeping the game loop and state machine understandable.
Downsides
It will initially answer from logs only. It will not solve checkpoint/resume, and it may need careful path discovery for partial batches.
I2

Evidence-Backed House MC

Conf 82% Cx Medium

Change the House from "summarize the last N transcript entries" to "ask the game MCP for cited evidence, then narrate." The House could query recent Mingle conversations, vote reasoning, player timelines, contradictions, or standout moments, and the tool result would include event IDs and source pointers. This should make summaries and diary questions sharper while reducing prompt stuffing.

Axis
House integration and information boundaries
Basis
direct: house-interviewer.ts builds House summaries from recent transcript slices, while House prompts demand specific players, quotes, or events.
Rationale
Evidence-backed retrieval lets the House be more perceptive without overloading every prompt with raw logs.
Downsides
The House can become too omniscient or spoilery unless paired with a visibility contract.
I3

Strategy Snapshot Stream

Conf 78% Cx Medium

Add a structured stream of player strategy snapshots: certainties, suspicions, allies, threats, plan, and optionally a redacted prompt-context summary. The code already has a strategic reflection tool shape, but simulator validation currently disables it. Making this opt-in for local analysis would let a future House ask "what did Vera believe before this vote?" or "which plan did Atlas abandon?"

Axis
Richer strategy and log schema
Basis
direct: getStrategicReflection records certainties, suspicions, allies, threats, and plan; buildUserPrompt contains the dynamic memory and Mingle context that later queries want.
Rationale
Raw decisions explain what happened. Strategy snapshots explain what changed in a player's model of the game.
Downsides
Extra LLM calls are expensive and may slow long local runs. Start with opt-in local flags or low-frequency snapshots.
I4

SQLite FTS Event Index With Source Pointers

Conf 84% Cx Medium

Materialize JSONL records into a small local SQLite database with FTS over text, thinking, reasoning, and normalized response fields. Keep source pointers on every row: batch directory, game number, file path, line or byte offset, event timestamp, phase, action, actor, scope, and visibility. The MCP tools can then answer quickly across one live run or many completed sessions.

Axis
Live log ingestion and indexing
Basis
reasoned: JSONL scans are fine for one game, but cross-session questions and live tailing benefit from structured filters plus lexical search.
Rationale
FTS handles exact names, quotes, phases, and action vocabulary before adding semantic embeddings.
Downsides
Needs schema versioning and tailer correctness. It also introduces a derived store that must stay rebuildable from source logs.
I5

Conversation and Relationship Query Pack

Conf 86% Cx Medium

Do not expose only a generic search box. Give agents a small set of game-native tools: search_turns, get_conversation, get_player_timeline, get_reasoning, find_commitments, and summarize_relationship. Each tool should return structured results plus citations, so a downstream House or analysis agent can move from "what happened?" to "what did this mean socially?"

Axis
MCP API and query grammar
Basis
direct: turn records include actor, text, phase, recipients, room id, visibility, action, response, thinking, and reasoningContext.
Rationale
Game-native tools make the MCP useful to agents that need social meaning, not just keyword hits.
Downsides
Derived relationship facts can hallucinate confidence if extraction is not citation-backed. Keep derived claims auditable.
I6

Role-Based Visibility Contract

Conf 88% Cx Low

Define access modes before the MCP touches any active agent: producer can inspect full logs for debugging and House MC, audience can see viewer-safe private-room content but not privileged native reasoning by default, and player-visible sees only what a contestant would know. Hidden thinking and reasoningContext require explicit tool arguments and should be off by default.

Axis
House integration and information boundaries
Basis
direct: project docs say hidden reasoning is for viewers, replays, and debug surfaces, and public player text must never contain hidden thinking.
Rationale
This keeps the query surface from accidentally turning Mingle into omniscient gameplay.
Downsides
Requires discipline in tool defaults and tests. It may feel slower than simply exposing everything.
I7

Multi-Session Model Evaluation Corpus

Conf 75% Cx Medium

Let the same MCP query multiple simulation batches after the fact: "show all places this model repeated lobby filler," "which Mingle turns contained real alliance movement?", "where did reasoning mention a plan that the public message ignored?", or "compare Gemma runs by completion and watchability." This turns local model experimentation into an accumulating corpus rather than isolated text files.

Axis
Multi-session analysis and replay leverage
Basis
direct: local model docs ask evaluators to record good strategy, bad strategy, repetition, empty responses, completion, and reasoning usefulness.
Rationale
The same MCP that helps the House live can become the post-run analyst for model quality and game watchability.
Downsides
Cross-session comparisons need normalized metadata for model, variant, players, branch, and dirty state to avoid misleading conclusions.
