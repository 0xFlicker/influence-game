Ideation: House Highlights Endgame Rendering Pipeline

Fresh ideation pass for taking the proven local House Highlights trailer renderer into a production end-of-game media pipeline: final render trigger, durable upload, a deployable render-worker container, accessible playback, and share tooling.

Date: 2026-07-09 Topic: house-highlights-endgame-rendering-pipeline Mode: repo-grounded Status: ideation, not implementation planning Scope floor: third render container required

Grounding Context

Codebase Context

  • The current local renderer lives in packages/web/src/scripts/render-house-highlights-trailer.ts. It fetches public House Highlights and completed-results data, builds a manifest, bundles Remotion, renders MP4, and writes cue JSON.
  • The manifest boundary in house-highlights-trailer-model.ts already refuses non-completed games, requires final jury/finalist/winner facts, emits cue segments, and keeps trailer facts as presentation-ready props.
  • The Remotion composition renders from props only. This is the right truth boundary: renderer animates; it does not choose scenes, infer votes, or mutate facts.
  • Existing Highlights scene sharing already uses contextual share URLs and dynamic OG card images that land viewers back on the full Highlights page.
  • Existing public upload storage uses Linode/Akamai Object Storage through S3-compatible helpers, with local filesystem fallback for development.
  • Private trace storage provides a useful parallel pattern: large bytes in object storage, safe manifests/checksums/status in Postgres, no raw internal storage keys on public surfaces.
  • Current Docker images are split into API and web. The web image is optimized for Next standalone serving, not headless Chrome/ffmpeg rendering.
  • Scope clarification: this production slice is not done until the repo has a deployable third container for rendering and a concrete handoff package for linode-iac to wire into deployment.

External Context

  • Remotion renderMedia() is the programmatic rendering API and accepts composition, bundle/serve URL, codec, output location, input props, and progress callbacks.
  • Remotion Docker guidance calls out headless Chrome dependencies and browser installation; this is a real runtime concern, not just a package.json concern.
  • Akamai/Linode Object Storage is S3-compatible and stores objects in buckets with metadata, matching the repo's existing storage posture.
  • MDN video guidance recommends captions and transcripts for accessibility; the <track> element supports captions, descriptions, chapters, and metadata.

Completion Bar

This slice should not be considered complete merely because the API can record media, the web can display a trailer player, or a local command can render a final MP4. The production boundary now includes a third deployed service: a render-worker container that can be built, smoke-tested, configured, and handed off to linode-iac.

  • The worker container must include the Remotion, Chromium/headless-browser, ffmpeg, asset, and music-muxing runtime needs.
  • The API/web containers should not absorb the rendering runtime just to ship faster.
  • The app repo should provide the image contract, command shape, environment variables, storage expectations, health/smoke command, and job-status behavior needed for deployment wiring.
  • The linode-iac handoff should be specific enough that deployment work is mostly compose/env/secret wiring, not reverse-engineering this repo's render path.

Ranked Ideas

1

Postgame Media Bundle

Axis: Durable media Confidence: 93% Complexity: Medium-high Status: Unexplored

Treat the public trailer as a durable postgame media bundle rather than "an MP4 somewhere." The bundle should include MP4, poster frame, cue JSON, transcript or WebVTT, manifest hash, duration, content type, byte size, object keys, render version, and public status.

Basis: direct: current private trace architecture stores large bytes in S3-compatible storage while Postgres owns safe manifest metadata; the current trailer renderer already emits MP4 plus cue JSON.

Rationale: This gives playback, sharing, rerendering, and debugging one product contract. Object storage owns bytes; the database owns truth about whether the trailer is ready and what generated it.

Downsides: Requires durable schema/storage work and careful public/private key separation.

2

Dedicated Render Worker Image With Local Parity

Axis: Render runtime Confidence: 88% Complexity: High Status: Unexplored

Package rendering as a dedicated worker/container command that runs the same manifest-to-video renderer as local development, with Chromium and ffmpeg dependencies installed and a tiny smoke render in validation.

Basis: direct/external: Dockerfile.web is a Next standalone runtime, while Remotion Docker guidance requires headless browser libraries and browser installation for rendering.

Rationale: This avoids stuffing public web/API request containers with fragile video-rendering dependencies. It also makes production failures easier to isolate: the worker either renders, uploads, and records status, or it fails visibly.

Downsides: Adds an image/build/deploy surface. This is no longer optional scope; the slice must produce a clean handoff for the Linode deployment repo.

3

Idempotent Completion-Triggered Render Job

Axis: Completion trigger Confidence: 90% Complexity: Medium Status: Unexplored

When completed results and House Highlights are available, create or run one idempotent trailer render job. The job should not block the completed-game state; render failure updates media status and diagnostics only.

Basis: direct: current product guidance says render-at-completion is acceptable at this scale; the manifest already fails clearly when a game is not completed or lacks final jury/winner facts.

Rationale: Viewers should not wait for render-on-demand, and gameplay completion should not be held hostage by video tooling. This keeps game truth and media publication separate.

Downsides: Needs clear sequencing around completed results, highlights availability, retry behavior, and explicit rerenders.

4

Contextual Accessible Trailer Player

Axis: Playback UX Confidence: 91% Complexity: Medium Status: Unexplored

Add a Highlights/results trailer module using native <video controls>, poster image, transcript/cue section, copy-link button, download/open-file affordance, and honest ready/unavailable states.

Basis: external: MDN recommends captions/transcripts for video, and <track> supports timed text; direct: Visual Card share links already land users back in context.

Rationale: Native video controls are boring in exactly the right way for the first public player. The product value is the trailer and share flow, not custom playback chrome.

Downsides: Native controls are less branded. Transcript and chapter generation needs public-copy-safe cue labels.

5

Contextual Trailer Share URL With Poster OG

Axis: Sharing and ops Confidence: 89% Complexity: Medium Status: Unexplored

Make the share target a web URL for the trailer/player module, with Open Graph and Twitter metadata using the stored poster frame and descriptive title/copy. The MP4 URL is the media source, not the primary share link.

Basis: direct: existing scene card sharing uses route metadata and deterministic OG images; raw video URLs cannot carry fallback states, transcript, surrounding context, or product navigation.

Rationale: Sharing links are the end goal. A contextual share page can handle media not ready, poster-first social previews, analytics, and links back to Highlights/results/replay.

Downsides: Requires metadata coordination between stored media status and Next route metadata.

6

Render Versioning And Admin Rerender Console

Axis: Sharing and ops Confidence: 84% Complexity: Medium-high Status: Unexplored

Store render version, manifest schema, manifest hash, source timestamps, status, failure reason, and explicit producer rerender action. Successful shared videos should remain stable unless intentionally replaced or versioned.

Basis: direct: trailer copy, background, music, and card facts have already changed through iteration. Public media can drift from current code unless each render records what produced it.

Rationale: This prevents silent share-link drift and gives producers a sane way to inspect failures, rerender after improvements, and understand what a viewer saw.

Downsides: Admin tooling can become a rabbit hole. Keep it status-first, not a video CMS.

7

Cue-To-Music-And-Accessibility Bridge

Axis: Playback UX Confidence: 82% Complexity: Medium Status: Unexplored

Promote cue metadata from producer-only JSON into a shared artifact that drives music-bed selection, transcript/chapter generation, and player navigation.

Basis: direct: the current cue sheet defines deterministic segment timing and exact music variants; external: timed tracks can expose chapters, captions, descriptions, and metadata.

Rationale: The same timeline can serve Suno/music composition, public accessibility, and viewer navigation. No need to maintain three subtly different timing systems. That way lies spreadsheet sadness.

Downsides: Cue labels need a public-safe copy pass. Producer labels are not automatically viewer captions.

Open Product Questions

  • What exact render-worker image contract should the app repo hand to linode-iac: image name, command, env vars, mounted temp space, object-storage credentials, API credentials, concurrency, and smoke command?
  • Should successful rerenders replace the canonical trailer URL or publish a new version while preserving old shared links?
  • Should music be required for public trailer publication, or should silent trailers publish with music added later?
  • Should the player live on the Highlights page, results page, or a dedicated trailer route that embeds surrounding context?
  • How much of cue JSON should be public, and what labels need to be rewritten as viewer-facing transcript/chapter text?
  • What exact local parity command should validate the render image on the Linode box before enabling completion-time renders?

Recommended Brainstorm Seed

The strongest next brainstorm seed is Postgame Media Bundle, with the render worker and contextual player treated as connected concerns. That seed can define the product contract for media status, publication semantics, object bundle contents, share URL behavior, accessibility artifacts, rerender policy, and admin diagnostics without dropping into database migrations or route design too early.

Working stance: MP4 is only one artifact. The product needs a durable, shareable, inspectable postgame media bundle. The trailer should feel cinematic, but the publication pipeline should be boringly correct.
