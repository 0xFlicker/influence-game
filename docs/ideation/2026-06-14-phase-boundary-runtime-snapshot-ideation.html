Codebase Context

Grounding Context

The current durable checkpoint path is real but still forensic-only. `GameRunner` flushes canonical events, writes a checkpoint, and flushes the stream buffer at phase boundaries. The API writer then seals the row with owner epoch and event-head receipt evidence. That gives this slice a good durable boundary, but not yet a hydrateable runtime snapshot.

The strongest gaps are specific: the snapshot manifest marks XState actor state, phase accumulators, transcript cursor, token cursor, owner epoch, and continuity as missing; meanwhile, player and House continuity capsules are already collected and persisted beside that manifest. `TokenTracker.toCursor()` already exists, but live checkpoints pass `tokenCostCursor: null`. Transcript checkpoints pass an in-memory entry count, which the passport intentionally rejects as non-durable.

The product boundary is also clear. Phase-boundary v1 should not implement `GameRunner.fromCheckpoint()`, mid-phase recovery, raw prompt/reasoning replay, or an S3-first storage split. It should make the next checkpoint honest, typed, and inspectable enough that future resume work has a real object to trust.

1

Boundary Receipt Runtime Snapshot Contract

Storage/effect boundary Conf 88% Cx Medium

Make the phase-boundary checkpoint a single Postgres truth envelope: event head, projection hash, owner epoch, event receipt, no-pending assertion, manifest stamp table, actor snapshot, accumulator capsule, token cursor, transcript cursor, and continuity evidence.

Basis

direct: `GameRunner` writes checkpoints after durable event flush; the API writer injects owner epoch and event commit receipt; the checkpoint table already has JSON snapshot, transcript cursor, token cursor, event head hash, and projection hash fields. external: transactional outbox guidance supports keeping state and message/cursor evidence in a single durable transaction boundary.

Rationale

This becomes the shared object future passport, resume, admin, and test work all talk about. It also answers the S3 question: v1 runtime truth belongs in Postgres.

Downsides

It is a contract slice, so it can feel abstract unless paired with at least one concrete new stamp. Keep it phase-boundary-only to avoid smuggling resume execution into the contract.

2

Evidence-Derived Manifest Stamp Ledger

Passport/read-model honesty Conf 91% Cx Medium

Derive component statuses from persisted evidence instead of hand-maintained engine booleans. The ledger should allow statuses like `captured`, `private_reference_only`, `durable_cursor`, `diagnostic_only`, `blocked`, `malformed`, and `missing`, each with a reason.

Basis

direct: `writeCheckpoint()` marks player and House continuity as missing while collecting those capsules immediately afterward; the API persists them in the checkpoint snapshot; the passport already treats `private_reference_only` continuity as a captured manifest component.

Rationale

This turns the hydration passport into a trustworthy readiness ledger. Operators should not have to reconcile a manifest that says missing with a payload that proves present.

Downsides

The stamp vocabulary needs restraint. Too many statuses can become a second unclear taxonomy unless every stamp maps to an actual validator path.

3

Boundary-Tap Actor Snapshot Witness

XState actor snapshot Conf 86% Cx Medium

Pass the actual post-handler actor snapshot into checkpoint creation at the boundary and store it as an actor coordinate or witness. It should say where the phase machine stood, not that the game can resume today. Include machine/version metadata and validate the actor coordinate against projection facts.

Basis

direct: the runner creates the XState actor, later reads `actor.getSnapshot()` in the loop, and currently marks `xstateActor` missing. `docs/statefulness-plan.md` names an XState snapshot as checkpoint material. external: XState documents persisted snapshots and restore caveats: actions are not re-executed, invocations restart, and actor state must be serializable.

Rationale

This is the cleanest way to close the actor gap without inventing a reconstruction factory. It also gives tests something concrete to compare against projection round, phase, and boundary sequence.

Downsides

The current code uses `getSnapshot()`, while XState distinguishes emitted snapshots from persisted snapshots. The slice should choose the intended XState shape deliberately and document the semantics.

4

Phase Accumulator Registry With Closed/Empty Proofs

Runner accumulator boundary Conf 84% Cx Medium

Replace vague `phaseAccumulators: missing` with a registry where each runner-owned accumulator reports one of three states: captured payload, drained or empty proof, or intentionally not v1-hydratable. For v1, this should stay strictly phase-boundary and avoid partial in-flight recovery.

Basis

direct: runner-owned state includes `completedHouseSummaryRounds`, `mingleInbox`, `eliminationOrder`, and tribunal accusations. Mingle clears and repopulates room state inside the phase, then records room allocations after `assertCanAcceptCommit()`. The statefulness plan explicitly names `GameRunner` accumulated data as checkpoint material.

Rationale

The registry turns an open-ended audit into a finite checklist. It also gives future phase features a place to declare whether they leave boundary-owned state behind.

Downsides

It forces careful classification. Some accumulators may be safe to prove empty at phase boundaries, while others, especially House summary state, may need captured payloads or explicit blocked status.

5

Durable Cursor Pair

Transcript/token cursors Conf 90% Cx Small/Medium

Persist the token cursor that already exists and split transcript cursoring into two clearly named states: local transcript length for diagnostics, and durable transcript boundary for hydration stamps. A checkpoint can improve token readiness immediately while keeping transcript readiness failed or blocked until there is a real durable watermark.

Basis

direct: the API creates a `TokenTracker`; `TokenTracker.toCursor()` returns versioned totals and per-source usage; live checkpoints still write `tokenCostCursor: null`. The checkpoint currently writes `{ entries }`, and the passport rejects that exact shape as an in-memory count rather than a durable boundary cursor.

Rationale

This is the cheapest positive runtime stamp. It improves cost inspection and validates the cursor/plumbing path without requiring full transcript persistence or resume.

Downsides

A durable transcript watermark still needs a real storage story. The slice should not fake it with entry counts or force every transcript row into mid-game persistence unless that becomes separately justified.

6

Passport Snapshot Fixture Factory

Passport/read-model honesty Conf 82% Cx Small/Medium

Turn passport stamps into a reusable fixture matrix: valid boundary, stale projection, mismatched owner epoch, bad event receipt, malformed continuity, local-only transcript cursor, missing token cursor, and unsupported manifest version.

Basis

direct: existing passport tests already prove some all-stamps and failure paths, and the checkpoint writer already validates event boundary and projection consistency before insert. The verifier marked this sound as a focused acceptance harness.

Rationale

This compounds every later stamp. It lets the team add runtime snapshot fields with confidence that a fake positive `hydration_candidate` will be caught.

Downsides

It is test infrastructure, so it should trail the chosen contract rather than become the whole slice. Best paired with ideas 1, 2, or 5.

7

Postgres Capsule, Evidence References Only

Storage/effect boundary Conf 88% Cx Small

Keep v1 runtime snapshot truth entirely in Postgres. Object storage, including S3, can remain optional evidence storage keyed to event sequences or evidence manifests later, but should not be required for hydration candidacy in this slice.

Basis

direct: `game_checkpoints` already has JSONB snapshot and cursor columns, event/projection hashes, owner epoch, and hydrateable metadata. `game_evidence_manifests` separately has storage-provider and storage-key style fields for bulky evidence references.

Rationale

This protects the next slice from infrastructure expansion. S3 becomes useful when raw evidence volume or retention policy demands it, not when the runtime snapshot still lacks local truth.

Downsides

Postgres JSONB is not where large transcripts or raw artifacts should live forever. The boundary is that runtime resume metadata stays in Postgres; optional raw evidence can move out later.

Cuts

Rejection Summary

Idea Reason Rejected
Effect Receipt Fence Directionally useful but too easy to overrun into generalized effect replay or resume design. Fold only the boundary-result principle into the receipt contract.
Owner-Epoch Capability Object Owner checks are real, but a new capability-object refactor is broader than this runtime snapshot slice needs.
Full Transcript Persistence Now Durable cursor semantics do not require every transcript row to be persisted mid-game.
Resume Factory Now `GameRunner.fromCheckpoint()` and actual resume execution are explicitly later work.
Raw LLM Evidence As Resume Truth Conflicts with the privacy validator and the private-continuity boundary.
Workflow Engine Replacement External analogy only and replaces the subject instead of advancing this slice.
Mid-Phase Recovery Contradicts the selected safe phase-boundary scope and would need pending-effect semantics outside v1.
Per-Event Snapshots Conflicts with the current phase-boundary checkpoint cadence and recent planning direction.
