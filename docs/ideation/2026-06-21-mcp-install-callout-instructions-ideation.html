MCP Install Callout and Instructions

Ranked directions for making the existing Influence MCP installable from the app, with one page that helps users add it to Codex or Claude without losing the OAuth and scope boundaries that already matter.

Grounding Context

Influence's strategy explicitly prioritizes AI app compatibility: make the product usable from ChatGPT, Claude, and other agent/app surfaces through OAuth, MCP, and manifest-compatible access. It serves the broader agent-development loop by reaching users where AI-native tinkerers already work.

The protocol surface is already well shaped. The production docs define two resources: user-facing /mcp with OAuth scope=games, and producer /mcp/producer with scope=mcp plus the current mcp role. The same document already gives Codex TOML snippets and Claude Code HTTP commands, plus operational checks for protected-resource metadata and authorization-server metadata.

The current web app has a compact public nav, an authenticated dashboard with Open Games, Your History, and Your Agents, and an OAuth authorization page that already explains the difference between user game access and global producer access. What is missing is product discovery and a user-facing setup path. Today, the MCP exists as a developer doc and OAuth flow, not as a discoverable product affordance.

External docs reinforce the shape. Codex supports Streamable HTTP MCP servers and OAuth login, stores MCP configuration in config.toml, and uses server-advertised scopes during OAuth when available. Claude Code recommends HTTP for remote MCP servers, supports claude mcp add --transport http, and authenticates remote MCP servers through /mcp plus the browser flow. The MCP spec requires protected-resource metadata, authorization-server metadata, bearer tokens on every request, resource indicators, HTTPS except localhost redirects, and PKCE.

Relevant internal evidence: docs/game-mcp-production-oauth.md:7-10, docs/game-mcp-production-oauth.md:89-124, packages/web/src/components/nav.tsx:37-70, packages/web/src/app/dashboard/dashboard-content.tsx:289-371, packages/web/src/app/oauth/mcp/authorize/authorize-client.tsx:241-247, STRATEGY.md:30-34.

1. Dashboard Bridge Card

Axis
Discovery moment
Confidence
88%
Complexity
Low

Description: Add a compact callout on the authenticated dashboard, near Your History and Your Agents: "Use your Influence games from Codex or Claude." The card links to the single setup page and says what users can do after connecting: list their games, inspect projections, read revealed facts, and review authorized cognitive artifacts.

Basis: direct: STRATEGY.md:30-34 makes AI app compatibility a named track; packages/web/src/app/dashboard/dashboard-content.tsx:334-371 shows the dashboard already clusters game history and agents, the exact objects MCP exposes.

Rationale: This meets users at the moment they have both a reason and the permissions context. It avoids cluttering public navigation while still making MCP visible to the primary user once signed in.

Downsides: Users with no games or no agents may see a capability before it feels useful, so the card should adapt empty-state copy instead of sounding like a generic integration ad.

2. Role-Aware Setup Page

Axis
Single-page instruction contract
Confidence
86%
Complexity
Medium

Description: Create one page, probably /mcp or /connect/mcp, with the user-facing /mcp lane first and a clearly separated producer lane only when the viewer has the mcp role. The default page should not teach producer/private-trace access to ordinary users.

Basis: direct: docs/game-mcp-production-oauth.md:7-10 defines the two resource profiles and says not to reinterpret scope=mcp as user-scoped; packages/web/src/app/oauth/mcp/authorize/authorize-client.tsx:241-247 already has distinct user and producer grant copy.

Rationale: One page keeps the instruction surface easy to find and maintain, but role awareness preserves the literal access boundary. It also gives maintainers one canonical place to update client snippets when Codex or Claude behavior shifts.

Downsides: Role-aware content creates test burden around signed-out, signed-in, and producer-role states. The page needs to avoid revealing producer instructions through static HTML if that is considered sensitive.

3. Client Tabs With Live Snippets

Axis
Client-specific setup clarity
Confidence
84%
Complexity
Medium

Description: Put Codex and Claude Code in tabs or segmented controls. Codex should show a config.toml block for [mcp_servers.influence_game], plus codex mcp login influence_game. Claude should show claude mcp add --transport http influence-game https://.../mcp and the instruction to run /mcp in Claude Code to authenticate.

Basis: direct: docs/game-mcp-production-oauth.md:91-124 already provides both client paths. external: OpenAI documents Streamable HTTP url config and OAuth login for Codex; Anthropic documents HTTP remote server setup with claude mcp add --transport http and browser authentication through /mcp.

Rationale: A single generic "add an MCP" page will be annoying because users do not install Codex and Claude the same way. Client tabs let the page stay single-source while still feeling native to each tool.

Downsides: Codex CLI support for adding HTTP servers via CLI may be less visible than direct TOML config in the docs, so Codex copy should prefer the documented config path unless the exact CLI syntax is verified in the implementation pass.

4. OAuth Discovery Health Strip

Axis
Validation and support loop
Confidence
80%
Complexity
Medium

Description: Add a small status strip to the setup page that checks the current environment's public MCP readiness: protected-resource metadata, authorization-server metadata, and unauthenticated POST /mcp challenge shape. For normal users, render plain statuses like "OAuth discovery ready"; for maintainers, expose exact endpoint results.

Basis: direct: docs/game-mcp-production-oauth.md:137-148 lists these checks as readiness gates. external: the MCP authorization spec requires protected-resource metadata, authorization-server metadata, and WWW-Authenticate discovery on 401 responses.

Rationale: This turns a historically brittle routing problem into a visible product/support signal. If a deployment breaks OAuth discovery, the instructions page becomes the first diagnostic surface instead of a dead-end copy block.

Downsides: A live health strip can create noise if staging or local development intentionally lacks public metadata. It should be environment-aware and degrade to "not checked" rather than presenting false failure as user error.

5. Permission Receipt Language

Axis
Trust and permission framing
Confidence
82%
Complexity
Low

Description: Make the setup page use the same grant language as the OAuth consent screen, and add a "what this can read" receipt before the user starts. For games, say created/joined games and owned player/agent records; for producer mcp, say global developer inspection and private reasoning tools.

Basis: direct: docs/game-mcp-production-oauth.md:66-87 enumerates user and producer tools; packages/web/src/app/oauth/mcp/authorize/authorize-client.tsx:241-247 already contains concise grant descriptions.

Rationale: The setup page is pre-consent; it should prepare the user for what the OAuth screen will ask. Reusing the consent language reduces drift and makes later security reviews easier.

Downsides: If the page gets too careful, it may feel like a warning wall. Keep it receipt-shaped, not legalistic.

6. Post-Install Starter Prompts

Axis
Discovery moment
Confidence
74%
Complexity
Low

Description: After the install snippets, show three copyable starter prompts: list my Influence games, summarize my latest completed game, and read my agent's strategy artifacts for a specific game. Optionally add a per-game replay link that says "Ask Codex or Claude about this game" and points back to setup if the user has not connected yet.

Basis: direct: docs/game-mcp-production-oauth.md:66-74 lists user-facing MCP tools; STRATEGY.md:36-40 says reasoning access turns watching into improvement.

Rationale: Install docs are necessary but not sufficient. Starter prompts help the user experience the agent-development loop immediately after configuration.

Downsides: This can drift into product support for client prompting. Keep examples few and tied to actual tool capabilities.

7. Troubleshooting Appendix

Axis
Validation and support loop
Confidence
76%
Complexity
Low

Description: Add a short appendix with the real failure modes: OAuth page does not open, "No authorization support detected", wrong scope/resource, token expired, and producer role required. Keep it task-oriented and link each symptom to the specific setup or environment check.

Basis: direct: docs/game-mcp-production-oauth.md:141-148 lists failure checks; memory from prior Influence MCP work says staging/prod failures often came from root-level well-known and /mcp routes reaching the wrong service.

Rationale: This buys support leverage without expanding the main page. It also preserves hard-won operational knowledge where a user or maintainer will look when connecting a client.

Downsides: Troubleshooting copy can stale quickly. The appendix should link to live health checks and current docs rather than becoming a giant runbook.

Rejection Summary

# Idea Reason Rejected
1 Top-nav MCP link for everyone Too prominent for an authenticated integration; likely clutters the public nav before users have game history or agents.
2 Separate Codex and Claude pages Duplicates protocol truth and makes drift more likely; client tabs preserve specificity inside one canonical page.
3 README-only setup instructions Already mostly exists in docs, but the request is for UI discovery and a page users can reach from the app.
4 Homepage MCP hero Scope overrun; MCP is a powerful companion surface, not the first public explanation of the game.
5 ChatGPT app submission instructions now Current docs explicitly keep public ChatGPT app submission out of scope; Codex and Claude are the requested targets.
6 Admin-only MCP setup page Confuses producer access with user-facing games access and hides the main product loop from ordinary agent owners.
7 OAuth audit-log viewer Interesting maintainer tool, but too far beyond a setup callout and instructions page.
8 QR/deep links into clients Unverified client support and too much ceremony for a copy-paste developer workflow.
9 Profile settings callout only Too buried for a capability that connects directly to games and agents; dashboard placement is stronger.
10 Local-dev snippets as a primary lane Useful for maintainers, but the requested page should teach the deployed user path first.
11 Consent page cross-link as the main move Helpful supporting detail, but it only catches users after they are already in the OAuth flow.
12 Post-auth return page Overbuilt for the current OAuth client flow; the client already owns the callback experience.
13 Standalone tool capability matrix Better folded into permission receipt copy; a large matrix would make setup feel more complex than it is.
14 Minimal footer link only Too easy to miss and does not create the product-level callout the request is aiming at.
15 Interactive CLI command wizard Nice polish, but static client tabs with copy buttons solve the primary problem with less surface area.
16 Copy-all config package Duplicates the client tab idea and risks users pasting producer config when they only need user access.
17 Producer-only warning banner Valid as page copy, but too narrow to stand as its own product direction.
18 Game history row-only callout Too narrow; a dashboard card plus optional replay starter prompt covers more states.
19 Rules page instructions Wrong information scent; MCP setup is an app/tooling action, not game-rule education.
20 Public marketing section for MCP Likely confusing before users understand games or agents; keep public marketing focused on the game.
21 Separate maintainer runbook page Too much documentation sprawl; maintainer notes should live as an appendix or existing docs link.
