Ideation: Live Game Cost Tracking

Precise cost tracking for live Influence game runs should become an operational accounting spine, not a refreshed estimate in the create-game form. The current code already sees provider usage and router billing at the call boundary; the missing work is durable capture, normalized cost confidence, and administrator visibility.

Grounding Context

The older cost path is estimate-first. `docs/cost-analysis.md` is explicitly a historical March 2026 snapshot, while the web helper still computes informational create-time estimates from fixed tier costs and player-count scaling. The admin create form then displays that estimate beside a model list where Katana has no estimate at all.

The more important fact: model-call usage is already observable. `InfluenceAgent` and `LLMHouseInterviewer` both extract OpenAI-shaped usage fields and `usage.imgnai` router billing. API-backed private traces also preserve usage and router billing metadata at the producer/admin boundary.

The durable-run layer already treats token state as part of recovery readiness. `TokenTracker` can serialize a checkpoint cursor, checkpoints persist `token_cost_cursor`, and supported recovery requires it. But terminal `game_results.token_usage` is written only when `persistCompletedGame` runs, so active, suspended, and failed runs do not have a reliable admin cost source.

Recommendation

Start with the accounting spine: per-call ledger, live rollup, and provider confidence normalization. Build the admin console on top of those rows. Pretty charts over stale estimates are still stale estimates, just wearing nicer shoes.

Idea 1. Model-Call Cost Ledger

Introduce a compact operational ledger, likely `game_model_call_costs` or `game_provider_spend_entries`, with one row per provider call attempt. Insert it from a new cost sink at the same boundary that currently feeds `TokenTracker` and private traces.

Axis: Provider Call Capture Confidence 92% Complexity High
Description
The ledger should capture `gameId`, `ownerEpoch`, actor, action, phase, round, provider profile, catalog id, model id, reasoning policy, API surface, call status, attempt number, retry relationship, provider response id, trace manifest id when available, latency, token counters, provider-native units, actual/estimated micro-USD, currency, source confidence, diagnostics, and safe router billing facets. `TokenTracker` can remain a compatibility aggregate, but it should stop being the only durable cost structure.
Basis
direct: `agent.ts:1329-1361` and `house-interviewer.ts:633-663` already extract tokens and `usage.imgnai`; `private-trace-writer.ts:41-76` already models safe usage metadata.
Rationale
This is the source-of-truth move. Once every call becomes a ledger entry, retries, failures, model experiments, and suspended runs stop disappearing into aggregate totals or raw debug blobs.
Downsides
Requires schema, idempotency, and careful placement so gameplay does not fail because accounting writes are temporarily degraded. Needs a crisp privacy boundary: the ledger stores safe facts and links to traces, not raw prompts or responses.
2

Idea 2. Durable Game Cost Rollup

Create a live rollup keyed by `gameId` plus `ownerEpoch` that tracks totals across active, suspended, cancelled, and completed states. Completion can snapshot the rollup into `game_results.tokenUsage`, but the rollup is the thing administrators trust during the run.

Axis: Durable Persistence Confidence 90% Complexity High
Description
The rollup should maintain total calls, failed calls, prompt/cached/completion/reasoning tokens, provider-native charges, actual cost, estimated cost, unavailable cost count, and breakdowns by provider, model, actor, action, phase, and round. It can be updated synchronously with ledger inserts or rebuilt from ledger entries when needed.
Basis
direct: `game-lifecycle.ts:366-450` writes token usage only on completion, while `game-lifecycle.ts:989-1085` can suspend/fail runs without a result row. `game-runner.ts:497-522` and `game-checkpoints.ts:249-264` already persist token cursor state at checkpoints.
Rationale
This fixes the operational blind spot. A game that spends money and then suspends is exactly the case an administrator needs to inspect, not the case that should lose its accounting trail.
Downsides
Rollups must handle duplicate owner epochs and resume paths without double-counting. The cleanest implementation treats ledger entries as immutable and rollups as rebuildable summaries.

Idea 3. Provider Cost Normalization And Confidence Ladder

Define a `ProviderUsageEnvelope` and cost confidence model that separates provider actuals, router actuals, org-reconciled actuals, catalog estimates, static estimates, and unavailable costs. Unknown cost is a state, not zero with nicer formatting.

Axis: Pricing Confidence Confidence 88% Complexity Medium
Description
Normalize OpenAI Responses usage, Chat Completions usage, Katana `usage.imgnai`, and local/OpenAI-compatible missing usage into one contract. Store integer micro-USD for normalized costs, provider-native units for credits or other billable units, original currency, pricing version, and `costSource` such as `provider_actual`, `router_actual`, `org_reconciled`, `catalog_estimate`, `static_estimate`, or `unavailable`.
Basis
direct: `token-tracker.ts:46-90` uses a static OpenAI table; prior Katana work says unknown/router models must not fall back to fake OpenAI pricing. external: OpenAI exposes usage fields and organization cost endpoints; Katana responses in repo notes include `usage.imgnai.credits_charged`.
Rationale
The current problem is not just missing math; it is missing provenance. Administrators need to know whether a number is a bill, a reconciliation, a catalog-derived estimate, or a shrug in a trench coat.
Downsides
Katana cost fields need a fresh real sample for Grok game calls. OpenAI organization costs may not map perfectly to a game without project/API-key tagging discipline.

Idea 4. Admin Cost Console

Add administrator cost summaries and drilldowns to the existing admin surface: game list totals, game detail live meter, cost by phase/action/player/House/model, top expensive calls, retry/failure spend, and exportable rows.

Axis: Admin Visibility Confidence 86% Complexity Medium
Description
Expose `GET /api/admin/games/:idOrSlug/costs` for per-game detail and include compact cost fields in `GET /api/admin/games`. The UI should show total actual cost, estimated cost, unavailable cost count, token counts, provider-native charges, confidence badges, and hotspots. It should link to durable-run inspection and authorized private trace evidence rather than embedding raw trace content.
Basis
direct: `routes/admin.ts:361-405` already returns admin game summaries; `routes/admin.ts:353-359` exposes durable-run inspection; `routes/admin.ts:284-310` exposes avatar generation provider/cost attempts.
Rationale
Accounting that only exists in tables is still hard to use. The admin needs fast answers: what did this run cost, is the number real, what got expensive, and did a failed retry path burn money?
Downsides
The dashboard must avoid turning into a trace browser. Keep default views compact and make raw evidence a deliberate producer/admin drilldown.

Idea 5. Trace Backfill And Provider Reconciliation

Backfill historical cost data from existing private trace manifest metadata and terminal game results, then reconcile current ledger totals against provider-level reports where possible.

Axis: Pricing Confidence Confidence 78% Complexity Medium
Description
Write a one-off or admin-only job that reads `game_evidence_manifests.metadata.usage` and `game_results.tokenUsage`, creates ledger entries marked `backfilled_trace` or `backfilled_result`, and records missing confidence where raw call-level data is unavailable. Separately, use OpenAI organization usage/cost endpoints for periodic reconciliation and keep Katana reconciliation provider-specific until its account/billing API is confirmed.
Basis
direct: private trace manifests already store safe usage metadata with `accessScope: producer_admin`; `game_results.token_usage` has old terminal snapshots. external: OpenAI exposes organization usage and cost endpoints with grouped amount/currency data.
Rationale
This gives immediate historical baselines and catches accounting drift. It also tells you when the internal ledger and provider bills disagree, which is where billing bugs get expensive quietly.
Downsides
Backfill can never be as precise as native per-call capture. Older runs may lack private traces, non-owner paths may be partial, and provider cost grouping may not identify a game unless request tagging or project separation exists.

Idea 6. Measurement-First Budgets And Alerts

After the ledger is reliable, add budgets as warnings and governance metadata before hard stopping games. Label high-spend experiments, alert on anomalies, and block only starts that cannot be measured safely.

Axis: Tokenmax Loop Confidence 80% Complexity Medium
Description
Support per-game, per-day, per-provider, and per-model alert thresholds. Add an explicit `tokenmax` or experiment label for intentional high-spend runs. Start with admin warnings, status badges, and logs. Later, add hard caps as an explicit product choice, not as the first control mechanism.
Basis
reasoned: user goal is optimization after knowing current run cost. Budget controls before reliable measurement risk hiding spend, breaking active games, or nudging the system back toward fake estimates.
Rationale
This keeps the product honest during the tokenmax phase: spend can be intentional, visible, and bounded without prematurely sacrificing game quality.
Downsides
Bad thresholds can create alert fatigue. Hard stops can damage a live match, so cap semantics need product judgment after actual cost patterns are visible.
7

Idea 7. Tokenmax ROI Report

Once costs are trustworthy, report spend against game quality and watchability signals: completed run quality, strategic depth artifacts, postgame richness, retries, latency, and model/reasoning policy differences.

Axis: Tokenmax Loop Confidence 72% Complexity High
Description
Add a report that compares cost by model, reasoning policy, rule track, and prompt version against outcome proxies: game completion, transcript richness, postgame summary usefulness, strategic artifact density, and human review notes when present. This becomes the optimization map for deciding where extra tokens actually buy better content.
Basis
direct/reasoned: repo strategy and observability docs prize watchable strategic play, and the user explicitly framed this as prerequisite measurement before optimization.
Rationale
Raw spend answers "what did it cost?" ROI reporting answers "what was worth it?" That is the difference between tokenmaxxing as taste and tokenmaxxing as an engineering loop.
Downsides
Quality signals are noisier than cost signals. This should follow the accounting spine, not block it.

Recommended First Slice

Build the accounting spine before the console: a per-call ledger, provider usage normalizer, and live rollup with one admin read endpoint. Keep `TokenTracker` and terminal `game_results.tokenUsage` as compatibility/export surfaces, but make the ledger and rollup the operational source of truth.

Minimum useful slice: add provider usage fixtures, normalize OpenAI Responses, OpenAI Chat Completions, and Katana `usage.imgnai`; insert cost ledger rows from agent and House calls; maintain a game cost rollup; expose `GET /api/admin/games/:idOrSlug/costs`; add a compact cost summary to `/api/admin/games`; verify suspended/failed runs retain cost rows.
