MCP Agent Management and Queue Enrollment

A ranked set of product and backend directions for expanding the Influence production MCP from read-only game inspection into safe agent management: rules discovery, roster inspection, agent creation and updates, and pre-match queue enrollment. The line stays bright: no active-match participation through MCP.

Grounding Context

The current production MCP is a manually-routed JSON-RPC server with two profiles: user-facing /mcp on scope=games, and producer /mcp/producer on scope=mcp. User tools are presently read-only and game-inspection-oriented; producer trace tools are kept behind the producer profile. That split is the right skeleton for the new surface, but the user profile now needs carefully bounded mutations.

Reusable seams already exist. Bearer validation checks OAuth audience, purpose, scope, resource, client, subject, and expiry before building a GameMcpAuthContext. User claims already resolve created games, joined games, player IDs, and owned agent profile IDs. REST routes already implement agent profile create/list/get/update, free queue status/join/leave, and waiting-game join flows with ownership checks.

The biggest mismatch is semantic, not mechanical. The desired conversation asks "which agent has the highest ELO?", while current free-game rating updates write account-level users.rating and only increment agent profile games/wins. The old free_track_ratings table still exists, but account-level ELO is the live path. The MCP should not promise agent-level ELO until the backend either derives it honestly or restores an agent-level rating source.

Boundary

MCP consumers may help a user understand rules, inspect and tune their own agents, and enroll an agent before a match starts. They must not vote, empower, expose, send lobby or mingle messages, act as council, perform ready checks, manipulate timers, moderate, or take any action inside an active match.

Topic Axes

  • Rules discovery and strategy education. MCP consumers need enough rules and strategy context to answer "how does Influence work?" without touching active matches.
  • Roster and performance read model. Agent lists, search, current queue status, games, wins, and rating semantics must be returned in one rich response.
  • Agent mutation services. Create and update should be coarse-grained tools over server-side validation, not field-by-field orchestration.
  • Queue and open-game enrollment. Daily-free comes first, but the API shape should support open games, ranked queues, tournaments, parties, and invite codes later.
  • MCP guardrails, auth, and ergonomics. Tool descriptors, annotations, error wording, and tests must keep the surface management-only and LLM-legible.

Ranked Ideas

I1.

Management-only MCP tool contract

Confidence 92% Complexity Medium

Define the user-facing MCP as an explicit agent-management resource instead of a game-control resource. The first-class tool set should be get_rules, search_rules, list_agents, get_agent, search_agents, create_agent, update_agent, get_queue_status, join_queue, leave_queue, list_open_games, and join_open_game. Every description should say when to call it, when not to call it, required permissions, and side effects.

Axis
MCP guardrails, auth, and ergonomics
Basis
direct: The current server advertises user MCP as read-only and scope=games-scoped at packages/api/src/game-mcp/server.ts:95, lists only read tools at packages/api/src/game-mcp/server.ts:144, and rejects mutation-shaped tools at packages/api/src/game-mcp/server.ts:198.
Rationale
This is the highest-leverage choice because it makes the safety boundary legible to every provider before individual services are implemented.
Downsides
Requires touching descriptors, docs, tests, and server routing at once; a thin implementation that skips descriptions would invite provider misuse.
I2.

Roster read model with honest rating semantics

Confidence 88% Complexity Medium

Build one agent-summary read model that can power list_agents, get_agent, and search_agents. Each summary should include identity, archetype, biography, prompt excerpt or owner-visible prompt, avatar, cosmetics, games played, wins, current queue membership, active-game exclusion state, and rating fields with explicit provenance. If rating remains account-level, return accountElo and omit or null agentElo rather than inventing "highest ELO agent."

Axis
Roster and performance read model
Basis
direct: Agent profiles store games_played and games_won at packages/api/src/db/schema.ts:83, lifecycle completion increments those profile stats at packages/api/src/services/game-lifecycle.ts:466, but the live free-game ELO update writes users.rating at packages/api/src/services/game-lifecycle.ts:537.
Rationale
The desired UX depends on ChatGPT being able to compare agents without a follow-up call. Honest rating provenance prevents the tool from laundering stale schema into product truth.
Downsides
Requires a product decision: expose account ELO in agent cards now, derive agent-relative performance, or wire a real per-agent rating path before claiming agent ELO.
I3.

Idempotent queue enrollment service

Confidence 90% Complexity Medium

Introduce a QueueManagementService behind MCP and REST-shaped adapters. The public MCP tools should be get_queue_status({ queueType }), join_queue({ queueType, agentId }), and leave_queue({ queueType, agentId? }). Support daily-free first, reject unsupported queue types clearly, and make repeat joins/leaves friendly: same-agent join returns the existing enrollment; leave with no entry returns a successful "not queued" response.

Axis
Queue and open-game enrollment
Basis
direct: free_game_queue already enforces one row per user at packages/api/src/db/schema.ts:757, and the current REST join/leave endpoints are non-idempotent with 409 and 404 responses at packages/api/src/routes/free-queue.ts:158 and packages/api/src/routes/free-queue.ts:189.
Rationale
LLMs retry. Queue tools should behave as safely repeatable commands, especially when provider clients may replay calls after network or UX interruptions.
Downsides
The MCP wrapper can soften errors without changing REST, but a shared service is cleaner and avoids drift between app and MCP behavior.
I4.

Shared agent profile command service

Confidence 84% Complexity Medium

Extract agent create/update validation and persistence into an AgentProfileCommandService that both REST routes and MCP tools call. MCP names should be human-facing: displayName, archetype, personalityPrompt, publicBiography, avatarUrl, and optional cosmetics. Internally map those to existing name, personaKey, personality, backstory, avatarUrl, and future cosmetic storage.

Axis
Agent mutation services
Basis
direct: The REST route already validates archetype keys at packages/api/src/routes/agent-profiles.ts:29, creates owned profiles at packages/api/src/routes/agent-profiles.ts:172, and updates only profiles matching both ID and current user at packages/api/src/routes/agent-profiles.ts:269.
Rationale
This keeps validation server-side and prevents the MCP implementation from becoming a parallel route with subtly different ownership, avatar, or archetype behavior.
Downsides
The current update route contains a stats reset path for personality changes; the MCP docs and responses should not advertise that because the product direction is to remove it.
I5.

Rules catalog and search source of truth

Confidence 78% Complexity Low

Promote rules content into an importable rules catalog that can feed the website and MCP responses. get_rules should return a compact full-game explanation plus strategy notes; search_rules should return matching sections with enough surrounding context to answer "how does this work?" without pulling in active-match state.

Axis
Rules discovery and strategy education
Basis
direct: docs/rules-page-content.md:1 has the core rules, phases, and win condition, but docs/rules-page-content.md:124 says "ten archetypes" while the server currently accepts thirteen keys at packages/api/src/routes/agent-profiles.ts:29.
Rationale
Rules discovery is the safest high-value MCP surface: it teaches gameplay without authorizing gameplay. Making the content shared also fixes stale docs before external clients amplify them.
Downsides
Rules content touches product copy; it needs a small editorial pass, especially around ELO wording and archetype count, before becoming API-grade.
I6.

Open-game enrollment as a waiting-only queue adapter

Confidence 72% Complexity Medium

Treat open games as another enrollment target, but only when the backend exposes a waiting game with available slots. list_open_games should filter to joinable public or user-accessible waiting games and return player count, slots remaining, ruleset, queue type, and estimated start when known. join_open_game({ gameIdOrSlug, agentId }) should reject active games and reuse server-side profile ownership checks.

Axis
Queue and open-game enrollment
Basis
direct: The REST game list already returns status, player count, track type, and timing fields at packages/api/src/routes/games.ts:222, and the join route rejects non-waiting games at packages/api/src/routes/games.ts:366, full games at packages/api/src/routes/games.ts:375, and profiles not owned by the caller at packages/api/src/routes/games.ts:407.
Rationale
Open-game join is useful if it remains pre-match enrollment. Framing it as a queue adapter keeps the future path open for ranked, tournament, invite, and party enrollment without teaching MCP clients match-control verbs.
Downsides
Open-game privacy and slug lookup need a careful pass; the current join path resolves by ID, while user-facing conversation may naturally mention slugs.
I7.

Descriptor and forbidden-action tests

Confidence 86% Complexity Low

Refactor the tool descriptor helper so read tools, idempotent mutations, and destructive mutations can declare accurate annotations and side-effect language. Add tests that enumerate user-facing MCP tools and fail if forbidden active-match verbs appear, if a mutation is marked read-only, if a tool lacks an OAuth security scheme, or if user tools are accidentally exposed under the producer-only assumptions.

Axis
MCP guardrails, auth, and ergonomics
Basis
direct: The current tool() helper hardcodes annotations.readOnlyHint: true for every descriptor at packages/api/src/game-mcp/server.ts:368. external: the MCP tools spec exposes tool descriptors through tools/list and annotations through the schema, so descriptors are part of the client contract.
Rationale
Mutation safety is not only runtime authorization; provider clients inspect descriptors before choosing tools. The descriptors should tell the truth.
Downsides
Tests must avoid becoming brittle snapshots of every word. Assert required semantics and forbidden surfaces; leave room for description copy to improve.

Rejection Summary

# Idea Reason Rejected
1 Expose vote, empower, expose, council, mingle, lobby, diary-room, timer, ready-check, or moderator MCP tools Subject boundary violation. These are active-match actions and must remain inside the Influence client.
2 Mirror existing REST routes one-for-one as MCP tools Too granular and too leaky. The LLM should call coarse commands, not orchestrate REST-shaped workflow fragments and status codes.
3 Use scope=mcp for agent management because mutations feel privileged Breaks the documented profile split. User management belongs on /mcp with scope=games; producer scope is for developer/global evidence.
4 Expose another player's draft agents, prompts, queue state, or matchmaking internals for recommendation quality Security and trust failure. The authenticated user can inspect and mutate only their own agents and safe public/open-game metadata.
5 Name the queue tools join_daily_queue and leave_daily_queue Too narrow. queueType keeps daily-free, open-game, ranked, tournament, invite, and party queues from requiring a future redesign.
6 Report per-agent ELO from free_track_ratings immediately Basis mismatch. The table exists, but current free-game ELO updates write account-level user ratings; agent-level ELO needs a product/backend decision first.
7 Build a full MCP App UI before adding tools Interesting but lower leverage. The desired experience is conversational tool use; rich app UI can follow once the tool contract is stable.
8 Accept userId or ownership hints from MCP arguments Security footgun. Ownership must be derived from the bearer token and database, never from client-provided identity fields.
9 Allow join_open_game to target active, running, or completed games Scope overrun. Joining is pre-match enrollment only; active matches are outside MCP.
10 Highlight the current update-time ELO reset in MCP responses Product drift. The reset is planned for removal, so MCP should not train providers or users around it.
