Grounding Context

The current producer trace stack is intentionally broad and privileged. PrivateDecisionTrace content carries prompts, raw provider responses, tool arguments, emitted thinking, native reasoningContext, decisionLog, strategy packet revision data, actor/action metadata, and source pointers. The API writer stores that raw JSON under private_decision_trace and creates a manifest with accessScope: "producer_admin" only.

The deployed Game MCP is already split into two profiles. /mcp with scope=games is subject-scoped to created or joined games and owned player/agent records; /mcp/producer with scope=mcp stays producer-only and exposes private trace tools. Existing docs explicitly say not to expose private trace content or metadata through scope=games.

Engine capture points already exist. Agent calls surface thinking, reasoningContext, decisionLog, strategicLens, strategicLensRationale, and strategy packet updates in typed returns and in emitPrivateDecisionTrace. API lifecycle wiring already has a single adapter seam, createPrivateTraceSink, where owner-backed runs enrich traces with gameId and ownerEpoch before storage.

External MCP grounding: the MCP resources spec treats resources as URI-addressed application data and calls for per-resource access controls; the MCP authorization spec requires HTTP servers to validate resource-bound bearer tokens before serving protected content.

Recommended Next-Slice Contract

Create a new game_cognitive_artifacts read model for new games only, populated at write time from the existing private trace sink, and expose it through scope=games using subject claims. Do not read, sanitize, or fallback to producer trace content when the split artifacts are absent.

Keep producer_trace exactly where it is: full raw evidence behind producer/admin access. Add first-class user artifacts as separate records with their own schema, ownership, and visibility. The writer may derive those records from the same in-memory PrivateDecisionTrace object at capture time, but once written they are independent product data, not a filtered view over private traces.

Producer/admin access should also use the split artifact records directly. Raw producer traces remain available for forensic inspection, but they are too large and too noisy to be the normal producer read path for reasoning, thinking, or strategy.

Artifact Payload contract User-facing access Fallback behavior
producer_trace Full raw prompt, response, tool args, internal metadata, reasoning, thinking, strategy, source pointers. None through scope=games; producer/admin through existing producer MCP. Existing behavior only.
reasoning Raw/native reasoningContext if captured, plus minimal action/actor/game boundary metadata. Owning user/agent only. If absent, return not_captured; never read producer trace.
thinking Explicit model-emitted thinking, plus minimal action/actor/game boundary metadata. Owning user plus other participants in the same game. If absent, return not_captured; never read producer trace.
strategy decisionLog, strategicLens, strategicLensRationale, strategy packet/reflection summaries when present. Owning user plus other participants in the same game. If absent, return not_captured; never read producer trace.

Ranked Ideas

  • 1. game_cognitive_artifacts as the new product boundary
  • 2. Write-through artifact extraction at createPrivateTraceSink
  • 3. Claims plus producer/admin artifact read model
  • 4. MCP raw artifact handles under scope=games
  • 5. Web/API derived artifact endpoint without UX polish
  • 6. New-games-only capability flag and no-capture response

1game_cognitive_artifacts as the new product boundary

Conf 92% · Cx M

Add a new table/read model for user-authorized cognitive artifacts. It stores only the fields users may eventually see: reasoning, thinking, and strategy records with owner/participant access metadata. It does not store raw prompts, provider responses, tool argument blobs, storage keys, or producer source pointers in returned payloads.

Axis
Artifact and storage boundary
Basis
direct: game_evidence_manifests currently constrains accessScope to producer_admin, and the private trace writer records thinking/reasoning byte counts plus decisionLog and strategy packet revision metadata before writing a producer-only manifest.
Rationale
This cleanly honors the user's central constraint: user access must not depend on reading producer traces.
Downsides
Requires a migration, service, and tests. Some duplication with trace metadata is intentional.

2Write-through artifact extraction at createPrivateTraceSink

Conf 88% · Cx M

Derive the split artifacts from the in-memory PrivateDecisionTrace in the API lifecycle adapter, beside the existing producer trace write. The order should be explicit: capture allowed product artifacts from the trace object, write producer trace as before, and treat artifact-write failures as nonfatal diagnostics that do not affect gameplay.

Axis
Write-time capture points
Basis
direct: game-lifecycle.ts has a single createPrivateTraceSink adapter that enriches traces with gameId and ownerEpoch; InfluenceAgent.emitPrivateDecisionTrace has the raw fields while they are still separated.
Rationale
This is lower risk than teaching every phase runner a second persistence path, and it avoids post-hoc trace parsing.
Downsides
The extractor must be careful not to copy prompt, response.raw, toolArguments, or raw source pointers into user payloads.

3Claims plus producer/admin artifact read model

Conf 86% · Cx M

Give the cognitive-artifact read model two explicit modes: producer/admin can read all split artifacts directly, while scope=games callers use resolveGamesMcpClaims and player/profile participation. For user-facing reads, reasoning requires actor ownership; thinking and strategy require same-game participation. Created-only games can remain visible in the game list while user-facing artifact reads require an actual player/profile claim.

Axis
Access policy and claims
Basis
direct: the existing claims resolver already returns gameIds, createdGameIds, joinedGameIds, playerIds, and agentProfileIds; the MCP read model already blocks inaccessible games for scope=games.
Rationale
This avoids inventing another user auth model, makes the owner-only reasoning rule precise, and gives producers a compact direct surface instead of treating raw traces as the only inspection path.
Downsides
The product must decide whether a game creator who did not play should count as a participant. The conservative default is no.

4MCP raw artifact handles under scope=games

Conf 80% · Cx M

Add user-facing MCP tools such as list_cognitive_artifacts and read_cognitive_artifact on /mcp, and expose the same split-artifact reads on /mcp/producer with producer-wide access. The list tool should return stable custom URIs like influence-game://deployed/games/{gameId}/cognitive-artifacts/{artifactId}; resources/read can then return the authorized artifact for that URI. Producer trace tools stay separate and absent from scope=games.

Axis
MCP and web read-model design
Basis
direct: the current MCP server already uses resources/read for influence-game://deployed/games and tools for parameterized reads; MCP resources are URI-addressed application data with per-resource access controls.
Rationale
This gives clients raw authorized artifacts, gives producers a smaller inspection surface than raw traces, and avoids building a polished web UX first.
Downsides
Resource templates can wait; clients can discover URIs through the list tool first.

5Web/API derived artifact endpoint without UX polish

Conf 78% · Cx M

Add REST read-model endpoints for web consumption, but return index/snippet/derived fields rather than raw producer-style content. A minimal shape is GET /api/games/:id/cognitive-artifacts?type=thinking|strategy|reasoning&actor=... plus GET /api/games/:id/cognitive-artifacts/:artifactId, with the same authorization service as MCP.

Axis
MCP and web read-model design
Basis
direct: /api/games/:id/transcript currently exposes replay entries without thinking; the docs say web/admin affordances and derived thinking resources were out of scope until a separate data-shaping design exists.
Rationale
The web path can unblock backend integration and future summaries without committing to final UI.
Downsides
Raw reasoning over web should stay out until product copy and user expectations are clearer.

6New-games-only capability flag and no-capture response

Conf 90% · Cx S

Mark newly created games with cognitiveArtifactsVersion: 1 in games.config and gate the writer on that version. For games without the flag, the read model returns a clear not_captured_for_game response. It must not attempt a best-effort producer trace read.

Axis
New-games-only rollout and migration
Basis
direct: routes/games.ts already builds the per-game config at creation, and lifecycle code has access to that config before constructing the runner and trace sink.
Rationale
This makes the rollout testable and prevents old-game ambiguity from infecting the new access contract.
Downsides
Waiting-room games created before deployment will not capture unless explicitly recreated or manually marked.

Code Boundaries to Change First

  1. Add the schema/migration and types for game_cognitive_artifacts; keep game_evidence_manifests producer-only.
  2. Add writeCognitiveArtifactsFromTrace(...) beside private-trace-writer.ts, with leak tests using prompts, raw responses, tool args, and storage keys as sentinel strings.
  3. Thread a cognitiveArtifactsEnabled flag from routes/games.ts config into createPrivateTraceSink in game-lifecycle.ts.
  4. Add an artifact-access.ts or cognitive-artifacts-read-model.ts service with explicit producer/admin and games-subject modes; games-subject mode uses resolveGamesMcpClaims and enforces owner vs participant access.
  5. Add MCP list/read methods on ProductionGameMcpReadModel and wire them through server.ts for both scope=games and producer scope=mcp, with different authorization modes.
  6. Add minimal REST endpoints in routes/games.ts or a sibling route module for web read-model access.
  7. Update docs/game-mcp-production-oauth.md, docs/reasoning-transcript-observability.md, and CONCEPTS.md with the new artifact terms.

Intentionally Deferred

  • Final web UI polish, cards, summaries, and timeline presentation.
  • LLM summarization of authorized artifacts. Start with deterministic snippets and structured payloads.
  • Object storage for user artifacts unless JSONB size becomes a real problem.
  • Resource template discovery in MCP. Return concrete resource URIs from list tools first.
  • Changing scope=mcp semantics. Producer MCP remains producer/global.
  • Trying to grant users access to producer trace manifests or trace metadata.
  • Old-game reconstruction from producer traces.

Migration Recommendation for Old Games

Do not migrate old games in the next slice. The available old surfaces are incomplete and ambiguous: transcript rows may contain thinking but not reasoning; strategy artifacts may exist only inside producer traces, turn JSONL, or live-agent memory; and ownership may not map cleanly for generated agents. Backfilling from producer traces would violate the strongest boundary in the prompt.

The only worthwhile old-game lift is a small availability marker: old games return artifactCapture: "not_enabled" or not_captured_for_game so clients can explain absence without guessing.

Test Strategy

  • Writer unit tests: fixture trace with sentinel secrets proves product artifacts contain allowed fields only.
  • Access tests: owner can read own reasoning; same-game participant can read thinking/strategy but not reasoning; nonparticipant and created-only caller are denied.
  • Producer/admin tests: producer can list/read reasoning, thinking, and strategy artifacts directly, including when raw trace storage is unavailable or intentionally not consulted.
  • MCP tests: scope=games lists/reads authorized artifacts and still cannot discover or call trace tools; producer scope=mcp can list/read split artifacts without using read_trace_content.
  • Producer regression tests: existing list_trace_manifests, read_trace_content, and search_reasoning_traces behavior remains unchanged.
  • New-games-only tests: flagged game captures artifacts; unflagged old game returns not_captured and never calls private trace storage.
  • Route tests: web/API endpoints share the same read model and return the same denials.
  • Documentation checks: docs name the no-fallback rule and the owner/participant access matrix.

Rejection Summary

# Idea Reason Rejected
1 Sanitize producer traces on demand for users Directly violates the no-fallback constraint and keeps raw prompt/tool/storage data in the read path.
2 Add owner_only and participants to game_evidence_manifests.accessScope Too easy to couple user reads to raw evidence storage and private trace metadata.
3 Expose transcript thinking as the first web feature Incomplete and misleading: no reasoning, weak strategy coverage, and old-game semantics would look better than they are.
4 Put cognitive artifacts into canonical events Pollutes replayable board truth with private cognitive state.
5 Put cognitive artifacts into public transcript rows Conflates player-visible replay with hidden/participant-authorized data.
6 Make scope=mcp user-scoped for this feature Contradicts current OAuth docs and recent boundary decisions; use scope=games for user-facing access.
7 Build final web UX before storage/auth Scope overrun. The next slice is the clean boundary and read model, not UI polish.
8 Backfill old games from local simulation JSONL Not available for deployed games, not normalized to user ownership, and outside the required new-games-only contract.
9 Give all game creators participant access to every agent artifact Product-risky default. Created game access and participated-in-game access should remain distinguishable.
10 Store user artifacts only in object storage Premature complexity for small structured records; keep object storage as a later size-driven variant.
