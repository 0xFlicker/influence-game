Grounding Context

Codebase Context

The current route already makes DramaticReplayViewer the main rendering path for completed games with transcript data and for live in-progress games, with ?mode=classic as the remaining escape hatch. The useful prototype should turn that implicit default into the product default: MatchWatchShell owns the watch route, and old admin/classic framing stops shaping the work.

The current dramatic viewer is not throwaway. It already owns playhead state, live catch-up, speed, thinking visibility, scene grouping, and phase-aware center renderers. The right move is to wrap and split it: keep the existing theaters alive in the center while adding stable header, cast rail, timeline, and inspector surfaces around them.

The data gap is equally clear. Today some board state comes from transcript parsing, some from visible-message reconstruction, and some from canonical events/projection services. A first-class watch shell needs one typed watch-facts adapter so receipts, relationship edges, cast state, phase labels, and final stats derive from backed storage without each panel inventing its own truth.

User decisions applied here: build toward audience-omniscient, let viewers see thought/strategy summaries, show explicit relationship edges plus selected strategy notes, derive receipts from backend storage, start final results with stats, and replace live admin immediately.

Idea 1. MatchWatchShell as the new route owner

Axis: Surface ownership and routing Confidence: 90% Complexity: Medium
Description
Create MatchWatchShell as the default route target for completed games with transcript data and in-progress games. Treat DramaticReplayViewer as the existing theater/controller seam to split, not as a disposable prior version. The shell owns the stable header, cast rail, right inspector, and bottom timeline while the current center renderers continue to carry phase content.
Basis
direct: game-viewer.tsx:818 already routes completed replay and live in-progress games into DramaticReplayViewer unless mode !== "classic"; the user explicitly said this replaces the live admin view immediately.
Rationale
This removes the biggest product ambiguity. The prototype is useful because it becomes the surface the user actually watches, not a sidecar experiment that later has to win its way back into routing.
Downsides
The first implementation will expose old component names such as Whisper internally unless the split is careful. It also forces live-view rough edges to surface sooner, which is good product pressure but not free.

Idea 2. Watch facts spine before new chrome

Axis: Durable watch facts Confidence: 86% Complexity: Medium
Description
Add a typed local adapter such as buildWatchFacts({ game, players, messages, scene, playhead }) that returns cast status, current pressure lane, timeline events, visible receipts, selected agent summary, and finale stats. It can start from existing transcript parsers and progressively prefer canonical projection data as API support catches up.
Basis
direct: DramaticReplayViewer reconstructs eliminated players from visible system message text at dramatic-replay-viewer.tsx:246, while message-parsing.ts:3 contains regex parsers for vote, power, council, jury, and winner facts. Canonical events and projections already model the durable target shape.
Rationale
The shell has four persistent surfaces that all need the same truth. A watch-facts spine prevents the cast rail, inspector, receipt matrix, and timeline from drifting into four slightly different interpretations of the same game.
Downsides
It is tempting to over-design this into a new API contract immediately. For v0, keep it local and typed, then graduate only the parts proven by the UI.

Idea 3. Audience-omniscient inspector v0

Axis: Audience omniscience Confidence: 74% Complexity: Medium
Description
Build the right inspector around one selected agent, with tabs for overview, thoughts or strategy, receipts, and timeline. V0 should show explicit relationship edges plus selected strategy notes, but label inferred House or strategy-note material as analysis rather than canonical truth.
Basis
direct: The user said viewers can see thought/strategy summaries and chose explicit relationship data plus selected strategy notes even though it crosses implicit MCP boundaries. CONCEPTS.md:77 defines cognitive artifacts as product read-model records for reasoning, thinking, or strategy, and CONCEPTS.md:97 defines House long-form summaries as audience catch-up material.
Rationale
This is the screenshot's emotional center: the viewer should understand what Atlas thinks he is doing, who he trusts, who he fears, and which receipts support that read. It also makes the new view feel like a broadcast cockpit rather than a rearranged chat log.
Downsides
The boundary is intentional but sensitive. The UI needs provenance language and a narrow selection policy so viewers see sharp summaries instead of a raw reasoning dump.

Idea 4. Mingle as the first signature theater

Axis: Phase theaters Confidence: 88% Complexity: Medium
Description
Upgrade Mingle first by extracting the useful internals of OpenWhisperRoomsView into a Mingle-named theater: room strip or map, selected-room conversation, movement/room status, and inspector linkage. Keep old Whisper naming only where compatibility forces it.
Basis
direct: buildReplayScenes already special-cases MINGLE scenes and room metadata at spectacle-viewer.tsx:93; buildWhisperStageData parses room metadata, fallback allocation text, participants, and messages at whisper-phase.tsx:73. CONCEPTS.md:9 says Mingle is the current phase and not a display rename for Whisper.
Rationale
Mingle is where the show becomes social strategy. It is also the phase most visually aligned with the provided screenshot: private rooms, pressure, deals, room focus, and audience-only context all converge here.
Downsides
Renaming while preserving historical compatibility is tedious. Avoid a broad rename pass; extract Mingle-facing components while leaving old storage compatibility alone.

Idea 5. Receipt matrix for votes, pressure, and promises

Axis: Durable watch facts Confidence: 72% Complexity: Medium
Description
Add a compact receipt matrix that starts with durable vote facts: empower, expose, council, elimination, jury, power action, and protected targets. Then reserve a second lane for selected social receipts such as warning promises, protection asks, favors, and named information leads.
Basis
direct: The user said vote/deal receipts should be derived from backed storage. Canonical event types include vote.cast, power.action_set, council.vote_cast, player.eliminated, and jury.vote_cast, while CONCEPTS.md:25 defines the revealed vote ledger as named standard-round vote receipts.
Rationale
Receipts make the omniscient view legible. They turn "Atlas is under pressure" from a vibes panel into a traceable public/audience fact set the viewer can audit while watching Mingle and votes unfold.
Downsides
Deal receipts are less clean than vote receipts. For v0, ship vote/power/council receipts first and only include social receipts when the text or artifact evidence is explicit enough to avoid fake certainty.

Idea 6. Timeline as a command surface

Axis: Playback, accessibility, and closure Confidence: 82% Complexity: Medium
Description
Replace the passive progress strip with phase/chapter buttons that jump scenes, expose current position, and work in live and replay. The bottom dock should own play/pause, step, speed, cinema/strategy lens, and phase navigation, while the center content remains focused on the current theater.
Basis
direct: The current dramatic viewer renders a scene progress bar at dramatic-replay-viewer.tsx:765 but it is not the cockpit-level timeline in the screenshot. external: WAI carousel guidance recommends actual buttons/lists and clear current-state labels for slide navigation; WAI tabs guidance covers keyboard behavior for lens tabs; MDN documents prefers-reduced-motion for reducing non-essential animation.
Rationale
This is what makes the shell useful for admin monitoring and replay analysis. It should be possible to jump to R2 Vote, pause on a reveal, inspect an agent, then continue without losing orientation.
Downsides
Timeline controls can easily become crowded. Keep v0 to phase-level jumps and a small set of transport controls, then add fine-grained event navigation only after receipts exist.

Idea 7. Stats-first finale with a House summary slot

Axis: Playback, accessibility, and closure Confidence: 85% Complexity: Low
Description
For completed games, end the shell with a stats-first results view: winner, finalists, rounds survived, vote totals, power outcomes, eliminations, jury breakdown, and strongest receipt trails. Leave a visible but optional House summary region for a later narrative-awards layer.
Basis
direct: The user said final results should start with stats and later assume some House summary. game-projection-read-model.ts:43 already summarizes durable players, vote state, outcomes, and winner.
Rationale
Stats give viewers the clean "what happened" close before the richer "why it mattered" layer arrives. It also gives the replacement admin view a practical completed-game landing state.
Downsides
Without the House layer it may feel dry on first pass. That is acceptable because the slot is intentionally preserved without blocking the prototype.

Rejection Summary

Cuts that protect scope and product clarity
# Idea Reason Rejected
1Replay-only prototypeScope mismatch. The user clarified that the persistent shell replaces live admin immediately.
2Classic fallback/deprecated pathProduct ambiguity. Keeping a legacy/classic lane alive in the normal flow fights the new default.
3Full LLM deal ledgerToo expensive for v0. It makes social receipt extraction model-dependent before vote receipts are even stable.
4Relationship heatmap as the main stageCompetes with phase theaters. Relationships belong in the inspector until the watch shell proves itself.
5New WatchFrame API prerequisitePremature. A typed local adapter can prove the contract before adding API surface.
6Static screenshot-match prototypeBelow the usefulness bar. The request needs the real default surface and real derived state, not just CSS mimicry.
7Public-only lens firstDirection mismatch. Public-only remains valuable later, but the current build target is audience-omniscient.
8Raw thought firehoseWrong artifact level. Viewers need selected summaries and strategy notes, not unbounded hidden trace output.
9Full custom vote animation rewriteDuplicate burden. Existing reveal and tally components are enough for the first shell.
10Cinema-mode sidecarPreserves the old admin surface instead of replacing it, which delays the real product decision.
