MCP Agent Revision Loop

Six grounded tracks for making one agent remain one competitive identity while its behavior improves: connected LLMs continue an existing agent through update_agent, active revisions follow eligible enrollments automatically, and only an explicit draft or pin stops that movement.

What is true today

Grounding Context

One identity, immutable evidence

An Agent Profile is the persistent competitive identity and career. An Agent Revision is an automatic analytical boundary underneath it: behavior-bearing edits create the next immutable ordinal, presentation-only edits do not, and public career or season history does not reset.

The observed failure

A connected MCP LLM edited Lillith Voss by calling create_agent, producing a second profile with separate history. It then treated switching the Daily Free queue as “deploying the new revision.” The improved behavior and standing entry eventually had to be restored to the original profile; the duplicate remained a zero-game agent.

The MCP contract teaches the wrong boundary

create_agent allocates a UUID and initial revision but does not emphasize that it starts a separate career. update_agent says to tune an agent “before enrollment,” even though postgame editing is the intended loop. Owner-scoped list, search, get, queue, and active-enrollment reads already provide enough identity context to do better.

Useful mutation facts are discarded

The shared service already computes whether a revision was created or reused, then MCP and REST return only the updated agent. Clients cannot tell which revision is active, whether a change was presentation-only, what will play next, or which enrollments followed the update—despite MCP already supporting typed output schemas and structured content.

“Re-enroll” is not the storage operation

Daily Free stores the stable profile ID, so its standing row already follows the profile at the next draw; leaving and joining again would only churn queue state. Waiting open-game and drawn-game seats copy persona/runtime fields and can become stale. Runtime and suspended recovery correctly consume the frozen seat snapshot.

The safety boundary already has a shape

Game start uses an atomic waiting-to-in-progress claim. That transaction is the natural last point to resolve every unpinned owned seat to the active revision and then freeze it. Eager reconciliation makes waiting rooms look right; freeze-time resolution is the correctness backstop.

Visibility has a missing zero

Owner season analysis groups only revisions with eligible completed-game receipts and shows comparison after multiple measured cohorts exist. A newly active revision with zero games is therefore invisible precisely when the player needs confirmation that the next game will use it.

Prior art converges

Google resource APIs distinguish Create from Update. Temporal separates Auto-Upgrade from Pinned. Kubernetes reconciles desired state and exposes rollout outcomes. MLflow separates a stable registered identity from immutable versions and mutable current/candidate aliases. The common shape is stable identity, current pointer, automatic following, explicit exception.

Identity and MCP tool choice

1. Connector Loop Contract

Description: Make every postgame and owner-agent read a continuation point for the same competitive identity. Return the stable agentId, the revision that played, the current active revision, and suggestedMutation: "update_agent". Rewrite update_agent as the default path whenever an owned agent already exists—even while enrolled. Rewrite create_agent as explicit allocation of a separate career, require the caller to acknowledge that intent when a likely owned match exists, and return a typed conflict with candidate IDs instead of silently forking.

Conceptual identity fork: ordinary improvement stays on one resource; only explicit career creation crosses into a new profile.

Axis
Identity and MCP tool choice
Basis
direct: create_agent unconditionally allocates a new UUID; update_agent requires the stable ID but says “before enrollment”; owner-scoped search already exists; the Lillith incident demonstrates the failure. external: Google Create acts on a collection while Update targets a named resource and duplicate identity creation fails.
Rationale
This changes both model guidance and server behavior. The LLM gets an obvious continuation handle, while the API prevents one mistaken tool choice from manufacturing a second competitive identity.
Downsides
Duplicate detection must fail closed without blocking legitimate same-name careers. Ambiguous matches need a compact candidate response and an explicit confirmSeparateCareer-style escape hatch.

Cross-cutting mutation spine

2. Revision-Aware Update Orchestrator

Description: Put the entire active-update loop behind one shared transaction/service used by MCP and REST: authorize the stable identity, apply profile changes, classify the effective runtime fingerprint, create or reuse the immutable revision, move the active pointer when appropriate, reconcile every eligible follower, and return one typed mutation receipt. The LLM should never separately decide whether to “make a revision” or ask whether to update the queue; those are domain postconditions of update_agent.

The transaction can become a god service if it absorbs read concerns. Keep projection and reconciliation as named collaborators, cap detailed enrollment arrays, and expose partial failures without claiming a fully active rollout.

Enrollment propagation and start boundary

3. Follow Current Until Freeze

Description: Make follow_active the default tracking policy for Daily Free and owned seats in waiting games. An active update eagerly refreshes waiting seat persona, config, and revision through one owned-seat projection service. Every game start then resolves unpinned seats again inside the atomic waiting-to-in-progress claim and pins the resulting revision forever. Started and suspended games never move. Eager reconciliation is visible responsiveness; freeze-time resolution is correctness.

Conceptual lifecycle: follow while a roster is provisional; resolve once at the starting gun; preserve historical truth afterward.

Axis
Enrollment propagation and start boundary
Basis
direct: Daily Free stores agentProfileId; waiting seats persist copied persona/config; runtime consumes stored seat data; start already has an atomic status claim. external: Temporal's Auto-Upgrade versus Pinned policy and Kubernetes desired-state reconciliation closely match the before/after-freeze split.
Rationale
This delivers the user's “re-enroll latest revision” intent without corrupting queue age or pretending a standing row is deployment state. It also closes the update-versus-start race rather than hoping an eager write wins.
Downsides
Requires disciplined lock ordering and concurrency tests. Operators must understand that a waiting roster is provisional. A new roster_frozen event should not be added unless existing status and startedAt predicates prove insufficient.

Revision lifecycle and activation

4. Active by Default, Draft by Exception

Description: Keep the normal edit path quiet: update_agent creates or reuses a revision, marks it active, and lets followers move. Add one explicit exception such as activation: "draft"; it persists a candidate revision without changing the active pointer or reconciling enrollments. Publishing the draft later activates it and runs the same orchestrator. Revision readiness (active or draft) remains independent from enrollment tracking (follow_active or pinned). A draft is not a queue pause.

Conceptual policy split: the default path remains frictionless; holding a revision requires explicit intent and does not mutate queue membership.

Axis
Revision lifecycle and activation
Basis
direct: Influence defines revisions as quiet automatic comparison boundaries and already moves currentRevisionId when behavior changes; Standing Daily Agent explicitly has no pause state. external: MLflow aliases/tags and Temporal's version/routing split show why readiness and tracking are separate concepts.
Rationale
This preserves the intended low-ceremony improvement loop while giving sophisticated owners and LLMs a deliberate “save, do not play yet” escape hatch. Default behavior stays legible: unless the caller says draft, the update is the next behavior to play.
Downsides
Durable drafts add publish/discard semantics and a choice about whether one or multiple candidates may exist. The option must remain subordinate in tool schemas so it does not become the new default ceremony.

Receipts, visibility, and behavior measurement

5. Two-Clock Revision Learning Card

Description: Keep revisions quiet but visible on the existing agent detail surface—not as another Lillith card. Show two clocks: the active revision that will play next and the most recent revision with completed evidence. A just-edited agent reads “Current r4 · plays next · awaiting games” beside “Last measured r3 · 4 games · 1 win.” If a draft exists, show it as a held candidate. The same fields belong in owner MCP reads so web and connected LLMs share one vocabulary.

Axis
Receipts, visibility, and behavior measurement
Basis
direct: season analysis builds revision groups only from eligible receipts and hides comparison until multiple measured revisions exist; therefore a zero-game current revision is absent. The update service and revision table already know the active ordinal.
Rationale
Players can confirm that an edit will affect the next game without turning revisions into a deployment dashboard. Separating “what will play” from “what has evidence” also prevents premature claims that the latest revision is better.
Downsides
Revision language can become dashboard clutter. Keep the current state to one compact card and put historical comparison behind the existing disclosure. Avoid mandatory behavioral hypotheses until users prove they can supply useful ones.

Receipts, visibility, and behavior measurement

6. Connector Revision Gauntlet

Description: Turn the Lillith failure into a behavioral regression corpus across ChatGPT, Claude, Grok, Codex, and Claude Code. Give each connector the same scenarios: tune an established agent after a game, disambiguate two owned matches, save an explicit draft, update a waiting enrollment, and update after roster freeze. Score the actual tool sequence, stable ID preservation, receipt interpretation, queue-churn avoidance, and final state. Pair pre-release runs with prompt-free production counters for avoidable creates, same-name conflicts, updates without an identity read, and propagation outcomes by provider.

Axis
Receipts, visibility, and behavior measurement
Basis
direct: the repo already identifies five MCP provider profiles and records provider-tagged tool calls, while current tests validate schemas rather than end-to-end LLM choices. The incident itself was a connector behavior failure. external: MCP makes tool descriptions part of the server-owned selection contract, so behavior—not copy review—is the relevant proof.
Rationale
This makes “connected LLMs understand the loop” measurable. It catches provider-specific interpretation drift before release and shows whether the contract works in production without storing private prompts.
Downsides
LLM behavior is nondeterministic and hosted connectors change. Keep the corpus small, score invariants rather than wording, store correlation IDs and outcomes instead of prompts, and distinguish a flaky model run from a server contract regression.
