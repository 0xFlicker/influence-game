Durable Event Read Model

Make the persisted Postgres event log useful, not just durable: load API-backed game_events, replay them into the canonical projection, and show an operator the durable truth of a suspended or completed run before attempting full checkpoint resume.

Repo Grounding

Grounding Context

  • API events are durable but not yet queryable as truth. packages/api/src/services/game-events.ts appends canonical envelopes under owner epoch, validates game identity, enforces contiguous sequences, stores stable hashes, and suspends on append failure. There is no paired read service that loads those rows and turns them back into a projection.
  • The replay reducer already exists. packages/engine/src/game-projection.ts exposes replayCanonicalEvents(...), validates contiguous ordering, rejects wrong game IDs and unsupported payload versions, and rebuilds the canonical projection used by engine tests.
  • The simulator side has the model to borrow from. packages/engine/src/game-mcp/read-model.ts reads simulation event JSONL, validates canonical event records, replays projection, filters events by visibility, and returns source citations. API read paths need the same semantics over Postgres rows, not the filesystem/MCP wrapper.
  • Admin UX has a suspended bucket but not an inspection surface. packages/api/src/routes/admin.ts and packages/web/src/app/admin/admin-panel.tsx expose redacted kernel counts and a "needs inspection" status. The game viewer currently tells users replay is unavailable for suspended games and only shows durable event/checkpoint/manifest counts.
  • Checkpoint capsules are intentionally forensic. packages/engine/src/game-runner.types.ts and packages/api/src/services/game-checkpoints.ts record projection summaries and missing hydration inputs while keeping hydrateable=false. The next read model should surface that honestly instead of implying resume.
  • External grounding supports this order. XState documents persisted actor snapshots and notes caveats around restore/replay; Microsoft Azure's event-sourcing guidance emphasizes ordered event streams, materialized projections, versioned events, and the operational cost of treating events as permanent truth; PostgreSQL's JSONB docs support the current row-plus-envelope shape but warn that JSON documents should have predictable structure.

Surface Map

Topic Axes

  • Event Retrieval And Integrity Loading rows, validating envelopes, recomputing hashes, and determining whether the log head is coherent.
  • Projection Replay And Parity Reusing the engine reducer so API Postgres events and simulator JSONL events prove the same canonical contract.
  • Operator Inspection Surface Giving admin/operator routes enough summarized truth to debug suspended and completed games without reading raw tables.
  • Checkpoint Hydration Readiness Making forensic checkpoint capsules inspectable while keeping resume explicitly deferred.
  • Private Evidence And Future Raw Logs Keeping prompt/response/reasoning evidence behind manifest metadata so Linode S3 raw logs can land later without leaking into UX-facing events.

1

Durable Truth Read Model Service

Event Retrieval And Integrity Confidence 88% Complexity Medium

Description: Add getPersistedGameEvents(gameId) and getPersistedGameProjection(gameId) in the API service layer. The first loads ordered Postgres game_events, validates that stored metadata agrees with each envelope, recomputes hashes, and returns typed canonical events plus warnings. The second feeds those events into replayCanonicalEvents(...) and returns either a canonical projection or an explicit "no durable projection yet" result for games with no event log.

Basis
direct: game_events already stores sequence, type, hash, visibility, source pointers, payload version, run source, and JSONB envelope; replayCanonicalEvents(...) already rebuilds a projection from ordered canonical events.
Rationale
This is the smallest move that changes the event table from "write-only durability artifact" into a system of record an API/admin path can interrogate.
Downsides
It will expose old or manually inserted rows that fail stricter canonical validation; the service must report those as diagnostics instead of crashing admin pages.
2

Admin Durable Run Endpoint

Operator Inspection Surface Confidence 85% Complexity Medium

Description: Add a producer/admin route such as GET /api/admin/games/:id/durable-run. It should return the latest owner row, event head, replayed projection summary, checkpoint timeline, evidence-manifest summary, and diagnostics. It should not return raw prompts, raw responses, hidden reasoning, private object keys, or a player-facing event stream.

Basis
direct: /api/admin/games already enriches summaries with redacted kernel health; the current suspended viewer only shows counts and says replay is unavailable.
Rationale
Operators need one authoritative payload that answers "where did this run actually stop?" without hand-querying four tables.
Downsides
Route response shape can become a kitchen sink. Keep the first version summary-first, with paginated/filterable event rows left to the service or later endpoint.
3

Event Integrity And Completeness Auditor

Event Retrieval And Integrity Confidence 82% Complexity Medium

Description: Layer diagnostics over the read service: sequence continuity, envelope validation, hash recomputation, owner head versus event max sequence, checkpoint boundary hashes, projection last sequence, terminal game_results presence, and games.status consistency. Return warnings with stable codes such as event_gap, hash_mismatch, owner_head_behind, checkpoint_ahead_of_head, and terminal_result_missing.

Basis
direct: the append path already treats identity, ordering, owner status, and hash as durable-write invariants; the read path should independently verify those invariants before future hydration depends on them.
Rationale
This turns suspended games from "something failed" into a precise forensic record and exposes event-completeness gaps before pause/resume uses the log as input.
Downsides
Some diagnostics will be noisy for old pre-kernel games or direct test fixtures. The response needs compatibility modes rather than assuming every game should have a projection.
4

Projection Parity Harness Across API Postgres And Simulator JSONL

Projection Replay And Parity Confidence 80% Complexity Medium

Description: Add tests that prove persisted API events replay through the same projection contract as simulator JSONL. The lightweight first version can use a mock or fast deterministic runner, append to Postgres through the durable sink, load via getPersistedGameEvents, replay via getPersistedGameProjection, and compare to the runner's getDomainProjection() or to the existing engine replay fixture.

Basis
direct: docs now state simulator event JSONL uses the same canonical envelope that API-backed games persist in Postgres; the game MCP already rebuilds projections from JSONL, and engine tests compare replayed projection to live domain projection.
Rationale
This is the proof that simulation/API unification is real at the event-contract layer, while preserving the current decision that CLI simulation remains a separate execution/storage path.
Downsides
A full API game can be slow or LLM-dependent, so the first tests should avoid hosted model calls and target the projection contract rather than end-to-end game quality.
5

Checkpoint Timeline And Hydration Readiness Matrix

Checkpoint Hydration Readiness Confidence 78% Complexity Low-Medium

Description: Include a checkpoint list in the durable-run response: kind, round, phase, last event sequence, event-head hash, projection hash, transcript cursor, token-cost cursor, hydrateable, degraded reason, and missing hydration inputs. Render this as "readiness, not resume" so the next checkpoint v0 work has a concrete checklist to close.

Basis
direct: GameCheckpointCapsule already names missing inputs: xstate snapshot, phase accumulators, agent memory state, pending LLM calls, and token cost cursor. The DB stores hydration status, snapshot, transcript cursor, token cost cursor, and degraded reason.
Rationale
Checkpoint hydration becomes less spooky when every suspended run can say exactly which state categories are present and which are still absent.
Downsides
It may create pressure to flip hydrateable early. The display language should keep "forensic only" obvious until every required field is populated and tested.
6

Minimal Admin Inspect Panel

Operator Inspection Surface Confidence 72% Complexity Medium

Description: Add a small admin-only "Durable Truth" panel reachable from suspended/completed game rows. It should prioritize event head, projection headline, alive/eliminated counts, latest phase/round, latest checkpoint, and diagnostics. The event list can be summarized or paginated later; the first panel is for answering the operator's immediate question quickly.

Basis
direct: the admin panel already groups suspended games under "Needs Inspection"; the normal suspended viewer has no replay and only displays durable counts.
Rationale
A read service without a human-facing operator surface will still leave failures feeling opaque. A compact panel makes the kernel visibly useful without turning it into a public replay product.
Downsides
UI can distract from service correctness. Build it after the API payload is stable, and keep it admin-only with private evidence redacted.
7

Private Evidence Manifest Preview Contract

Private Evidence And Future Raw Logs Confidence 70% Complexity Low-Medium

Description: Return evidence manifest summaries in the durable-run endpoint: counts by evidence type, event-sequence coverage, retention class, redaction status, and storage provider presence. Keep raw content and dereferenceable storage keys out of UX-facing events. This prepares for Linode S3 raw prompt/response/reasoning logs while keeping Postgres the canonical event system of record.

Basis
direct: game_evidence_manifests already has event sequence, evidence type, retention class, access scope, redaction status, storage provider/bucket/key, source pointers, and metadata; prior requirements explicitly keep hidden reasoning out of canonical/public surfaces.
Rationale
The operator needs to know whether raw evidence exists for debugging, but this slice should prove metadata and access boundaries before exposing raw evidence retrieval.
Downsides
Even manifest metadata can leak if it includes local paths or object keys. The first response should be redacted-by-default and producer/admin-scoped.

Rejection Summary

#IdeaReason Rejected
1Materialized Projection Cache TableUseful later if replay cost becomes real, but premature before on-demand replay and diagnostics prove the event log is complete.
2Full Event BrowserToo much UI before the service contract; also increases risk of surfacing producer/private data casually.
3Postgres MCP ServerInteresting after API read paths exist, but admin/API is the immediate operator surface for suspended games.
4Immediate GameRunner.fromCheckpoint()Rejected as unsafe because checkpoint capsules remain explicitly non-hydrateable and omit XState, accumulators, agent continuity, pending calls, and token-cost state.
5Redis/PubSub/Horizontal ScalingOut of scope for the current question. The urgent gap is durable explanation of one single-writer run, not multi-instance distribution.
6Public Player Replay APIPrivacy and product-contract risk. Keep producer/admin inspection first; public replay can be designed after visibility semantics are proven.
7Raw S3 Evidence BrowserManifests, access policy, redaction, and read auditing should precede any raw prompt/response/reasoning browser.
8Simulation Import NowLeave the simulation_import door open, but do not make CLI simulation depend on DB/API in this slice.
9Historical Event Rewrite/Repair ToolingDefer until real schema-evolution pain appears. Start with tolerant validation, warnings, and version-aware replay rather than mutation tooling.
