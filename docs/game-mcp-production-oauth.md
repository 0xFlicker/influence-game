# Production Game MCP OAuth

This is the deployed HTTP Game MCP surface for fresh Codex/Claude/ChatGPT-style clients. It preserves the current access contract:

`mcp` role + OAuth `scope=mcp` + valid bearer token for the canonical `/mcp` resource = global access to the wired Production Game MCP tools.

Do not add per-user, per-agent, private-agent, per-game, row-level, or private-reasoning authorization inside this V0. Private reasoning and raw trace content are intentionally available through explicit developer tools behind the same global MCP gate.

## Server Surface

- MCP endpoint: `POST /mcp`
- Optional MCP GET stream: not supported in V0; authenticated `GET /mcp` returns `405 Method Not Allowed`
- Protected resource metadata: `GET /.well-known/oauth-protected-resource`
- Path-specific protected resource metadata: `GET /.well-known/oauth-protected-resource/mcp`
- Authorization server metadata: `GET /.well-known/oauth-authorization-server`
- Dynamic public client registration: `POST /api/oauth/mcp/register`
- Authorization endpoint: web `/oauth/mcp/authorize`
- Token endpoint: API `/api/oauth/mcp/token`

The HTTP endpoint requires `Authorization: Bearer <mcp-token>` on every request, rejects bearer tokens in query strings, validates `Origin` when present, accepts one JSON-RPC message per POST, returns JSON for normal requests, and returns `202 Accepted` for accepted notifications/responses.

## Environment

Set these per deployed environment:

```bash
MCP_OAUTH_RESOURCE_URI=https://<api-host>/mcp
MCP_OAUTH_WEB_BASE_URL=https://<web-host>
MCP_OAUTH_AUTHORIZATION_SERVER_ISSUER=https://<api-host>
MCP_ALLOWED_ORIGINS=https://<api-host>
```

Optional overrides:

```bash
MCP_OAUTH_AUTHORIZATION_ENDPOINT=https://<web-host>/oauth/mcp/authorize
MCP_OAUTH_TOKEN_ENDPOINT=https://<api-host>/api/oauth/mcp/token
MCP_OAUTH_REGISTRATION_ENDPOINT=https://<api-host>/api/oauth/mcp/register
MCP_OAUTH_ALLOWED_REDIRECT_URIS=https://<client-callback>
MCP_OAUTH_LOOPBACK_REDIRECT_PATH=/oauth/callback
# Optional; unset behaves as false.
MCP_OAUTH_ALLOW_DYNAMIC_HTTPS_REDIRECTS=false
```

Dynamic client registration is enabled for public MCP clients. Registered loopback `http(s)` redirect URIs are allowed for fresh local Codex/Claude-style clients regardless of `MCP_OAUTH_ALLOW_DYNAMIC_HTTPS_REDIRECTS`.

For non-loopback hosted client callbacks, the default is allowlist-only: keep `MCP_OAUTH_ALLOW_DYNAMIC_HTTPS_REDIRECTS` unset or `false`, then list exact callback URLs in `MCP_OAUTH_ALLOWED_REDIRECT_URIS`. Set `MCP_OAUTH_ALLOW_DYNAMIC_HTTPS_REDIRECTS=true` when testing hosted clients whose HTTPS callback URL is generated by the client platform and cannot be known ahead of time, such as ChatGPT/OpenAI developer-mode compatibility runs. The static `MCP_OAUTH_CLIENT_ID` path remains for local/back-compat clients and uses the configured loopback redirect path.

Some MCP clients request `grant_types: ["authorization_code", "refresh_token"]` during dynamic client registration even when the advertised server metadata only supports authorization-code. V0 accepts that registration metadata for compatibility, but registers and returns only `["authorization_code"]`; the token endpoint does not issue refresh tokens.

Private trace tools require the same private content storage env used by API durable runs:

```bash
LINODE_PRIVATE_CONTENT_ENDPOINT=...
LINODE_PRIVATE_CONTENT_ACCESS_KEY=...
LINODE_PRIVATE_CONTENT_SECRET_KEY=...
LINODE_PRIVATE_CONTENT_BUCKET=...
```

`INFLUENCE_MCP_INTROSPECTION_SECRET` remains for the local stdio bridge/introspection endpoint. Deployed `/mcp` validates opaque tokens directly against the DB-backed OAuth token rows and does not expose the introspection secret to clients.

## Tools

Production Game MCP exposes only read-only tools:

- `list_games`: recent deployed games with event-log/projection status.
- `read_projection`: replay persisted canonical events into a durable projection summary.
- `filter_events`: filter persisted canonical events by type, phase, actor, sequence, visibility mode, and limit.
- `player_timeline`: canonical event timeline for a player ID or name.
- `inspect_durable_run`: durable-run inspection summary and evidence counts.
- `list_trace_manifests`: private trace metadata for one game.
- `read_trace_content`: explicit raw private trace read by manifest ID.
- `search_reasoning_traces`: bounded private reasoning search previews inside one game.

The private trace tools are not a separate authorization realm in this slice. They are explicit tools behind the same `scope=mcp` global developer access boundary.

`read_trace_content` defaults to an 8 MiB raw trace read limit and clamps tool-supplied `maxBytes` at 64 MiB. `search_reasoning_traces` exposes `limit` for result count and returns short previews; object scan limits are server-side safety bounds, not caller-tuned result-size controls. These are response-content bounds, separate from the `/mcp` request body limit.

## Client Paths

### Codex

Codex is the release gate. Configure the deployed server as a Streamable HTTP MCP server, then run OAuth login:

```toml
[mcp_servers.influence_game]
url = "https://<api-host>/mcp"
tool_timeout_sec = 60
```

```bash
codex mcp login influence_game
```

Codex prefers `scopes_supported` advertised by the server, so metadata must include `["mcp"]`. The authorization metadata advertises `registration_endpoint`, allowing a fresh public client to register its callback before the PKCE authorization-code flow. If the callback must use a fixed non-loopback HTTPS address, configure `mcp_oauth_callback_port` or `mcp_oauth_callback_url` in Codex config and add the resulting HTTPS redirect URI to `MCP_OAUTH_ALLOWED_REDIRECT_URIS`.

Ready means a fresh Codex session can initialize the server, complete OAuth in the browser, store/use the token, call `list_games`, and then call at least one game-specific tool such as `read_projection` or `filter_events`.

### Claude Code

Claude Code should use the HTTP transport:

```bash
claude mcp add --transport http influence-game https://<api-host>/mcp
```

Use `/mcp` inside Claude Code to authenticate when it reports OAuth is needed. Claude Code checks protected resource metadata first, then authorization server metadata, can use dynamic client registration for public clients, and can override metadata discovery with `authServerMetadataUrl` if a deployment proxy blocks standard well-known paths.

### ChatGPT Developer Mode / Apps SDK

ChatGPT/App SDK compatibility depends on:

- HTTPS well-known protected resource metadata.
- `WWW-Authenticate: Bearer resource_metadata="...", scope="mcp"` on `401`.
- Authorization server metadata with authorization endpoint, token endpoint, PKCE S256, and `scopes_supported`.
- Dynamic client registration for public clients, or an explicitly configured HTTPS callback URI for the developer-mode client.
- Tool descriptors carrying OAuth security schemes for `scope=mcp`.

This slice builds toward an installable ChatGPT app, but public app submission, app UI components, per-tool linking policy, and broad tester rollout remain out of scope.

## Operational Checks

Before calling the slice ready on staging:

1. `GET https://<api-host>/.well-known/oauth-protected-resource` returns `resource: https://<api-host>/mcp`.
2. `GET https://<api-host>/.well-known/oauth-authorization-server` returns the web authorization endpoint, API token endpoint, registration endpoint, `scopes_supported: ["mcp"]`, and `code_challenge_methods_supported: ["S256"]`.
3. Unauthenticated `POST /mcp` returns `401` with a `WWW-Authenticate` protected-resource metadata challenge.
4. Query-string bearer tokens are rejected before dispatch.
5. Wrong resource, wrong scope, expired, revoked, app-session, or role-removed tokens fail before any read model runs.
6. A valid token can initialize, list tools, list deployed games, read a projection, filter events, and explicitly read/search private trace content when storage is configured.
7. Audit events include correlation ID, method/tool, user/client/resource, result, status, and denial reason, but never raw tokens, auth headers, authorization codes, PKCE verifiers, raw prompts, raw responses, reasoning bodies, or storage credentials.

## Out Of Scope

- App-side rate limiting. Put rate limiting behind a real gateway in a later durable deploy hardening pass.
- Per-user, per-game, per-agent, row-level, private-agent, or private-trace authorization once a token is valid.
- Refresh tokens.
- Confidential-client management, client secrets, and a general third-party OAuth app platform.
- Mutation tools or game lifecycle controls.
- Public ChatGPT app submission and app UI widgets.
